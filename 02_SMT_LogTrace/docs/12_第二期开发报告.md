# 第二期开发报告

完成日期：2026-07-13。

## 1. 阶段结论

第二期“增量消费与日志解析”已完成实现与复验。`logsearch_server` 能在后台按
`archive_id` 扫描一期只读归档，对 `RUNTIME_LOG` 和 `TEST_REPORT` 执行明确解析，并将成功批次
原子发布为 `PARSED` 工件。失败归档保留稳定错误码和首个失败行号，不阻塞后续归档。

最终状态为“复验通过，用户已确认”。本期没有构建倒排索引或注册搜索接口；用户确认后，
项目已进入第三期，从 `PARSED` 工件构建不可变 Segment 并发布 `READY` 查询快照。

## 2. 已完成范围

- 严格索引配置：轮询周期、源归档批量、文档批量和单行字节上限；
- 一期 `archive_file` 只读仓储和二期批次/归档状态仓储；
- `kv_runtime_v1` 与 `fct_csv_v1` 两种固定解析器；
- UTF-8、BOM、CRLF、CSV 引号、数值、时间和单行上限校验；
- 流式计算文件大小与 SHA-256，并与一期元数据回验；
- 以原始字节记录每个文档的 `offset/length`，排除 LF 或 CRLF 行结束符；
- 相对路径穿越和符号链接逃逸防护；
- 文件级原子解析：任一行失败时不保留该归档的部分文档；
- `logsearch_server` 后台轮询和 `logtrace_admin` 单次扫描/明确重建命令；
- `index_root/.indexer.lock` 跨进程串行化，防止常驻进程与管理命令并发改写状态；
- 确定性固定三线样本生成器。

## 3. 状态机与重建

`002_parsed_batch.sql` 将批次状态扩展为 `PARSING/PARSED/BUILDING/READY/FAILED`，将归档状态扩展为
`PENDING/PARSED/INDEXED/FAILED`，并增加解析工件路径、工件摘要和失败行号。

第二期成功批次停在 `PARSED`。上次中断的 `PARSING` 批次会被明确标记失败，不猜测继续。
`rebuild --archive-id ID` 以原批次为单位清理工件，将该批归档重置为 `PENDING`，由后续扫描重新解析。

## 4. `PARSED` 工件

每个成功批次原子发布到 `index_root/parsed/batch_<batch_id>/`：

```text
manifest.json
archives.jsonl
documents.jsonl
```

`manifest.json` 记录批次和文件统计，`archives.jsonl` 记录成功归档元数据，`documents.jsonl`
记录结构化文档和字节范围。工件不保存完整原文，可从一期归档重建，不是原始业务事实。

## 5. 稳定错误码

本期已验证或实现的主要错误包括：

- 解析配置：`PARSER_PROFILE_NOT_FOUND`、`PARSER_PROFILE_UNSUPPORTED`；
- 归档完整性：`ARCHIVE_FILE_NOT_FOUND`、`ARCHIVE_FILE_OPEN_FAILED`、`ARCHIVE_FILE_READ_FAILED`、
  `ARCHIVE_FILE_CLOSE_FAILED`、`ARCHIVE_SIZE_MISMATCH`、`ARCHIVE_SHA256_MISMATCH`；
- 行与编码：`LINE_TOO_LONG`、`INVALID_UTF8`、`RUNTIME_LOG_FORMAT_INVALID`；
- FCT CSV：`FCT_CSV_EMPTY`、`FCT_CSV_FORMAT_INVALID`、`FCT_CSV_HEADER_INVALID`、
  `FCT_CSV_METADATA_MISMATCH`、`FCT_CSV_ROW_INVALID`；
- 批次工件：`NO_ARCHIVE_PARSED`、`PARSED_ARTIFACT_WRITE_FAILED`、`REBUILD_REQUESTED`。

失败记录不写入完整原文，日志也不输出归档正文或数据库凭据。

# SMT 日志索引与故障检索系统面试回答参考

## 1. 文档用途

本文对应简历中的“SMT 检测设备日志索引与故障检索系统”，用于准备项目介绍、技术细节和业务场景
追问。内容以当前项目实际实现和测试结果为依据，不把框架内部能力说成自己从零实现，也不把本机
仿真数据说成真实工厂生产指标。

面试回答建议采用这样的顺序：先说结论，再说明业务原因，然后讲实现，最后补充异常场景和取舍。
文中的回答是口语化参考，不建议逐字背诵。真正面试时应根据问题控制长度，并以自己确实理解和参与
过的内容为准。

## 2. 项目介绍与个人职责

### 2.1 请介绍一下这个项目

**30 秒版本：**

这个项目主要用于 SMT 产线的设备故障排查。前一个采集归档系统已经把 AOI、SPI、FCT 等设备产生的
运行日志和测试记录可靠地保存下来，这个项目在此基础上做增量解析和索引，提供关键词检索、错误码
查询、异常日志筛选和原始日志回读。系统使用 Wfrest 提供 HTTP 接口，Gateway 和 Search Server
之间通过 SRPC、Protobuf 通信；搜索侧实现了倒排索引、BM25、Top-K 和 SLRU，并用 Redis 缓存查询
结果。命中日志后根据 offset 和 length 使用 pread 回读一期归档中的原始内容。

**两分钟版本：**

这个项目解决的是设备日志分散、人工逐文件排查慢、历史故障难追溯的问题。数据来源包括 AOI、SPI、
FCT 设备运行日志、产线 IPC 日志和测试报告。前一个 DataStream 项目负责采集、校验和归档；LogTrace
只读消费归档元数据和不可变文件，不重复做上传和设备认证。

系统分为 Gateway 和 Search Server 两个进程。Gateway 基于 Wfrest，对外提供关键词搜索、异常列表、
错误码知识和日志详情接口，负责 HTTP 参数校验、Bearer Token 鉴权以及 RPC 错误到 HTTP 状态码的
转换。Search Server 负责增量扫描归档、解析日志、构建不可变 Segment、维护查询快照和执行检索。
两个进程之间使用 Protobuf 定义消息，通过 SRPC 通信。

索引按批次增量构建。每条日志记录为一个文档，结构化字段包括设备、工位、工单、产品 SN、等级、
模块和错误码，正文则构建倒排索引。多关键词查询使用 AND 语义，先处理文档频率低的词，再用 BM25
和业务权重打分，通过 Top-K 小顶堆保留最优结果。日志正文没有重复写入 MySQL，而是在索引中保存
文件定位信息，详情查询时使用 pread 回读原始归档。

缓存分为两层。本地 SLRU 缓存热点详情、错误码知识和文件元数据；Redis 缓存规范化查询对应的日志
ID 列表和空结果。Redis Key 包含索引快照版本，Redis 不可用或缓存值损坏时直接回到本地索引，保证
结果正确。

### 2.2 你在项目中主要负责什么

参考回答：

我主要负责服务端检索链路，包括 Gateway 的 HTTP 接口、Protobuf 和 SRPC 契约、增量索引、倒排
索引、BM25 排序、Top-K、本地 SLRU、Redis 查询缓存以及原始日志回读。我也参与了索引状态表、错误
码知识库和部署配置的设计，并通过单元测试、真实 MySQL/Redis 集成测试以及跨项目 E2E 验证从设备
文件归档到 HTTP 查询的完整流程。

如果面试官追问团队分工，应按真实经历说明自己负责或参与的部分，不要为了显得工作量大而虚构团队
人数、上线规模或他人的职责。

### 2.3 这个项目最难的地方是什么

参考回答：

我认为有三个难点。第一个是增量索引的一致性，索引由多个文件组成，不能让查询看到构建到一半的
数据，所以需要临时目录、完整校验、原子改名、数据库 READY 状态和内存快照交换共同保证。第二个
是检索性能，既要做多关键词 AND、结构化过滤和相关性排序，又不能对全部命中结果完整排序，因此
采用低 DF 优先求交和 Top-K 小顶堆。第三个是缓存正确性，索引发布后结果集合会变化，所以 Redis
Key 必须带快照版本，而且 Redis 故障只能影响性能，不能影响查询结果。

### 2.4 项目和前一个 DataStream 项目是什么关系

参考回答：

DataStream 负责把设备文件可靠地送进正式归档，处理设备认证、分片上传、完整性校验和归档元数据。
LogTrace 是下游只读消费者，读取 `archive_file` 元数据和归档文件，对运行日志和测试报告建立索引。
两个项目的职责是分开的：一期解决“文件怎么可靠收上来”，二期解决“归档后怎么快速检索和排障”。

## 3. SMT 真实业务场景

### 3.1 SMT 产线上为什么需要日志检索系统

参考回答：

设备发生异常时，工程师通常需要结合设备编号、时间、产品 SN、错误码和上下文日志判断原因。如果
只有按日期保存的原始文件，就要先找到对应工控机，再逐个目录和文件搜索，历史问题尤其耗时。这个
系统把分散的归档日志统一建立索引，可以先按设备、工单或时间缩小范围，再通过关键词和错误码查询，
最后回到原始日志确认上下文，能明显减少人工排查步骤。

### 3.2 SPI、AOI、FCT 的日志有什么区别

参考回答：

SPI 主要检查锡膏印刷，日志中可能出现高度、面积、偏移和印刷异常；AOI 主要检查贴装和焊接外观，
常见内容包括少件、错件、极性和识别异常；FCT 是功能测试，通常记录测试点、上下限、实测值和结果。
所以项目不能把所有文件都当成普通文本行。运行日志按物理行解析，FCT CSV 则把一个测试点作为一个
文档，同时保留来源类型，不能把测试点 NG 伪装成设备报警。

### 3.3 什么算异常日志

当前基线中，等级为 WARN、ERROR，或者带有非空错误码的运行记录属于异常；FCT 测试点结果为 NG
也属于异常记录。查询时仍保留原始来源、模块和等级，便于工程师区分设备报警、测试失败和采集侧
异常。

### 3.4 为什么要保存设备、工位、工单和产品 SN

这些字段对应不同的排障入口。设备和工位适合定位某台机器的连续故障；工单适合分析某个生产批次；
产品 SN 适合追踪单块板在多个工位的记录。真实排查通常不是只搜一个关键词，而是先用这些结构化
条件缩小范围，再查看正文。

### 3.5 一个典型排障过程是什么

参考回答：

比如 AOI 某段时间连续出现 `INSPECTION_NG`。工程师可以先按设备和时间查询异常日志，再查看错误码
说明和最近匹配记录。如果发现集中在同一工单或同类产品 SN，就继续查看对应日志详情和上下文，判断
是元件识别、相机、程序参数还是产品本身的问题。系统提供的是检索和证据聚合，不直接替代工程师做
工艺结论。

### 3.6 为什么不直接使用 Linux grep

grep 适合已知文件和单机临时排查，但这里的数据跨设备、工位、日期和文件类型，还要按工单、产品
SN、等级、模块等字段筛选。每次遍历大量历史文件也会重复消耗磁盘 I/O。建立增量索引后，可以先用
结构化字段和倒排表缩小候选，再按相关性排序，并通过统一接口提供给运维工具。

### 3.7 为什么没有直接使用 Elasticsearch

参考回答：

这个项目的目标之一是学习和实现倒排索引、BM25、Top-K、Segment 发布和缓存，所以选择了自研的
轻量索引。同时当前是单机、固定数据格式和明确查询范围，没有引入 Elasticsearch 集群的必要。
如果生产规模扩大，需要分布式索引、副本、复杂聚合和成熟运维生态，我会优先评估 Elasticsearch
或 OpenSearch，而不是继续扩大自研范围。

## 4. 架构与进程拆分

### 4.1 为什么把 Gateway 和 Search Server 拆成两个进程

参考回答：

两个进程的职责和资源特点不同。Gateway 面向 HTTP，主要负责鉴权、参数校验、协议转换和错误映射；
Search Server 持有索引快照，负责增量构建和 CPU 密集型检索。拆分后可以分别维护和重启，HTTP
协议变化不会直接侵入索引代码，索引服务也可以限制为只接受内部 RPC。代价是增加了 RPC、超时和
部署复杂度，所以当前只拆这两个有明确边界的进程，没有继续细分更多微服务。

### 4.2 一次搜索请求经过哪些步骤

参考回答：

请求先到 Nginx，再转发给 Gateway。Gateway 校验 Bearer Token、关键词数量、分页和过滤条件，然后
组装 Protobuf 请求并调用 SRPC。Search Server 先根据规范化查询和当前快照版本查 Redis，命中后按
doc_id 从同一快照恢复摘要；未命中或 Redis 异常时执行本地索引查询。结果通过 SRPC 返回 Gateway，
最后转换为统一 HTTP JSON 响应。

### 4.3 为什么没有把索引构建单独拆成第三个进程

当前索引构建和查询共享 Segment 加载、READY 状态和快照交换逻辑，单机规模下放在 Search Server
中更容易保证发布顺序。运维场景仍提供 `logtrace_admin` 做单次扫描、构建和重建。如果后续构建任务
明显影响查询延迟，或者需要独立扩缩容，可以再把构建器拆出来，但需要增加跨进程发布通知和更严格
的共享索引目录协调。

### 4.4 Nginx 在项目中做什么

Nginx 负责 TLS 终止和反向代理，Gateway 只监听本机地址。配置中还限制请求体大小、连接超时和读取
超时，并添加 HSTS、nosniff 等安全响应头。Nginx 超时要大于正常 Gateway 请求时间，但不能设置得
过长，否则下游异常时会积压连接。

## 5. Wfrest、Workflow、网络与异步编排

### 5.1 为什么使用 Wfrest 和 Workflow

参考回答：

Wfrest 提供 C++ HTTP 路由和请求响应封装，Workflow 提供异步网络任务、MySQL/Redis 客户端以及
SeriesWork。项目可以把 HTTP 接入、RPC 和存储访问组织成异步任务链，避免为每个连接创建一个阻塞
线程。准确地说，我使用了框架提供的 Reactor 和异步任务能力，并不是在项目中从零实现了 epoll
Reactor。

### 5.2 你怎么理解 Reactor 和 epoll

Reactor 的核心是让事件循环等待 I/O 就绪，再把读写事件分发给对应处理逻辑。Linux 下常用 epoll
监听大量文件描述符，避免为每个连接安排一个长期阻塞线程。非阻塞 I/O 解决的是等待问题，不代表
回调中可以随意执行耗时任务；长时间 CPU 计算或阻塞文件操作仍会影响执行线程。

### 5.3 Workflow 的 SeriesWork 用在什么地方

SeriesWork 适合有先后依赖的异步步骤。例如 RPC 请求中先发 Redis GET，未命中后执行本地查询，再
把 SETEX 追加到同一个 Series。健康检查中多个依赖可以分别创建任务，最后由收尾任务汇总结果。
有依赖关系的步骤不能为了“异步”就全部并行，否则会破坏业务顺序。

### 5.4 Redis 查询失败后为什么不重试

Redis 在这里不是正确性的来源，失败后本地索引可以直接给出结果。自动重试只会增加请求延迟，并在
Redis 故障时放大压力，所以当前请求直接回退本地索引。查询接口本身也不做通用 RPC 重试，因为
重试会占用更多超时预算，而且无法保证下游负载情况。

## 6. SRPC 与 Protobuf

### 6.1 为什么 Gateway 和 Search Server 使用 SRPC

两个进程都使用 C++，SRPC 可以直接基于 Protobuf 生成客户端和服务端代码，减少手写序列化和协议
解析。相比进程间直接复用 HTTP，内部 RPC 契约更加明确，也便于区分网络错误、超时和服务端业务
错误。

### 6.2 Protobuf 中定义了哪些内容

主要包括搜索过滤条件、关键词、分页参数、日志摘要、详情、错误码知识和健康状态。分页和过滤字段
在消息中有明确类型，服务端返回稳定业务码。Protobuf 只负责消息结构和序列化，具体参数范围仍要
在接入边界校验。

### 6.3 RPC 错误如何映射为 HTTP

参考回答：

Gateway 区分三类错误：网络连接失败映射为 503，RPC 超时映射为 504，Search Server 返回的业务
错误则按业务码转换，例如参数错误是 400、日志不存在是 404、索引损坏是 500。这样调用方能判断是
请求问题、依赖暂时不可用，还是服务内部数据异常，而不是所有错误都返回 500。

### 6.4 Protobuf 的兼容性要注意什么

已经发布的字段编号不能随意修改或复用。新增可选字段时使用新的编号，删除字段应保留编号。当前
项目是同版本双进程部署，没有实现复杂的跨多版本兼容层；升级时先保证契约兼容，再按运维顺序替换
Search Server 和 Gateway。

## 7. 增量消费与日志解析

### 7.1 为什么使用 archive_id 作为增量游标

参考回答：

`archive_id` 是一期归档表的单调递增主键，适合作为稳定游标。时间字段可能重复，也可能因为设备
时间和归档时间不同而发生漂移。如果只按时间扫描，边界处容易重复或漏数据。按 `archive_id` 扫描
可以明确记录已经消费到哪个位置。

### 7.2 如何避免一个坏文件影响整个批次

扫描到归档后，每个文件先独立进行元数据、大小、SHA-256 和解析配置校验。无法解析或摘要不一致的
文件记录明确失败码，其他正常文件仍可以进入批次。这样设备产生一个损坏文件不会阻塞整条产线后续
日志，但失败记录不会被伪装成成功。

### 7.3 为什么还要校验一期文件的大小和 SHA-256

一期已经校验过，但二期读取时文件系统仍然是外部边界，文件可能被误删、覆盖或损坏。二期在解析前
重新核对元数据中的大小和 SHA-256，确保索引对应的确实是那份归档。这个校验不是不信任一期逻辑，
而是为了验证跨项目存储边界。

### 7.4 日志解析器怎么设计

当前通过 `parser_profile` 为设备和文件类型指定明确解析器。运行日志使用固定 KV 格式解析时间、等级、
模块、设备和错误码；FCT CSV 解析文件头和每个测试点。遇到未知配置就明确失败，不根据内容猜测
厂商格式，避免错误解析后把不可靠字段写进索引。

### 7.5 为什么每条日志记录都保存 offset 和 length

解析文件时已经知道每条物理记录在原文件中的字节范围。把这个范围存入文档表后，详情查询可以直接
定位原文，不需要重新扫描整个文件，也不用把正文重复写入数据库或索引文件。

### 7.6 如何处理长行、CRLF 和 UTF-8

解析器有最大行长度限制，超过上限的文件明确失败，避免异常设备输出造成无界内存占用。CRLF 的换行
符不计入正文范围，offset 和 length 保持字节级准确。文本还会校验 UTF-8，非法编码不会带着错误
内容继续建索引。

## 8. 倒排索引与 Segment

### 8.1 什么是倒排索引

正向索引是从文档找到文档中的词，倒排索引是从词找到包含它的文档。当前 Segment 中 Term 表保存
词和 Posting 范围，Posting 保存文档局部编号和词频。查询关键词时可以直接读取相关 Posting，而不
需要扫描所有日志正文。

### 8.2 Segment 中保存了什么

每个不可变 Segment 包含 Term、Posting、Document 和 File 四类数据。Document 保存结构化字段、
时间、文档词数、offset 和 length；File 保存 archive_id、相对路径、大小和摘要。正文仍留在一期
归档。manifest 记录格式版本、条目数量、文件大小和 SHA-256。

### 8.3 为什么使用不可变 Segment

不可变文件更容易校验和并发读取。构建完成后不在原地修改，查询线程可以安全持有旧快照；新批次
发布时只需要加载新 Segment 并交换快照。代价是 Segment 数量会逐渐增加，规模进一步扩大时需要
增加离线 Segment 合并，但当前版本没有在缺少实际需求时提前实现这项功能。

### 8.4 如何保证未完成的 Segment 不参与查询

参考回答：

构建先写入 `.building` 临时目录，写完全部文件后执行 fsync，再根据 manifest 完整加载和校验。校验
通过后原子 rename 到正式目录，最后在 MySQL 中把批次改为 READY。内存快照只加载数据库登记为
READY 且文件校验通过的 Segment。因此进程中途退出时，查询仍使用旧快照，不会看到半成品。

### 8.5 为什么还需要 MySQL READY 状态，只有 rename 不够吗

rename 只能证明目录已经原子出现在文件系统中，不能表达这个 Segment 是否已经完成业务登记。
MySQL READY 是查询可见性的权威状态。启动恢复时，正式目录存在但数据库不是 READY 的情况需要按
状态机判断，而不能仅凭目录名称猜测是否可用。

### 8.6 Search Server 重启后如何恢复

启动时先检查目录和依赖，再处理 BUILDING 批次和临时目录，然后查询全部 READY 记录，逐个验证
manifest 和文件摘要，全部成功后构造内存快照。如果任何 READY Segment 缺失或损坏，服务拒绝
启动，不会用空快照继续提供错误结果。

### 8.7 快照交换为什么线程安全

快照本身和其中的 Segment 都是不可变对象，查询线程通过 `shared_ptr` 持有当前版本。发布线程完整
构造新快照后，在短互斥锁内替换 shared_ptr。已经开始的查询继续使用旧对象，新查询获取新对象，
不会读到正在修改的数据。

## 9. 查询算法、BM25 与 Top-K

### 9.1 多关键词为什么采用 AND 语义

故障排查中输入多个关键词通常是为了缩小范围，例如同时搜索 `camera` 和 `timeout`。AND 能确保每个
结果都包含全部关键词，结果更容易解释。如果需要 OR，可以作为明确的新查询类型实现，不能在当前
语义下混用。

### 9.2 为什么先处理低 DF 关键词

参考回答：

DF 是包含某个词的文档数量。DF 越低，Posting 越短、筛选能力越强。AND 查询先拿最短 Posting 作为
候选，再与其他 Posting 求交，可以更早缩小集合，减少比较和后续评分开销。这个优化不改变结果，
只改变执行顺序。

### 9.3 Posting 如何求交

Posting 按文档局部编号有序，可以使用双指针线性求交。当前实现逐个处理按 DF 排序后的 Posting，
每次得到更小的候选集合；候选为空时可以立即结束当前 Segment。

### 9.4 BM25 解决什么问题

BM25 用于估算关键词和文档的相关性。词在当前文档中出现越多，通常分数越高；在所有文档中越少见，
区分度越高；同时会做文档长度归一化，避免长日志因为包含更多词天然占优。当前参数是常见基线
`k1=1.2`、`b=0.75`，没有声称经过真实工厂人工标注调优。

### 9.5 BM25 的核心公式怎么解释

可以口头解释为：IDF 衡量一个词在全局的稀有程度，TF 部分衡量它在当前文档中的出现频率，并通过
`k1` 控制词频收益逐渐饱和；`b` 控制文档长度归一化程度。实际回答不一定要完整背公式，但要能说明
为什么高词频、低文档频率和合适文档长度会影响分数。

### 9.6 为什么还要叠加业务权重

纯正文相关性不一定符合排障优先级。例如用户明确按错误码查询时，错误码精确匹配应该排在只在正文
中偶然出现该字符串的记录前；ERROR 通常也比 INFO 更值得先看。因此排序在 BM25 基础上增加错误码、
模块和等级权重。权重是明确规则，不应伪装成机器学习模型。

### 9.7 为什么使用 Top-K 小顶堆

如果命中很多文档，但页面只需要前 K 个，没有必要把所有候选完整排序。小顶堆最多保存
`offset + page_size` 个当前最优结果，新候选比堆顶更好时替换，复杂度从全量排序的
`O(n log n)` 降为 `O(n log k)`，内存也只保留 K 个命中。

### 9.8 为什么堆容量是 offset + page_size

因为分页到 offset 之后仍要知道前面有哪些结果。例如 offset 为 100、page_size 为 20，就需要先保留
前 120 个，最终再取第 101 到 120 个。项目限制结果窗口不超过 1000，避免深分页不断扩大堆和扫描
成本。

### 9.9 分数相同如何保证结果稳定

排序先按分数降序，再按发生时间降序，最后按 doc_id 降序。这样相同查询和快照会得到稳定顺序，
便于分页、缓存和测试复现。

### 9.10 结构化过滤在什么时候执行

Posting 求交得到候选后，先检查设备、工位、工单、SN、等级、模块、错误码和时间范围，再计算完整
BM25 和维护 Top-K。这样不符合结构化条件的文档不会进入后续评分。没有关键词时则需要遍历文档表
执行结构化筛选，所以接口要求提供精确条件或有限时间范围。

### 9.11 为什么限制最多 8 个关键词和 31 天时间范围

这些限制用于控制单次查询成本。关键词过多会增加 Posting 查找和求交步骤；没有精确字段时查询很长
时间范围会扫描大量文档。限制在 Gateway 和 Search Server 边界都形成明确契约，避免一个请求长期
占用检索资源。

## 10. 原始日志回读与 Linux 文件 I/O

### 10.1 为什么不用 MySQL 保存日志正文

参考回答：

原始日志已经在一期归档中保存，如果再写入 MySQL，会增加存储、备份和复制压力，也会形成两份正文。
二期只保存结构化索引和定位信息，详情查询时根据文件、offset 和 length 回读原始记录。MySQL 主要
保存索引批次状态、归档消费状态、解析配置和错误码知识库。

### 10.2 为什么使用 pread，而不是 read 加 lseek

`pread` 在一次系统调用中指定偏移，不会修改文件描述符的共享文件位置。多个查询并发读取同一个
归档文件时，不需要围绕 `lseek + read` 加锁，也不会因为共享偏移互相干扰。实现仍要处理短读和
`EINTR`，循环读取到指定长度。

### 10.3 为什么不用 mmap 回读单条详情

详情通常只读取一小段连续字节，pread 更直接，也不需要处理页对齐、映射范围和解除映射。mmap 更
适合较大范围或重复随机访问。项目一期在整文件哈希场景使用窗口化 mmap，二期单条日志回读选择
pread，是根据访问模式做的取舍。

### 10.4 如何防止相对路径逃逸归档目录

相对路径来自数据库，但仍按外部存储边界校验。解析时拒绝绝对路径、`..`、符号链接逃逸和非普通
文件，并确认最终真实路径位于配置的 archive_root 下。不能因为路径来自 MySQL 就直接拼接后打开。

### 10.5 如果索引命中但原文件被删除了怎么办

详情接口返回明确的存储错误，而不是返回空正文。运维需要根据 archive_id、相对路径和 SHA-256 从
一期备份恢复文件。二期不会生成替代正文，因为无法保证内容正确。

## 11. MySQL 设计

### 11.1 MySQL 在项目中保存什么

MySQL 保存索引批次、归档消费状态、Segment 发布信息、解析配置和错误码知识库。它不保存倒排索引
正文，也不保存原始日志内容。这样数据库负责持久状态和可管理配置，索引文件负责高频检索。

### 11.2 为什么源数据库和状态数据库分开

源数据库属于 DataStream，LogTrace 只需要读取 `archive_file`；状态数据库属于 LogTrace，用于记录
解析和索引状态。配置会拒绝两个连接指向同一数据库，避免二期误写一期业务库，也便于生产环境分别
授予只读和读写权限。

### 11.3 索引批次有哪些状态

核心状态包括 PARSED、BUILDING、READY 和 FAILED。PARSED 表示解析工件完整；BUILDING 表示正在
生成 Segment；READY 表示 Segment 已发布并可查询；FAILED 保存明确失败原因。状态转换和文件发布
顺序共同决定可见性，不能只改数据库状态跳过文件校验。

### 11.4 为什么迁移脚本要保存 SHA-256

已经执行的迁移如果后来被修改，数据库结构和仓库文件就会失去对应关系。迁移表保存文件摘要，再次
执行时发现摘要变化会明确停止，要求新增迁移，而不是悄悄覆盖历史。

### 11.5 错误码知识库为什么放 MySQL

错误码标题、说明和建议动作是长期维护的结构化数据，需要可审计和可更新，适合放 MySQL。本地 SLRU
缓存热点知识，减少重复查询。错误码对应的日志列表仍来自当前索引快照，不能把知识库记录和搜索
结果混成一个长期缓存对象。

### 11.6 这个项目如何理解 B+ 树索引

MySQL 的 B+ 树适合范围和有序查询，例如按批次状态、archive_id 或错误码查记录。倒排索引解决的是
正文关键词到文档集合的映射，二者用途不同。不能用一个普通 B+ 树联合索引替代全文倒排结构。

### 11.7 项目中有没有复杂事务和行锁

项目使用数据库原子更新保证批次发布状态，但没有设计复杂的长事务或高冲突扣减业务。面试时可以
解释事务隔离和行锁原理，但不应把未实现的复杂锁竞争说成项目经验。

## 12. Redis Cache-Aside

### 12.1 Redis 缓存什么

Redis 只缓存规范化查询对应的一页 doc_id、分数和 total_hits，也缓存真正的空结果。日志正文、完整
Document 和 Segment 不放 Redis。命中后仍从同一内存快照恢复结构化摘要。

### 12.2 为什么缓存 doc_id 列表，而不是整个 HTTP 响应

HTTP 响应包含接入层格式和请求标识，直接缓存会把协议层和索引层绑在一起。缓存 doc_id 和排序信息
更接近 Search Server 的业务结果，并且可以从当前快照恢复摘要。正文不缓存到 Redis，避免单个缓存值
和重复存储。

### 12.3 Redis Key 如何生成

先规范化关键词、等级和过滤条件，再序列化查询内容并计算 SHA-256。Key 还包含命名空间、格式版本
和当前最大 READY batch_id。使用摘要可以避免 Key 过长，也避免在 Redis Key 和运维输出中直接暴露
完整查询内容。

### 12.4 为什么 Key 必须包含快照版本

参考回答：

新 Segment 发布后，相同查询可能增加命中。如果 Key 不带版本，旧缓存就会覆盖新索引结果。版本
进入 Key 后，新快照自然使用新 Key，旧 Key 按 TTL 过期，不需要在线扫描 Redis 批量删除。

### 12.5 为什么缓存空结果

不存在的错误码或关键词组合可能被重复查询。如果每次都穿透到倒排索引，就会产生无效重复计算。
缓存空结果可以缓解缓存穿透，但 TTL 比非空结果短，避免新日志进入索引后长时间看不到结果。

### 12.6 为什么近期和历史查询使用不同 TTL

最近两小时仍可能有新日志进入，结果变化快，所以非空结果缓存 30 秒、空结果 10 秒。已经结束的
历史时间范围变化较少，因此非空结果缓存 10 分钟、空结果 5 分钟。TTL 是当前业务基线，生产环境
应根据日志延迟和查询量调整。

### 12.7 Redis 挂了为什么还能查询

参考回答：

Redis 缓存的是可重建结果，不是权威数据。GET 失败、返回损坏值或 SETEX 失败时，当前请求直接执行
本地倒排索引。这样 Redis 故障只影响延迟和 CPU，不影响结果正确性。项目没有用旧结果、空结果或
通用重试掩盖故障。

### 12.8 如何处理缓存穿透、击穿和雪崩

空结果缓存用于缓解穿透；快照版本和不同 TTL 限制旧结果影响。当前单机仿真没有实现热点 Key 的
互斥重建或 TTL 随机抖动，因为还没有证据表明存在大规模击穿和雪崩。若生产监控发现大量并发请求
集中重建同一个 Key，可以限制为同一时刻只由一个请求负责重建，其他请求等待结果；若大量 Key 同时
过期，再考虑让 TTL 在小范围内错开，但不能提前堆叠没有
验证需求的复杂度。

### 12.9 为什么不缓存日志正文

正文可能较大，而且已经存在一期归档。本地详情缓存只保存大小不超过配置上限的热点记录，Redis 则
完全不缓存正文，避免单个缓存值过大、网络开销增加和内存不可控。

## 13. SLRU 本地缓存

### 13.1 SLRU 和普通 LRU 有什么区别

参考回答：

普通 LRU 中，新访问的历史数据会直接挤占热点空间。SLRU 分为 Probation 和 Protected，新条目先
进入 Probation，只有再次命中才晋升 Protected。Protected 满时把最久未使用项降回 Probation，
Probation 满时再淘汰。这样一次性历史扫描不容易污染真正热点。

### 13.2 本地缓存了哪些内容

缓存热点日志详情、文件元数据和错误码知识。详情和文件定位的 Key 包含快照版本，避免重建索引后
复用旧定位；错误码知识本身不依赖快照。超过 `max_detail_bytes` 的正文不进入详情缓存。

### 13.3 SLRU 如何保证线程安全

缓存内部使用互斥锁同时保护哈希表、Probation 链表、Protected 链表以及晋升、降级和淘汰操作。
不能只给哈希表加锁，因为一次命中会同时修改链表位置和分区状态。测试使用多个线程并发 get/put，
验证容量不会超过两个分区上限。

### 13.4 为什么使用 list 加 unordered_map

双向链表适合 O(1) 移动和删除 LRU 节点，哈希表负责 O(1) 平均复杂度定位 Key。哈希表条目保存链表
迭代器，命中时可以直接把节点移动到分区头部，不需要遍历链表。

### 13.5 SLRU 容量怎么配置

容量按条目数限制，开发配置中 Protected 大于 Probation，让重复热点获得更多空间。条目数不是完整
内存上限，所以详情还增加正文大小限制。生产配置应结合平均对象大小和进程内存预算调整。

## 14. C++11 与并发

### 14.1 项目中使用了哪些 C++11 特性

使用了智能指针管理不可变 Segment 和快照生命周期，Lambda 作为 Workflow 回调，移动语义减少临时
对象复制，`std::thread`、`std::mutex` 和原子变量用于后台索引及缓存测试，STL 容器实现倒排表、
查找表和 Top-K。使用这些特性是为了解决具体生命周期和并发问题，不是为了刻意展示语法。

### 14.2 为什么快照使用 shared_ptr

查询开始后需要保证整个快照在请求结束前仍然存在。发布线程替换全局指针时，旧查询手中的
shared_ptr 继续持有旧快照，最后一个查询结束后对象自动释放。这比手工引用计数或全局长锁更安全。

### 14.3 项目如何使用 RAII

文件描述符、哈希上下文、锁和智能指针都遵循作用域管理。互斥锁使用 lock_guard，异常退出时也会
自动解锁；OpenSSL 上下文通过带自定义释放器的智能指针管理；临时对象离开作用域自动释放，减少
手工清理分支。

### 14.4 有没有自己实现线程池

没有。网络和异步任务调度使用 Workflow 提供的执行模型，后台增量索引使用明确的工作线程和停止
协议。面试中可以解释线程池的任务队列、线程复用、条件变量和安全退出原理，但不能说项目中从零
实现了通用线程池。

### 14.5 如何让后台索引线程安全退出

停止时先设置退出状态并唤醒等待，再 join 工作线程，确保线程不再访问已经销毁的依赖。服务停止顺序
是先停索引工作线程，再停 RPC 服务，避免对象析构期间仍有后台任务运行。

## 15. 错误处理与故障排查

### 15.1 Redis、MySQL 和索引故障分别怎么处理

- Redis 查询缓存故障：回退本地索引，结果仍正确；健康检查显示依赖异常；
- 状态 MySQL 故障：错误码首次查询和新索引发布失败，启动探测失败时拒绝启动；
- READY Segment 损坏：Search Server 拒绝启动，运维核对归档后执行明确重建；
- 一期原文缺失：搜索摘要仍可能存在，但详情返回存储错误，必须从一期备份恢复。

### 15.2 为什么 READY Segment 损坏时不跳过

数据库已经声明它属于可查询快照。如果静默跳过，服务会返回不完整结果，而且调用方无法知道数据
缺失。明确拒绝启动能让故障尽早暴露，运维再根据批次和 archive_id 重建。

### 15.3 构建过程中进程退出会怎样

旧快照不受影响。临时 `.building` 目录不会进入查询；重启后根据批次状态和正式目录判断恢复或清理。
如果 rename 已完成但数据库还处于 BUILDING，可以校验正式 Segment 后完成恢复；不能直接把未知
目录标记为 READY。

### 15.4 查询突然变慢怎么排查

我会先看请求是否缺少精确条件、时间范围是否过大、分页是否接近 1000；再看 Redis 是否可用和是否
因快照刚更新导致缓存未命中；然后检查 Segment 数、候选文档数、CPU、磁盘和 RPC 延迟。不能看到
延迟就直接加缓存或重试，需要先确认瓶颈在接入、缓存、索引还是原文 I/O。

### 15.5 HTTP 502、503、504 分别可能表示什么

502 通常表示 RPC 返回无法正确处理的上游错误，503 表示 Search Server 或依赖暂时不可用，504 表示
Gateway 等待 RPC 超时。具体仍要结合响应业务码和服务日志，不能只根据 HTTP 状态猜测。

### 15.6 为什么日志中要有 request_id

一个 HTTP 请求会跨 Gateway、SRPC 和 Search Server。request_id 可以关联两侧日志，区分同一时刻
的多个查询。它用于排查调用链，不应包含 Token、密码、原始正文等敏感信息。

## 16. 性能测试与容量边界

### 16.1 做过哪些性能验证

项目有固定 100 万文档内存负载，使用 8 个并发查询验证 Top-K 结果一致。本机一次记录约 232 到
237 毫秒，峰值内存约 435 MiB。这个测试主要验证算法和并发稳定性，不包含真实磁盘冷启动、长文本
分词、网络和多用户竞争。

### 16.2 能不能说系统支持百万级生产日志

可以说“项目使用 100 万文档固定负载做过本机回归”，不能直接说“生产环境稳定支持百万 QPS”或
“已经验证工厂容量”。文档数量、查询并发、日志长度、Segment 数和磁盘条件都会影响结果，生产
容量需要在真实硬件和业务负载下重新测试。

### 16.3 当前主要性能瓶颈可能在哪里

关键词少且过滤弱时，候选文档较多，BM25 和结构化过滤会增加 CPU；没有关键词的条件查询可能遍历
较多文档；Segment 太多会增加逐段查找；Redis 不可用时重复热点查询会回到本地索引；详情请求则受
归档磁盘随机读取影响。瓶颈需要通过指标和压测确认。

### 16.4 如果数据量继续增长，准备怎么优化

优先根据实际指标选择方案：Segment 数过多可以做离线合并；单机内存不足可以增加 mmap 索引和按需
加载；查询压力过高可以做只读副本或分片；复杂检索需求可以迁移到成熟搜索引擎。任何扩展都要先
重新定义快照一致性、缓存版本和故障恢复，不能只把进程数量加上去。

## 17. 构建、测试和部署

### 17.1 CMake 在项目中负责什么

CMake 管理 C++11 编译选项、Protobuf/SRPC 代码生成、第三方库链接、Debug/Release、集成测试、
覆盖率和 Sanitizer 开关。生产代码使用 `-Wall -Wextra -Wpedantic -Werror`，让编译告警在开发阶段
直接暴露。

### 17.2 项目有哪些测试

单元测试覆盖解析、Segment 格式、倒排索引、BM25、Top-K、SLRU 和配置；集成测试连接真实 MySQL、
Redis；E2E 启动 Search Server 和 Gateway 验证 HTTP/RPC；跨项目 E2E 还会真实启动一期 Collector
和 DataStream。终板 Debug/Release 各 46 项 CTest 通过，跨项目 E2E 独立验证了 1 条归档日志到
HTTP 命中的完整链路。

### 17.3 如何做内存和未定义行为检查

Sanitizer 构建运行 42 项非集成测试，ASan 和 UBSan 没有报告错误。Valgrind 对同一单元测试程序执行
完整检查，结果为 0 errors，确定、间接和可能泄漏均为 0。Protobuf 在退出时有 still reachable 的
全局描述符内存，但不属于泄漏错误。

### 17.4 覆盖率是多少

终板核心源码行覆盖率是 82.3%。覆盖率用于发现未测试分支，不代表代码一定正确，所以还需要真实
MySQL/Redis、损坏文件、进程中断和跨项目 E2E。

### 17.5 生产环境如何部署

Search Server 和 Gateway 使用独立 systemd 单元，Nginx 在前面终止 TLS。Search Server 对一期
归档目录只有读权限，对二期索引和日志目录有写权限。密码和 Operator Token 从受控环境文件注入，
不写入 JSON 或仓库。启动顺序是 MySQL/Redis、DataStream、Search Server、Gateway、Nginx。

### 17.6 如何备份和恢复

一期负责原始归档备份；二期备份状态数据库和索引目录，并生成 SHA-256 清单。恢复时先校验备份，
再恢复索引、导入状态库，启动 Search Server 验证全部 READY Segment，最后启动 Gateway。Redis 查询
缓存不恢复，因为可以从索引重建。

### 17.7 Git 在项目中怎么使用

按阶段拆分功能和测试，修改前检查工作树，避免覆盖无关改动；配置示例只保存环境变量名，真实 env
文件加入忽略规则；提交前执行格式、构建和测试。回答时应结合自己真实使用的分支和评审流程，不要
虚构仓库中不存在的 CI 平台或团队规范。

## 18. 常见追问与取舍

### 18.1 为什么错误码查询还要返回最近日志

错误码说明只能提供通用知识，工程师还需要结合当前设备和时间的实际记录。接口返回知识库内容和
最近匹配日志，可以从说明快速跳到现场证据。匹配日志仍受当前索引快照控制，不和知识库长期绑定。

### 18.2 为什么不对旧 Redis Key 主动删除

快照版本已经隔离新旧结果，旧 Key 不再被访问，并会按 TTL 自然过期。主动遍历删除会增加 Redis
负载，还需要处理并发发布，没有实际收益。

### 18.3 为什么没有实现模糊查询和通配符

当前业务基线是规范化 term 的 AND 查询，结果和成本容易解释。模糊查询需要额外字典结构、编辑距离
或 n-gram 索引，也会显著扩大候选集合。没有明确需求前不引入，避免把学习项目做成不可控的通用
搜索引擎。

### 18.4 为什么没有自动修复损坏索引

自动删除或重建可能掩盖归档损坏、磁盘问题或状态不一致。当前做法是明确失败并提供按 archive_id
重建命令，让运维先确认一期归档完整，再执行可审计操作。

### 18.5 如果同时发布新快照和执行查询，会不会混用数据

不会。一次查询开始时获取一个不可变快照 shared_ptr，整个查询和 Redis 恢复都基于该版本。新快照
发布只影响之后开始的查询。Redis Key 也包含版本，因此不会把旧 ID 列表恢复到新快照。

### 18.6 项目中哪些能力不是自己从零实现的

HTTP 路由由 Wfrest 提供，异步网络、MySQL/Redis 客户端和 Reactor 由 Workflow 提供，RPC 框架由
SRPC 提供，序列化由 Protobuf 提供，SHA-256 使用 OpenSSL。项目实现的是业务接口、任务编排、索引
格式、检索算法、缓存策略和故障处理。这样表述更准确，也更容易体现对边界的理解。

## 19. 回答时需要避免的表达

- 不说“从零实现了 epoll、Reactor、RPC 框架”，应说“基于 Workflow/Wfrest/SRPC 完成业务实现”；
- 不说“Redis 挂了系统完全没有影响”，应说“结果仍正确，但延迟和 CPU 压力可能上升”；
- 不说“BM25 能保证最准确”，应说“它提供可解释的相关性基线，并叠加了明确业务权重”；
- 不说“系统支持百万并发”，应说“完成了 100 万文档、8 并发的本机固定负载验证”；
- 不把 FCT 测试点 NG 说成设备 ERROR 日志，应保留真实来源类型；
- 不把 DataStream 的上传、HMAC 和分片能力说成 LogTrace 内部功能；
- 不声称生产环境已经使用 Elasticsearch、Kafka、Kubernetes 或分布式索引；
- 不用“绝对不会”“完全解决”“零延迟”等无法证明的表达。

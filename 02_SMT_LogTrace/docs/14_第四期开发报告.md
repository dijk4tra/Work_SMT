# 第四期开发报告

完成日期：2026-07-14。

## 1. 阶段结论

第四期“检索算法与 HTTP/RPC 业务链”已完成实现与复验，当前状态为“复验通过，待用户确认”。
本期把第三期 READY 快照转换为可对外使用的日志检索能力，并保持 Gateway 参数校验与 Search Server 索引执行分层。

## 2. 检索算法

- 使用与建索一致的 tokenizer 规范化关键词，去重后执行 AND；
- 每个 Segment 按文档频率升序处理 Posting，提前缩小候选集；
- BM25 使用全快照文档数、跨 Segment DF 和全局平均文档长度；
- 排序叠加错误码、模块精确命中以及 ERROR/WARN 等级权重；
- 小顶堆只保留 `offset + page_size` 个最优候选，避免全量排序；
- 稳定顺序为分数降序、发生时间降序、doc_id 降序。

BM25 参数固定为 `k1=1.2`、`b=0.75`。这是工程基线，不声称已经真实工厂查询日志或人工标注调优。

## 3. HTTP 与 RPC

Protobuf/SRPC 新增 `SearchLogs`、`ListAnomalies`、`GetLogDetail` 和 `GetErrorCode`。Gateway 实现：

- `POST /api/v1/logs/search`；
- `GET /api/v1/logs/anomalies`；
- `GET /api/v1/logs/{doc_id}`；
- `GET /api/v1/error-codes/{code}`。

业务路由使用从 `SMT_LOGTRACE_OPERATOR_TOKEN` 加载的部署级 Bearer Token，并使用 OpenSSL
`CRYPTO_memcmp` 常量时间比较。只读 RPC 保持零重试，区分超时、连接不可用和服务端业务错误。

## 4. 边界约束

- 最多 8 个关键词，单个最长 64 字节；
- `page_size` 为 1–200，`offset + page_size <= 1000`；
- 没有精确设备、工单、产品 SN 或错误码时，必须提供时间范围；
- 单次时间范围最长 31 天；
- 异常日志定义为 WARN、ERROR 或非空错误码；
- 详情仍从一期归档 `pread`，不将正文复制到 MySQL。

## 5. 错误映射

已实现 `INVALID_ARGUMENT`、`OPERATOR_TOKEN_INVALID`、`LOG_NOT_FOUND`、
`ERROR_CODE_NOT_FOUND`、`INDEX_CORRUPTED`、`STORAGE_IO_ERROR`、`MYSQL_UNAVAILABLE`、
`SEARCH_RPC_UNAVAILABLE` 和 `SEARCH_RPC_TIMEOUT` 到 HTTP 400/401/404/500/502/503/504 的稳定映射。

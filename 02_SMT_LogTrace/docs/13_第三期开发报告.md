## 3. Segment 固定格式

每个批次发布到 `index_root/segments/segment_<batch_id>/`：

```text
manifest.json
terms.bin
postings.bin
documents.bin
files.bin
```

`manifest.json` 保存格式版本、批次、源归档范围、源文件数、成功解析文件数、文档数、Term/Posting 数、平均文档长度、
时间范围以及四个二进制文件的大小和 SHA-256。加载器拒绝未知字段、缺失字段、统计不一致、摘要不符和时间范围非法。

四个二进制文件均使用“8 字节 magic + 4 字节小端版本 + 8 字节记录数”文件头：

| 文件 | magic | 记录内容 |
|---|---|---|
| `files.bin` | `LTFIL001` | archive_id、文件大小、SHA-256、相对路径 |
| `documents.bin` | `LTDOC001` | doc_id、文件序号、字节范围、时间和结构化字段 |
| `postings.bin` | `LTPST001` | local_id 和 term frequency |
| `terms.bin` | `LTTRM001` | term、document frequency 和连续 posting 范围 |

变长字符串使用 4 字节小端长度和原始 UTF-8 字节。加载器拒绝错误 magic/版本、截断、尾随数据、越界索引、非连续文档号和非法 Posting 范围。

`doc_id` 的高 32 位为 `batch_id`，低 32 位为 Segment 内连续 `local_id`，因此批次发布后文档标识稳定。

## 4. 原子发布与数据库状态

构建流程为：

1. 锁定最早的 `PARSED` 批次并将其置为 `BUILDING`；
2. 严格加载解析工件，并核对一期原始归档；
3. 在 `segments/.building/segment_<batch_id>` 写入全部文件；
4. 对文件和目录执行持久化，再用加载器完整校验临时 Segment；
5. 将临时目录原子 `rename` 为正式目录；
6. 用单条 MySQL 多表 `UPDATE` 同时写入批次 `READY`、Segment 定位与摘要，并把该批次成功归档置为 `INDEXED`；
7. 重新加载全部 READY Segment，全部成功后一次替换查询快照。

`003_ready_segment.sql` 增加约束：`PARSED/BUILDING/READY` 必须保留解析工件路径与摘要；`READY` 还必须保留 Segment 名称、摘要和发布时间。

## 5. 中断恢复矩阵

| 中断位置 | 数据库/文件状态 | 恢复处理 | 查询影响 |
|---|---|---|---|
| 临时 Segment 写入中 | `BUILDING` + `.building` | 启动清理临时目录，批次恢复为 `PARSED` | 不可见，旧快照不受影响 |
| `rename` 后、READY 前 | `BUILDING` + 正式 Segment | 重新加载并校验，成功则补发布，失败则删除并记录失败 | 未 READY 前不可见 |
| 补发布时 MySQL 不可用 | `BUILDING` + 已校验正式 Segment | 保留文件和状态并让启动明确失败，下次启动继续补发布 | 未 READY 前不可见 |
| READY 后、快照交换前 | READY + 正式 Segment | 重启或下次刷新时加载 | 当前进程仍使用旧快照 |
| 正式 Segment 未在数据库登记 | 孤儿目录 | 记录告警但不加入快照 | 不可见 |
| READY Segment 缺失或损坏 | 数据库声明可查但实体无效 | Search Server 启动失败，不使用不完整快照 | 明确不就绪 |

## 6. 快照与原文回读

快照加载器先对全部 READY Segment 完成数据库摘要、manifest 和四个二进制文件的校验，只有全部成功才在短锁内替换
`shared_ptr<const IndexSnapshot>`。新 Segment 加载失败时，旧快照仍然完整可用，不会暴露部分刷新结果。

Segment 只保存文档定位和结构化字段，不保存完整正文。`IndexSnapshotStore::readOriginal` 通过文档表取得文件序号与字节范围，
通过文件表取得一期归档相对路径，安全解析后循环执行 `pread`，返回不包含 LF/CRLF 行结束符的原始字节。

## 8. 第四阶段边界

第三期交付的是可验证、可恢复、可原子加入快照的索引数据，不是对外搜索业务链。用户确认本阶段后，第四阶段已实现：

- 多关键词低 DF 优先 AND 求交与结构化过滤；
- BM25 和业务权重排序；
- Top-K 小顶堆、分页上限与稳定排序；
- `SearchLogs`、`ListAnomalies`、`GetLogDetail`、`GetErrorCode` RPC；
- 对应 HTTP 接口、Operator Token 和错误映射。

第四阶段的最终实现与复验证据见 `docs/14_第四期开发报告.md`。

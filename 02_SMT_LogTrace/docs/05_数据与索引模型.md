# 数据与索引模型

## 1. MySQL 状态表

### 1.1 `index_batch`

记录每次增量构建的归档范围。第二阶段使用 PARSING/PARSED/FAILED，保存解析工件路径与摘要；
第三阶段再使用 BUILDING/READY 并保存 Segment 名称与摘要。只有 READY 批次可以进入查询快照。

### 1.2 `indexed_archive`

以一期 `archive_id` 为主键，记录解析器、版本、PENDING/PARSED/INDEXED/FAILED 状态、失败行号和
所属批次。失败归档不会阻塞后续扫描，但必须保留原因；解析器升级后通过明确重建命令重新置为
PENDING。

### 1.5 第二阶段解析工件

解析成功批次原子写入 `index_root/parsed/batch_<batch_id>/`：

```text
manifest.json
archives.jsonl
documents.jsonl
```

工件只保存归档元数据、结构化文档字段和 `offset/length`，不保存完整原文。第三阶段从 PARSED
批次读取这些字段，并按字节范围回读一期归档构建 Segment。工件可删除重建，不属于原始业务事实。

### 1.3 `parser_profile`

按 `device_id + file_type` 选择固定解析器。首版只有 `kv_runtime_v1` 与 `fct_csv_v1`，不使用
未知格式自动探测。

### 1.4 `error_code_catalog`

保存错误码、模块、标题、说明和建议动作。它是运维知识，不参与原始日志真实性判断；日志中的
错误码即使未录入知识库也必须能够被检索。

## 2. 文档模型

每个文档至少包含：

- `doc_id`：由批次编号和 Segment 内局部编号组成的稳定 64 位值；
- `archive_id`、`byte_offset`、`byte_length`；
- `occurred_at`、`archived_at`；
- `line_id`、`station_id`、`device_id`、`collector_id`；
- `work_order`、`product_sn`；
- `source_type`、`level`、`module_name`、`error_code`、`event_name`；
- `term_count` 和解析状态。

## 3. Segment 文件

目标 Segment 目录：

```text
index_root/segments/segment_<batch_id>/
├── manifest.json
├── terms.bin
├── postings.bin
├── documents.bin
└── files.bin
```

`manifest.json` 保存格式版本、批次、源归档范围、文档数、平均长度、时间范围和各文件摘要。
二进制文件使用固定版本布局和显式大小检查，加载时发现截断、越界或版本不符则拒绝整个 Segment。

### 3.1 固定二进制版本 v1

四个二进制文件都使用 8 字节 magic、4 字节小端版本和 8 字节记录数。变长字符串使用
4 字节小端长度和原始 UTF-8 字节，加载器必须消费完整文件，不允许尾随数据。

| 文件 | magic | 核心记录 |
|---|---|---|
| `files.bin` | `LTFIL001` | archive_id、文件大小、SHA-256、相对路径 |
| `documents.bin` | `LTDOC001` | doc_id、file ordinal、offset/length、时间、结构化字段 |
| `postings.bin` | `LTPST001` | local_id 与 term frequency |
| `terms.bin` | `LTTRM001` | term、document frequency 和连续 posting 范围 |

`doc_id` 的高 32 位为 batch_id，低 32 位为 Segment 内连续 local_id。首版每批最多 100,000 个文档，
并明确拒绝无法用该版本表示的批次编号或计数。

## 4. 倒排数据

Term 字典保存 term、全局/段内文档频率和 posting 范围。Posting 保存局部文档编号与词频，按文档
编号递增。AND 查询先从低 DF posting 开始求交，再执行结构化过滤和 BM25 评分。

第三阶段已完成 term 规范化、字典和 posting 生成，但不在本阶段执行 AND 求交、BM25 或 Top-K。
搜索执行和排序仍属于第四阶段。

## 5. 原文回读

详情查询从文档取得 `archive_id + offset + length`，从文件表取得相对路径，并执行：

1. 安全组合配置的 `archive_root` 与相对路径；
2. `open(O_RDONLY | O_CLOEXEC)`；
3. 校验普通文件及范围未越界；
4. 循环 `pread` 读取精确字节数；
5. 关闭 FD，并返回 UTF-8 原文。

索引不保存完整正文，MySQL 也不保存正文。
第三阶段的 `IndexSnapshotStore::readOriginal` 已实现该链路，并通过 CRLF 与普通 LF 样本验证
返回字节与一期归档完全一致。

## 6. 发布一致性

| 中断位置 | 恢复规则 |
|---|---|
| 临时 Segment 写入中 | 未 rename，不可查询；启动清理 `.building` 并把 BUILDING 恢复为 PARSED |
| rename 后、MySQL READY 前 | 加载和摘要校验通过后补发布 READY；损坏则删除并标记失败 |
| rename 后、补发布时 MySQL 不可用 | 保留 BUILDING 和已校验正式 Segment，启动失败；下次启动继续补发布 |
| MySQL READY 后、内存加载前 | 重启或下一刷新从 READY 批次加载 |
| 缓存仍含旧结果 | Key 带快照版本，不会命中新快照 |

查询快照在所有 READY Segment 全部加载成功后一次交换 `shared_ptr`。新 Segment 损坏或刷新失败时，
旧快照仍然完整可用，不会暴露部分加载状态。正式目录存在但数据库未 READY 的孤儿 Segment
不加入快照，启动时记录告警。

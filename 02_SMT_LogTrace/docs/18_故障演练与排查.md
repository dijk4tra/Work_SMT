# 故障演练与排查

## 1. Redis 运行期不可用

预期：健康检查变为未就绪，但已建立的业务请求仍可从本地 READY 索引得到正确结果；只增加延迟，
不返回缓存中的损坏数据。恢复 Redis 后新查询重新写入短 TTL Key，不执行批量补热。

## 2. MySQL 不可用

状态库不可用会影响错误码首次查询、增量扫描和新 Segment 发布；现有不可变快照仍保留在内存。
启动阶段 MySQL 不可用时服务拒绝启动。先恢复数据库，再核对 `index_batch` 和 `indexed_archive`，
不得直接修改状态绕过恢复流程。

## 3. Segment 缺失或损坏

manifest 大小或 SHA-256 不符时 Search Server 拒绝加载。隔离损坏目录，核对一期归档仍完整后，使用
`logtrace_admin rebuild --archive-id <id>` 重建；不要复制其他批次文件或伪造摘要。

## 4. 一期归档正文缺失

索引摘要仍可能命中，但详情 `pread` 返回存储错误。先停止一期清理任务，依据 `archive_id`、相对路径
和 SHA-256 从一期备份恢复正文，再验证详情。二期不得生成替代正文。

## 5. 查询延迟升高

依次检查时间范围和分页、Redis 命中、候选文档数、Segment 数量、CPU 和磁盘。热点查询缺失时确认
快照版本是否刚变化；大量旧 Key 由 TTL 自然淘汰，不在线执行全库 `KEYS` 或批量删除。

# SMT 检测设备日志索引与故障检索系统开发思路

## 1. 项目定位

一期 DataStream 已解决原始业务文件可靠归档问题，但运维人员仍需要登录服务器并逐个打开日志，
无法快速回答“某错误码在哪些设备出现”“某产品经过哪些异常”“同类故障最近是否集中发生”等
问题。二期在不改变一期归档事实的前提下建立可检索派生数据。

系统只处理已经归档且不可变的文件。索引可以删除和重建，MySQL 索引状态与本地 Segment 都不是
原始业务事实；一期 MySQL 元数据和正式文件才是重新构建索引的来源。

## 2. 首版业务边界

首版处理：

- SPI、AOI、FCT 和 IPC 的 `RUNTIME_LOG`；
- FCT 的 `TEST_REPORT`；
- 错误码查询、关键词 AND 检索、异常日志列表和日志详情；
- 产线、工位、设备、工单、产品 SN、时间、等级、模块和错误码过滤。

首版不处理：

- NG 图片、ZIP、PDF 和检测结果 JSON 的内容检索；
- OR、NOT、短语、正则、通配符、拼写纠错和中文分词；
- 多实例分布式索引、分片副本、跨机房复制和在线 Segment 合并；
- 告警推送、工单流转、Web 前端和复杂 RBAC。

## 3. 进程职责

### 3.1 `logtrace_gateway`

- 使用 Wfrest 提供 HTTP 接口；
- 完成外部参数、认证和请求大小校验；
- 把请求转换为 Protobuf；
- 通过 SRPC 调用检索服务；
- 区分网络错误、RPC 超时、协议错误和服务端业务错误。

### 3.2 `logsearch_server`

- 只读扫描一期 `archive_file`；
- 读取归档文件并按已配置解析器生成文档；
- 构建不可变索引 Segment 并原子发布查询快照；
- 执行结构化过滤、倒排求交、BM25 和 Top-K；
- 使用 `pread` 回读原始记录；
- 管理本地 SLRU 和 Redis 查询缓存。

索引任务与检索 RPC 处于同一进程，但索引构建使用独立有界执行器，查询只读取不可变快照。当前
三线单实例模型没有独立扩缩容需求，因此不再拆第三个 Indexer 进程。

## 4. 上游契约

索引器使用只读账号查询：

```sql
SELECT archive_id, line_id, station_id, device_id, collector_id,
       work_order, product_sn, file_type, original_filename,
       relative_path, file_size, file_sha256, produced_at, archived_at
FROM smt_datastream.archive_file
WHERE archive_id > ? AND file_type IN ('RUNTIME_LOG', 'TEST_REPORT')
ORDER BY archive_id ASC
LIMIT ?;
```

`archive_id` 是稳定增量游标。`relative_path` 只能与受控 `archive_root` 组合，不能接受 HTTP 输入。
索引器顺序读取文件时同步计算 SHA-256，与一期元数据比较，避免在源文件损坏时发布错误索引。

## 5. 文档与索引

一条物理日志行或一条 FCT 测试点记录对应一个文档。文档保存结构化字段、词数、原归档编号和
字节范围。倒排表从规范化 term 指向包含该 term 的 posting list，posting 保存文档编号和词频。

每批增量数据生成一个不可变 Segment：

1. 创建 MySQL BUILDING 批次；
2. 在 `index_root/.building/<batch_id>` 写入临时文件；
3. 校验文件布局、统计信息和摘要；
4. `fsync` 后原子 `rename` 为正式 Segment；
5. MySQL 事务把批次与成功归档标记为 READY/INDEXED；
6. 查询服务加载 Segment，并以短锁交换 `shared_ptr` 快照。

临时目录或正式 Segment 存在但 MySQL 未 READY 时，查询不会加载它。MySQL 已 READY 但进程尚未
加载时，重启或下一次刷新会读取该 Segment，不需要通用分布式事务框架。

## 6. 查询与排序

- 关键词统一小写并去除无意义分隔符，保留设备号和错误码中的数字、下划线和连字符；
- 多关键词只支持 AND；按全局文档频率从低到高求交；
- BM25 使用词频、逆文档频率和文档长度归一化；
- 错误码、模块精确匹配和 ERROR/WARN 等级作为额外业务权重；
- 遍历候选时维护 K 大小的小顶堆，最终只排序堆中结果；
- 页大小和最大结果窗口有明确上限，不支持无界深分页。

业务权重是仿真初始值，必须通过固定相关性样本和查询回归验证，不能声称已经由真实工厂调优。

## 7. 缓存

本地 SLRU 分 Probation 与 Protected：日志详情、错误码知识和文件元数据首次进入 Probation，
重复命中后晋升 Protected，降低一次性历史扫描对稳定热点的污染。

Redis 只缓存规范化查询对应的日志 ID 列表和空结果。Key 包含已发布最大批次编号，索引更新后旧
Key 不会命中新快照。近期范围使用短 TTL，已结束历史范围使用长 TTL。Redis 不可用时直接查询
本地索引，不能把缓存故障变成错误结果。

## 8. 安全与部署边界

- Nginx 对外终止 TLS，Gateway 只监听回环或受控内网；
- Search RPC 默认只监听回环地址；
- 一期数据库账号只授予 archive_file 和必要主数据的 SELECT；
- 一期归档目录以只读方式挂载给 Search Server；
- 二期数据库账号只能访问 `smt_logtrace`；
- 日志不输出 Token、数据库密码、查询正文或原始日志全文。

## 9. 发布原则

项目按六期连续生成。每期必须有可运行代码、测试和开发报告；只有六期全部完成后才能把仓库
标记为完整复现简历中的日志检索项目。

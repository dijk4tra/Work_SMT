# 日志检索系统实际业务差距与改进思路

## 1. 当前可证明范围

项目使用一期仿真归档和人工构造的错误码知识库，能够验证索引、RPC、缓存和故障恢复，但不能
证明已经兼容某家设备厂商的真实日志语法、错误码含义或生产容量。

## 2. 主要差距

### 2.1 厂商格式

真实 SPI/AOI/FCT 日志可能使用 GBK、二进制记录、多行堆栈或专有 CSV。首版只支持明确配置的
UTF-8 `kv_runtime_v1` 和 `fct_csv_v1`，未知格式记录失败，不自动猜测。

### 2.2 错误码知识

仿真错误码只用于学习。生产接入需要设备、工艺和运维共同确认错误码版本、适用机型、处置步骤
和失效时间，不能把代码字符串直接当成维修结论。

### 2.3 相关性权重

BM25 参数和业务加权需要真实查询日志、人工标注或故障工单评估。首版固定样本只能证明算法实现
和排序稳定，不能证明权重最优。

### 2.4 索引生命周期

首版按时间和批次生成不可变 Segment，不实现在线合并。单机试点可以通过时间范围裁剪控制查询
段数；当长期 Segment 数量成为瓶颈后，再基于压测引入离线合并。

### 2.5 高可用

单 Search Server、本地索引和只读归档挂载存在单点。生产需要根据恢复时间目标决定是否增加冷备
索引、主备 Search Server 或共享只读索引，不为了展示技术栈提前实现分布式一致性。

## 3. 生产校准输入

- 每种设备型号的日志样本、编码、轮转和版本变化；
- 日志日增量、保留周期和最常见查询时间范围；
- 实际错误码表、模块归属和维修动作；
- 运维人员常用查询与可接受响应时间；
- 一期归档目录的挂载、备份和恢复方式；
- Redis 故障时允许的性能下降程度；
- 索引重建时间目标和允许丢失的派生数据窗口。

## 4. 当前取舍

首版坚持两进程、单实例、不可变 Segment、只读上游和明确解析器。该方案能够覆盖简历中的技术
事实，并适合三线仿真和中小规模试点；不对任意厂商或海量日志作无证据承诺。

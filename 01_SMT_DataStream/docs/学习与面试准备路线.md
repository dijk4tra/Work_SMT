# 项目学习与面试准备路线

## 1. 学习目标

本仓库既要生成可运行代码，也要让简历中的技术描述能够被解释、演示和追问。每个阶段完成后，
学习结果必须同时包含源码、测试证据、故障场景和设计取舍，而不是只记录“接口已经完成”。

## 2. 六期学习主线

| 阶段 | 重点技术 | 必须能够回答的问题 |
|---|---|---|
| 第一期 | CMake、Workflow、Wfrest、配置、RAII | 服务如何启动、异步任务如何执行、依赖失败为何阻止启动 |
| 第二期 | OpenSSL、HMAC、Redis NX/EX、MySQL | 规范串如何构造、如何防重放、为什么不能把依赖故障返回成签名错误 |
| 第三期 | `posix_fallocate`、`pwrite`、Bitmap、并发状态 | 分片偏移如何计算、断点续传如何恢复、重复分片怎样幂等 |
| 第四期 | `mmap`、SHA-256、`rename`、索引、崩溃恢复 | 为什么要求同文件系统、文件与数据库没有事务时如何恢复 |
| 第五期 | 文件发现、稳定判定、本地 spool、客户端退避 | 如何避免上传半文件、断网和重启后怎样继续、哪些错误不能重试 |
| 第六期 | Nginx、systemd、压测、Valgrind、故障注入 | 如何部署、如何估算容量、如何证明没有文件损坏和重复归档 |

## 3. 每期必须形成的证据

1. 简历条目到源码文件、类、函数和测试的映射；
2. 一张核心调用链或状态机；
3. 正常场景与失败场景测试记录；
4. 至少一个真实发现并修复的问题复盘；
5. 可重复运行的构建、测试和演示命令；
6. 本期未实现能力及不实现原因。

第二期已经形成一个真实复盘：生产代码按配置写入 Redis 心跳键，但 E2E 测试写死旧前缀导致
TTL 断言失败。修复测试读取配置后全量 31/31 通过。面试中应能说明为什么先判断生产链路还是
测试假设错误，而不是为了让测试通过修改业务代码。

## 4. 建议动手练习

- 使用 GDB 跟踪一次心跳从 HTTP 路由到 Workflow MySQL/Redis 回调；
- 用 Redis CLI 查看防重放键、Hash 和 TTL；
- 第三期使用 `strace` 观察 `posix_fallocate` 和 `pwrite`；
- 第四期观察窗口化 `mmap`、`rename`，并用 MySQL `EXPLAIN` 检查联合索引；
- 第五期在上传中终止采集程序，检查 spool 状态和重启恢复；
- 第六期使用 ASan/UBSan、Valgrind 和压力工具复核资源释放与并发正确性。

## 5. 面试表达边界

- 可以说项目在三线仿真基线下完成可复现验证，不能说已经兼容任意真实厂商；
- 可以解释单实例方案的业务依据，同时说明扩展到对象存储或高可用需要哪些条件；
- 可以说明 Redis 会话丢失后重新上传的取舍，不能虚构不存在的跨存储事务；
- 可以展示代码中的 HMAC、Bitmap、`pwrite`、`mmap` 和 `rename`，不能把仅存在于文档的功能说成
  已经完成；
- 项目一负责可靠采集归档，项目二负责日志索引检索，不混淆职责。

## 6. 终板证据矩阵

最终新增 `简历能力与代码证据矩阵.md`，逐条对应 `SMT产线项目.md` 的主要工作，记录源码位置、
测试名称、演示命令、关键取舍和当前验证状态。矩阵在每期更新，只有对应代码和测试通过后才能
将条目标为已完成。

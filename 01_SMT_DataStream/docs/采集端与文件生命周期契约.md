# 采集端与文件生命周期契约

## 1. 责任边界

设备模拟器只负责在设备目录中产生文件行为；C++ 参考采集程序只通过目录观察文件，不读取
模拟器内部状态或期望清单；DataStream 服务只接收经过认证的心跳、元数据和文件分片。

```text
Python 模拟器 -> 设备目录 -> C++ 采集程序 -> HTTP/HMAC -> DataStream
                                              -> Redis/MySQL/归档文件系统
```

期望清单只由端到端测试读取，用于核对最终文件、大小、摘要和元数据，不能成为采集决策来源。

## 2. 目录映射

每条线只有一个 IPC，采集器配置显式列出三台授权设备及其输入目录：

```text
<inbox>/LINE-01/ST-SPI-01/SPI-ZM-01/
<inbox>/LINE-01/ST-AOI-01/AOI-VT-01/
<inbox>/LINE-01/ST-FCT-01/FCT-TRI-01/
```

路径只负责定位数据源，`line_id`、`station_id`、`device_id` 和 `collector_id` 来自已校验配置，
不能从任意外部路径片段直接推断后写入 API。

## 3. 文件完成判定

采集程序只接收以下三种明确封口信号：

1. 设备先写 `.tmp` 或 `.part`，完成后原子改为正式扩展名；
2. 文件大小和修改时间连续三个扫描周期不变，且能够以只读方式打开；
3. 厂商场景配置声明的配对完成标志出现。

扫描周期默认 2 秒，因此稳定判定至少持续 6 秒。运行日志允许持续追加，必须在轮转或模拟器
明确封口后上传，不能因为短时间没有新日志就上传仍会继续增长的文件。

零字节、无法识别扩展名、路径映射缺失和封口后再次变化的文件进入明确失败状态，不通过默认
文件类型或静默忽略掩盖问题。

## 4. 本地持久队列

第五期参考采集程序使用受控 spool 目录保存任务状态，不只依赖内存。每个任务至少记录：

- 源文件规范路径、设备、工位、产线和采集器；
- 文件类型、工单、产品 SN、产生时间；
- 文件大小和 SHA-256；
- 服务端 `upload_id`、分片大小和已确认进度；
- `DISCOVERED`、`READY`、`UPLOADING`、`COMPLETING`、`DONE`、`FAILED` 状态；
- 最近一次稳定错误码和下次允许尝试时间。

状态文件通过“写临时文件、`fsync`、原子 `rename`”更新。首期不引入 SQLite 或消息队列，仅用
目录和状态文件验证 POSIX 文件一致性、重启恢复和幂等调用。

## 5. 去重规则

同一源路径、设备编号、封口后的大小、修改时间和 SHA-256 共同形成采集任务身份。重复扫描只
更新已有任务，不创建第二个上传。相同内容出现在不同合法业务路径时仍视为不同业务文件，不能
只用 SHA-256 做全局去重。

服务端以请求标识防重放、分片编号和摘要保证分片幂等，以 `upload_id` 唯一约束保证完成幂等；
采集端去重和服务端幂等解决的是不同边界，不能只保留其中一层。

## 6. 重试与永久失败

采集程序只对以下失败重试：连接失败、调用超时、HTTP 503 和服务端明确可恢复的上传状态。
采用有上限指数退避，并在成功后恢复正常间隔。

签名错误、设备或绑定停用、字段非法、文件大小或摘要不一致属于永久失败，记录原因并等待人工
处理或重新发现一个已修正的新文件。服务端不主动循环重试 MySQL、Redis 或文件写入。

## 7. 模拟器必须覆盖的文件行为

- 一次写完后临时名原子改名；
- 多次追加后封口；
- 延迟关闭和长时间保持打开；
- 同名覆盖、重复导出和同内容不同路径；
- 零字节、截断、损坏和封口后变化；
- 三条线并发产生和短时间突发产生；
- IPC 断网、采集程序退出和重启恢复。

每个场景生成独立期望清单，明确应归档、应忽略和应失败的文件集合。

## 8. 不在首期实现的能力

- 直接连接设备私有协议或控制 PLC；
- 自动解释未知厂商格式；
- 跨 IPC 主备接管同一设备目录；
- 将本地 spool 建设为通用消息队列；
- 对永久性业务错误无限重试；
- 在采集端解析运行日志并构建索引。

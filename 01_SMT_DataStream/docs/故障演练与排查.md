# 故障演练与排查

## 1. 排查顺序

先判断影响面，再看依赖和持久状态：健康接口、systemd 状态、磁盘水位、MySQL、Redis、服务日志、
Collector spool。日志中的 `request_id` 用于关联请求；设备密钥、签名、Token 和正文不应出现在日志。

## 2. 固定演练

| 场景 | 注入方式 | 预期行为 | 恢复确认 |
|---|---|---|---|
| Redis 短暂不可用 | 维护窗口暂停 Redis 10 秒 | readiness 503；业务返回 `REDIS_UNAVAILABLE`；不写默认成功 | Redis 恢复后 readiness 200，旧上传会话丢失时 Collector 新建会话 |
| MySQL 短暂不可用 | 维护窗口暂停 MySQL 10 秒 | readiness 503；认证/查询返回 `MYSQL_UNAVAILABLE` | 恢复后心跳、查询正常，无测试残留 |
| 磁盘低水位 | 测试配置把最小剩余值调高 | 创建会话返回 `STORAGE_CAPACITY_EXCEEDED` | 恢复配置并重启后可创建会话 |
| DataStream 强退 | `kill -KILL` 测试实例 | systemd 拉起；临时状态按 Redis 恢复 | 重传/完成得到唯一归档记录 |
| Collector 强退 | `kill -KILL` 测试实例 | spool 状态和 payload 保留 | 重启续传，摘要和大小一致 |
| 完成响应丢失 | 完成后客户端重发 | 返回同一 `archive_id` | 正式文件和 MySQL 各一份 |

禁止在承载其他业务的共享 MySQL/Redis 上直接执行停服演练。自动化 E2E 通过独立 Redis 前缀、
临时目录和专用工单完成强退与会话丢失验证。

## 3. 常见故障

- `SERVICE_NOT_READY`：依次执行 MySQL `SELECT 1`、Redis `PING`，并核对临时区和归档区权限；
- `UPLOAD_LIMIT_EXCEEDED`：查看全局、设备和 IPC 活跃会话，不要直接无限增大配额；
- `FILE_INTEGRITY_MISMATCH`：比较 sidecar 大小/摘要与 spool payload，保留原文件调查设备覆盖；
- `STORAGE_CAPACITY_EXCEEDED`：停止新采集，扩容或按保留策略转移正式归档，禁止删除未知 `.part`；
- Collector `FAILED`：检查 `last_error`。永久 4xx 或损坏 sidecar需修正源数据后产生新封口文件，
  不能反复重试同一无效任务；
- Nginx 413：核对代理 9 MiB 与服务 9 MiB 请求限制，正常分片最大为 8 MiB。

## 4. 复盘模板

记录发生/发现/恢复时间、受影响产线与文件数、首个可观察信号、根因、状态机所在阶段、是否出现
正文损坏或重复归档、人工操作、恢复证据和防复发动作。将“Redis 会话丢失”与“归档数据丢失”
明确区分：Redis 是会话状态，MySQL 加正式文件才是归档事实。

# SMT 三线仿真业务基线

## 1. 文档定位

本文固定项目后续开发、容量估算、模拟数据和端到端验收使用的业务基线。所有参数均为学习与
测试用途的中小型电子制造企业仿真假设，不代表已经取得某家真实工厂的生产数据。

代码、种子数据和测试若需要偏离本文，必须先更新基线并说明原因，不能在单个测试中临时改变
规模后继续声称代表同一场景。

## 2. 工厂与产线拓扑

总体模拟 3 条结构相同的 SMT 生产线。每条线包含 1 台采集工控机、1 台 SPI、1 台 AOI 和
1 台 FCT，共计 3 台 IPC、9 台检测或测试设备。

```text
                         数据归档服务器
                               |
              -------------------------------------
              |                 |                 |
           LINE-01           LINE-02           LINE-03
              |                 |                 |
        IPC-L01-01       IPC-L02-01       IPC-L03-01
          /   |   \         /   |   \         /   |   \
        SPI  AOI  FCT      SPI  AOI  FCT      SPI  AOI  FCT
```

单条产线的标准业务关系是：一台 IPC 只采集本线三台设备，一台设备只归属一个工位，一个工位
只归属一条产线。首期不模拟跨线采集、设备搬迁或一主一备 IPC。

建议编号如下：

| 产线 | 采集器 | SPI | AOI | FCT |
|---|---|---|---|---|
| `LINE-01` | `IPC-L01-01` | `SPI-ZM-01` | `AOI-VT-01` | `FCT-TRI-01` |
| `LINE-02` | `IPC-L02-01` | `SPI-ZM-02` | `AOI-VT-02` | `FCT-TRI-02` |
| `LINE-03` | `IPC-L03-01` | `SPI-ZM-03` | `AOI-VT-03` | `FCT-TRI-03` |

现有第一期种子和样本仍是单线 SPI/AOI/ICT 旧基线。已执行的 `001` 迁移不修改；第三期新增
迁移扩展 `station_type` 为 FCT，并用新种子切换到三线基线。旧样本只作为历史回归数据，不能
继续代表最终仿真规模。

## 3. 数据类型与来源

| 数据 | 主要来源 | 平台文件类型 | 典型格式与大小 |
|---|---|---|---|
| SPI 检测结果 | SPI | `DETECTION_RESULT` | JSON/CSV，4 KiB 至 128 KiB |
| AOI 检测结果 | AOI | `DETECTION_RESULT` | JSON/CSV，8 KiB 至 256 KiB |
| FCT 测试报告 | FCT | `TEST_REPORT` | CSV，16 KiB 至 512 KiB |
| NG 图片 | SPI/AOI 异常板 | `NG_IMAGE` | PNG/JPEG，200 KiB 至 5 MiB |
| 设备导出报告 | SPI/AOI/FCT | `DEVICE_EXPORT` | ZIP/PDF，20 MiB 至 100 MiB |
| 设备运行日志 | SPI/AOI/FCT | `RUNTIME_LOG` | UTF-8 LOG，按封口文件上传 |

`DEVICE_EXPORT` 用于稳定覆盖几十 MiB 文件；2 GiB 仍是接口边界测试，不作为日常模拟文件
大小。项目一只归档运行日志，不解析日志正文；全文检索属于项目二。

## 4. 节拍与数据量假设

- 正常节拍：每条线每 30 秒完成一块板，即每小时约 120 块；
- 突发节拍：每条线每 10 秒一块，持续 10 分钟，用于积压与反压测试；
- 生产时长：每天两个 8 小时班次；
- 每块板至少产生 SPI、AOI、FCT 三份结果文件；
- SPI/AOI 合计 NG 图片概率默认 3%，允许测试场景覆盖 0% 和 100%；
- 每台设备每 4 小时生成一份 20 MiB 至 100 MiB 的设备导出报告；
- 运行日志持续追加，达到 10 MiB、跨小时或设备明确封口时形成可上传文件；
- 正常模拟运行一天约产生 17,280 份基础结果文件，另加 NG 图片、日志分段和导出报告。

这些数量用于测试规模和面试中的容量说明，不声称等同于真实产线节拍。生成器应允许缩短真实
等待时间推进业务时间，同时保持清单中的业务时间线可复现。

## 5. 网络与本地缓存假设

- IPC 到归档服务器为受控内网，正常情况下可直连；
- 断网场景最长模拟 8 小时，IPC 必须在本地持久保存待上传任务；
- 每台 IPC 的 spool 上限为 20 GiB，预留空间低于 2 GiB 时停止接纳新的稳定文件；
- 网络恢复后只补传未完成分片，不重新上传已确认完成的分片；
- 永久性 4xx 业务错误不自动重试；网络错误、超时和明确的 5xx 才进入有上限退避；
- 本地缓存达到配置水位后停止接收新的稳定文件并记录告警，不删除未上传业务文件；
- 服务端 Redis 会话丢失时，采集端创建新会话重新上传，旧临时文件由严格清理规则处理。

## 6. 身份与授权假设

HTTP 请求由 IPC 上的采集程序发起。首期继续使用数据源设备身份签名，IPC 受控保存本线三台
设备的开发密钥。服务端维护 `collector_id` 与 `device_id` 的授权绑定：

- 设备只能由绑定的本线 IPC 上报心跳和上传文件；
- `collector_id` 不是任意审计字符串，必须通过绑定校验；
- 设备停用、工位停用、产线停用或绑定停用时拒绝请求；
- 首期不建设 PKI 或复杂密钥管理平台，生产密钥初始化、轮换和泄露处置写入部署规范。

该模型是当前仿真条件下对简历中“设备编号参与 HMAC”的明确解释。若未来现场确认 IPC 应使用
独立凭据，再通过新认证版本迁移，不能在 v1 中同时接受两种含义。

## 7. 存储与生命周期假设

- 单服务实例、单归档文件系统、一个 MySQL 和一个 Redis；
- 服务端临时目录同时满足“至少剩余 10 GiB”和“至少剩余文件系统容量 15%”才接受新会话；
- 全局最多 24 个在途会话，每台设备最多 2 个、每台 IPC 最多 6 个；
- 全局已预留临时文件总量上限为 50 GiB，使用 `posix_fallocate` 成功后才计入有效会话；
- Redis 上传状态使用持久化并采用 `noeviction`，不与可淘汰查询缓存混用；
- Redis 会话丢失的业务代价是重新创建会话和重传，不把每个分片同步写入 MySQL；
- 正式归档在本项目中默认保留 31 天仅用于生命周期演练，但首期代码不自动删除正式文件；
- 临时文件、失败会话和 Redis 状态按明确状态与时间清理；
- 正式归档删除、异地备份和真实保留年限属于生产治理，不由模拟参数替代。

## 8. 验收口径

终板端到端验收必须记录产线数、设备数、IPC 数、模拟业务时长、文件数量、数据量、NG 比例、
突发倍率和断网时长。至少验证：

1. 三条线并发产生并归档六类业务数据；
2. IPC 重启后恢复本地队列；
3. 分段写入、临时名改名、延迟关闭和日志追加不会被提前上传；
4. 同一文件重复发现不会生成重复归档；
5. 8 小时断网模型下任务可持久排队并在恢复后补传；
6. Redis 会话丢失后重新创建会话，正式归档不重复；
7. 突发节拍下磁盘水位、会话配额和退避行为符合配置；
8. MySQL 元数据、归档文件大小和 SHA-256 与期望清单一致。

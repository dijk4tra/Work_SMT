## 3. 项目介绍标准答案

### 3.1 请简单介绍这个项目

**30 秒版本：**

这是一个 SMT 设备原始数据采集和归档平台。三条产线上的 SPI、AOI、FCT 设备会产生检测结果、
测试报告、NG 图片、导出包和运行日志。产线 IPC 上的 Collector 负责判断文件是否已经写完、创建
本地持久化快照并上传。服务端使用 Wfrest 提供心跳、分片上传、完成归档和历史查询接口，Workflow
异步访问 MySQL 和 Redis。文件正文保存在文件系统，MySQL 保存可查询元数据，Redis 保存防重放、
心跳和上传会话。大文件支持乱序分片与断点续传，最终经过大小和 SHA-256 校验后原子归档。

**两分钟版本：**

项目解决的是不同厂商 SMT 设备数据分散在工控机、容易被覆盖或丢失、历史追溯困难的问题。项目按照
三条产线建模，每条产线由一台 IPC 采集 SPI、AOI、FCT 设备数据。Collector 支持原子改名、稳定
窗口和 `.done` 三种封口方式，先生成不可变的 spool 快照，再以设备身份签名上传。

服务端首先校验 HMAC、正文摘要、时间窗口和 Redis 防重放键，然后创建上传会话，使用
`posix_fallocate` 预分配临时文件。分片通过 `pwrite` 写固定偏移，Redis Bitmap 记录完成情况，
因此可以乱序上传，并在断线恢复后补传缺失分片。完成上传时，会话原子切换到 `VERIFYING`；服务端
使用窗口化 `mmap` 计算整文件 SHA-256。校验通过后，在同一文件系统内通过 `rename` 将文件移动到
正式路径，随后将元数据写入 MySQL。文件系统、Redis、MySQL 没有分布式事务，因此项目通过确定性
路径、会话状态和数据库唯一约束，处理各关键步骤中进程异常退出后的状态不一致问题。

历史查询支持设备、工位、工单、产品 SN、文件类型、结果和时间范围，使用
`(archived_at, archive_id)` 游标分页。项目最终有 54 项单元/集成/E2E 测试、81.8% 核心行覆盖率，
并验证了 Collector 进程被强制终止、Redis 会话丢失和 TLS 部署等场景。

备考提示：如果面试官追问团队分工，需要根据真实经历说明自己参与或负责的模块，同时明确与其他

### 3.3 项目最大的难点是什么

可以选择以下两个重点：

- **跨文件系统、Redis、MySQL 的一致性。** 这三个存储之间没有分布式事务，因此需要分析
  `pwrite` 后 Bitmap 未更新、`rename` 后 MySQL 元数据未写入、MySQL 写入后 Redis 状态未更新等
  情况，并通过幂等重试恢复进程异常退出后可能出现的不一致状态；

  因此使用固定偏移、分片摘要、Bitmap、会话状态和 Collector spool 共同保证。

## 4. SMT 业务问题

### 4.1 SPI、AOI、FCT 分别是什么

- SPI 是锡膏检测，关注印刷后的锡膏高度、面积、偏移等；
- AOI 是自动光学检测，关注贴装后的错件、少件、极性、焊接外观等，NG 时通常带缺陷图片；
- FCT 是功能测试，关注电压、电流、通信等功能测点，通常输出 CSV 或设备报告。

平台不解析这些厂商私有内容，而是将其统一映射为 `DETECTION_RESULT`、`TEST_REPORT`、
`NG_IMAGE`、`DEVICE_EXPORT`、`RUNTIME_LOG` 五类原始文件并保存追溯元数据。

### 4.2 为什么还需要产线 IPC

设备通常只会向本地目录导出文件，不一定支持统一协议。IPC 部署在产线侧，负责适配目录、判断封口、
缓存断网期间的数据，并统一调用服务端。由此，设备无需直接访问中心服务器，不同厂商的目录和
导出格式差异也由 IPC 适配层统一处理。

### 4.3 如何判断文件已经写完

检测到文件后，需要先判断文件是否已经写完，避免上传尚未完成的内容。当前支持三种明确的封口规则：

1. 临时名写完后原子改名；
2. 大小和纳秒 mtime 连续多个扫描周期不变；
3. 正文关闭后出现同名 `.done`。

运行日志优先使用完成标志，短暂没有新增内容不足以证明文件已经写完。确认封口后，Collector 会复制
不可变快照并再次校验摘要，避免设备覆盖原路径而影响正在进行的上传。

### 4.4 sidecar 为什么存在

文件名无法稳定表达工单、产品 SN、类型、结果和产生时间。sidecar 是设备/适配层与 Collector
之间的最小元数据约定。产线、工位、设备、IPC 和密钥来自受控配置；sidecar 作为外部输入，不能
覆盖这些受信配置。

### 4.5 为什么文件正文不存 MySQL

大文件 BLOB 会增加数据库存储、redo/undo、备份和主从复制的压力，也不利于顺序写和文件级校验。
项目使用文件系统保存正文，MySQL 只保存结构化元数据和相对路径，兼顾文件吞吐与条件查询。

### 4.6 为什么首期不用对象存储或消息队列

首期是单实例、单归档文件系统的三条产线试点，本地 `rename` 能提供明确原子语义；引入对象存储会
改变完成语义和一致性模型。上传正文已经通过本地 spool 和会话状态可靠保存，当前也没有异步消费
需求，因此没有引入消息队列。规模扩大并出现跨节点或异地复制需求时，再评估相应方案。

### 4.7 如何按产品追溯

归档元数据记录 `line_id`、`station_id`、`device_id`、`collector_id`、`work_order`、
`product_sn`、文件类型、结果、设备产生时间和平台归档时间。查询接口可按工单和产品 SN 精确
定位该板在不同工位产生的原始文件。

## 5. 架构与技术选型

### 5.1 为什么首期采用单体服务

认证、上传、归档和查询共享同一套会话、文件系统和部署边界，首期也没有独立扩缩容需求。此时拆分
会增加 RPC、部署和一致性成本。当前先按源码模块划分职责，待出现明确的独立容量或团队边界后，
再评估服务拆分。

### 5.2 为什么选择 Wfrest 和 Workflow

Wfrest 提供 C++ HTTP 路由和请求响应封装，Workflow 提供异步网络任务、SeriesWork 和 MySQL/
Redis 客户端。接入层可以先完成开销较小的参数校验，再将有依赖关系的调用组织为任务链，避免同步
阻塞网络线程。

准确的表述是“项目使用了 Workflow/Wfrest 提供的 Reactor 和异步任务能力”，而不是“在项目中
自行实现了 epoll Reactor”。如果被问到框架底层，可以解释 Reactor 通常由 epoll 等 I/O 多路复用
机制监听就绪事件，再由事件循环将事件分发给连接处理器和任务状态机。非阻塞 I/O 可以避免线程在
等待网络数据时持续占用执行资源，但业务回调仍不适合执行耗时的 CPU 密集型任务或阻塞文件操作。

### 5.3 Workflow 的 SeriesWork 有什么价值

同一个请求经常有明确依赖顺序，例如先查设备，再验签，再写防重放键，最后执行业务操作。
SeriesWork 让任务按序执行，并在共享状态中保存结果；没有依赖关系的任务可以使用 ParallelWork。
存在前置依赖的步骤不应直接并行执行，否则会破坏安全校验顺序。

### 5.4 Nginx 在这里做什么

Nginx 终止 TLS，服务端只监听 `127.0.0.1:8080`；同时配置 9 MiB 请求体、连接/发送/读取超时、
反向代理头和 HSTS、nosniff、DENY 等安全响应头。Nginx 的请求体上限应不小于服务端限制，而
服务端限制需要覆盖最大 8 MiB 分片。

### 5.5 系统的单点和扩展边界

当前部署包含一个 DataStream 实例、一个本地归档文件系统，以及各一个 MySQL 和 Redis 实例。
按 `upload_id` 命名的
Workflow 文件任务队列只保证单实例内磁盘任务串行，当前不支持多个实例共同写入同一个上传会话。
若扩展为多实例，需要引入共享正文存储、跨实例会话互斥、请求路由或会话亲和，并重新设计原子完成
流程。

## 6. HMAC 认证与安全

### 6.1 HMAC 签名了哪些内容

规范串固定为：

```text
v1\n
<HTTP_METHOD>\n
<PATH>\n
<DEVICE_ID>\n
<TIMESTAMP>\n
<REQUEST_ID>\n
<CONTENT_SHA256>
```

分片的 `upload_id` 和 `chunk_no` 位于 PATH 中，因此自然参与签名。正文先做 SHA-256，HMAC 再
保护规范串，避免直接将大体积请求正文放入签名串。

### 6.2 HMAC 和普通 SHA-256 有什么区别

SHA-256 只能证明内容摘要一致，攻击者修改正文后可以重新计算摘要。HMAC 使用双方共享密钥，
未持有密钥的一方无法为篡改后的请求生成合法认证码。项目同时保留正文摘要，是为了完整性校验和
分片幂等比较。

### 6.3 为什么还需要 TLS

HMAC 提供请求完整性和设备身份认证，但不会加密正文，也不能隐藏工单、SN、图片和 Token。
TLS 负责传输机密性、服务端身份和链路保护，两者职责不同。

### 6.4 如何防重放

先校验请求头格式和时间窗口，再查设备密钥、校验正文摘要和 HMAC；签名成功后用 Redis
`SET key value NX EX` 登记 `device_id + request_id`。已存在返回 `REQUEST_REPLAYED`。

防重放键应在签名验证通过后写入，否则无效请求也会占用 Redis 空间。时间窗口负责拒绝过期请求，
唯一请求标识负责识别窗口内的重复请求，两者共同构成防重放机制。

### 6.5 为什么使用常量时间比较

普通字符串比较可能在首个不同字节处提前返回，理论上泄露匹配前缀的时序信息。摘要、签名和
Operator Token 使用常量时间比较，避免按内容位置提前结束。

### 6.6 密钥如何保存和轮换

当前数据库保存设备 HMAC 密钥的二进制值，进程配置只保存环境变量名称；日志禁止输出密钥、
签名和 Token。当前版本尚未实现双密钥轮换协议。生产环境如果需要无停机轮换，可以增加密钥版本
或短期双密钥验证窗口，并审计设备侧切换进度。

### 6.7 如何防路径穿越

客户端原文件名只作为元数据保存，禁止 `/`、反斜杠和控制字符。正式路径由服务端用经过验证的
产线/工位/设备标识、文件类型、日期和 `upload_id` 生成，扩展名来自允许列表。服务端不解压 ZIP，
因此压缩包内的路径不会参与服务器目录生成。

## 7. 分片上传与断点续传

### 7.1 为什么要分片

大文件一次上传失败需要整体重传，也容易超过代理请求体和超时限制。采用分片后，单次失败只需重传
对应分片，同时可以并发处理不同会话、支持分片乱序到达，并控制内存峰值。当前分片范围是 1 MiB
到 8 MiB。

### 7.2 分片偏移和最后一片如何计算

```text
chunk_count = ceil(file_size / chunk_size)
offset = chunk_no * chunk_size
last_length = file_size - chunk_no * chunk_size
```

除最后一片外长度必须等于 `chunk_size`。服务端独立计算 `chunk_count`，避免客户端参数错误造成
越界或缺片误判。

### 7.3 为什么使用 `posix_fallocate`

创建会话时提前预留磁盘块，可以尽早发现空间不足，减少上传到最后才因 ENOSPC 失败的概率，
也让随机偏移写入对应到已分配空间。预分配不等于文件内容已经有效，完整性仍以 Bitmap、大小和
最终摘要为准。

### 7.4 `pwrite` 相比 `write` 有什么优势

`pwrite(fd, buffer, size, offset)` 显式指定偏移，不修改共享文件偏移量，适合乱序分片并减少
`lseek + write` 的竞争窗口。系统调用可能发生短写或被 `EINTR` 中断，因此实现需要循环调用，
直到完成本分片的全部字节写入或遇到明确错误。

### 7.5 为什么写完文件后才置 Bitmap

Bitmap 表示该分片已持久完成。如果先置位再写文件，进程崩溃会让服务端误认为缺失内容已经存在。
当前顺序是登记摘要、通过 `pwrite` 完成全部字节写入、再原子置位。若进程在文件写入后、Bitmap
置位前退出，客户端可以重传同一偏移；相同内容的覆盖写不会破坏文件。

### 7.6 重复分片如何幂等

幂等键是上传会话和分片编号，Redis 摘要 Hash 保存该编号对应的内容 SHA-256：

- Bitmap 已置位且摘要相同：返回成功并标记 `already_completed=true`；
- 编号相同但摘要不同：返回 409 `CHUNK_CONTENT_CONFLICT`；
- 请求标识重复：认证层先返回 `REQUEST_REPLAYED`。

请求防重放和业务幂等属于不同层次：前者拒绝相同请求，后者允许客户端使用新的 `request_id`
重试同一业务分片。

### 7.7 断线后如何续传

Collector 持久化 `upload_id` 和本地 payload。恢复后调用进度接口，服务端从 Bitmap 计算
`missing_chunks`，Collector 仅上传缺失分片。进度查询不刷新 TTL，避免只查询而不上传的会话长期存活。

### 7.8 Redis 会话丢失怎么办

服务端返回 `UPLOAD_NOT_FOUND`。Collector 清除旧 `upload_id`，为同一个本地任务创建新会话并
重新上传。服务端不根据孤立临时文件推断 Bitmap，因为无法证明哪些字节完整；旧 `.part` 超过
保留期后由严格清理任务删除。

## 8. Redis 设计

### 8.1 Redis 保存什么

- 防重放：字符串键，`SET NX EX`；
- 设备心跳：Hash + TTL；
- 上传会话：Hash；
- 分片完成情况：Bitmap；
- 分片摘要：Hash；
- 活跃会话和预留空间：配额键。

MySQL 保存需要长期持久化的数据，Redis 保存高频、短生命周期、可恢复的运行状态。

### 8.2 为什么 Bitmap 适合记录分片

每个分片只需要一位，空间复杂度是 O(n/8)，`GETBIT`、`SETBIT`、`BITCOUNT` 都直接适配完成
状态。仅有 Bitmap 不能判断重复分片内容是否一致，因此还使用摘要 Hash；两者职责不同。

### 8.3 为什么使用 Lua

会话状态、Bitmap、摘要和配额跨多个键更新时，需要把“检查状态+更新多个键”作为一个 Redis
原子操作。Lua 在 Redis 单线程命令执行上下文中原子运行，避免客户端多条命令之间被其他请求
插入。脚本只处理 Redis 状态，不应在 Lua 脚本中执行文件 I/O 或 MySQL 操作。

### 8.4 TTL 如何设计

初始会话 TTL 为 24 小时。成功的新分片刷新会话、位图、摘要和配额 TTL；重复分片和进度查询
不刷新。归档或失败后缩短 TTL，保留有限诊断窗口并释放配额。心跳使用更短 TTL，在线状态由
最后心跳时间计算，不额外存永久在线布尔值。

### 8.5 Redis 不可用时为什么不降级到进程内存

Redis 保存防重放和上传会话的权威状态。如果临时改用进程内存，进程重启后状态会丢失，不同请求
也可能观察到不一致的状态，并削弱重放保护。因此 Redis 不可用时服务端明确返回 503；Collector
保留本地 payload，待依赖恢复后再上传。

### 8.6 这个项目用了 Cache-Aside 吗

历史归档查询当前直接访问 MySQL，没有缓存，因此该模块没有实现 Cache-Aside。简历中的
Cache-Aside、穿透、击穿和雪崩是通用技能或日志检索项目内容。若后续缓存归档查询，需要先定义
更新/失效规则、空结果缓存、TTL 和热点键保护。

## 9. `mmap`、哈希与原子归档

### 9.1 为什么完成时还要校验整文件

每片摘要只能证明收到的单片内容与请求一致，不能替代整文件顺序、总大小和客户端原始摘要。
完成时先检查 Bitmap，再通过 `stat` 检查大小，最后计算整文件 SHA-256，三项均通过后才归档。

### 9.2 为什么窗口化 `mmap`

一次映射整个大文件会占用较大的虚拟地址空间，也不利于控制资源。窗口化映射固定大小区域，逐段
调用 OpenSSL 增量 SHA-256，内存占用与文件总大小无关。映射偏移必须满足页对齐要求，实际数据
指针再加页内偏移；每个窗口用 RAII 或明确 `munmap` 释放。

### 9.3 `mmap` 一定比 `read` 快吗

不一定。它减少显式用户态复制并便于顺序遍历，但仍会发生缺页，超大文件也受页缓存影响。
选择它是为了控制单次映射范围并执行增量摘要，不能据此认为所有场景都比 `read` 快；应通过真实磁盘
和文件大小基准测试决定。

### 9.4 为什么 `rename` 必须在同一文件系统

同一文件系统内 `rename` 是目录项级原子切换，其他进程不会看到尚未完成的正式文件。跨文件系统会
返回 `EXDEV`，只能复制再删除，失去同样的原子语义，所以服务启动时检查临时区和归档区设备号。

### 9.5 为什么选择先移动文件，再写入 MySQL

如果先写入数据库，会产生“查询已经返回记录，但正文尚不存在”的窗口。当前先将校验完成的文件
原子移动到确定的正式路径，再插入元数据；如果插入失败，重试完成操作会识别已有文件，并补充写入
元数据。任何执行顺序都无法完全消除跨系统窗口，关键是选择能够检测并通过幂等操作恢复的顺序。

## 10. 幂等、一致性和并发

### 10.1 项目有哪些幂等层次

1. 认证层：相同 `request_id` 在时间窗口内拒绝重放；
2. 分片层：相同 `upload_id + chunk_no + digest` 返回幂等成功；
3. 完成层：`upload_id` 唯一，重复完成返回同一 `archive_id`；
4. 归档路径：由 `upload_id` 确定，并有 `relative_path` 唯一约束；
5. Collector：任务 ID 包含源路径、设备、大小、mtime 和摘要，重复扫描不会重复创建任务。

### 10.2 同一会话并发上传如何保证安全

不同会话可以并发；同一 `upload_id` 的磁盘任务使用 Workflow 命名队列串行执行，Redis Lua
原子保护摘要登记、Bitmap 和状态切换。完成操作只有在 Bitmap 全部置位后才能从 `UPLOADING`
切到 `VERIFYING`，因此所有成功分片都已经完成磁盘写入；切换后新分片会被拒绝。命名队列只适用于
单实例边界。

### 10.3 `pwrite` 使用显式偏移，为什么还要按会话串行文件任务

不同偏移的 `pwrite` 不共享文件偏移，理论上可以并行；当前按会话串行是为了限制同一大文件的
磁盘竞争，并简化相同分片恢复写入时的顺序控制。跨资源状态不由这个队列单独保证：写入前由 Lua
登记摘要，写入后由 Lua 置位；complete 只有在全部位图完成后才能切换状态。

### 10.4 进程在关键步骤退出后如何恢复

| 崩溃点 | 恢复方式 |
|---|---|
| 临时文件创建后 Redis 会话未写入 | 由当前请求删除；超期的孤立 `.part` 由清理任务删除 |
| `pwrite` 后 Bitmap 未置位 | 新请求覆盖同一偏移后置位 |
| Bitmap 已置位但响应丢失 | 相同摘要幂等成功 |
| 进入 `VERIFYING` 后退出 | complete 重试继续确定性校验 |
| `rename` 后 MySQL 元数据未写入 | 复用确定路径并补充写入元数据 |
| MySQL 元数据已写入但 Redis 未更新 | 以 MySQL 的唯一归档记录为依据修复 Redis |
| 完成响应丢失 | 重试返回同一 `archive_id` |

### 10.5 为什么不用分布式事务

本地文件系统、Redis 和 MySQL 没有共同事务管理器。引入两阶段提交成本较高，而且文件系统也
难以直接参与。当前通过幂等、唯一约束和确定性路径，处理各关键步骤异常退出后可能产生的不一致，
使操作最终恢复到一致状态，符合单实例首期边界。

## 11. MySQL 表与索引

### 11.1 核心表有哪些

- `production_line`：产线；
- `station`：工位与类型；
- `device`：设备、密钥、启用状态和最后心跳；
- `collector_device_binding`：IPC 与设备授权绑定；
- `archive_file`：正式归档元数据；
- `schema_migration`：版本和迁移文件摘要。

### 11.2 为什么设计这些联合索引

```text
(device_id, archived_at, archive_id)
(station_id, archived_at, archive_id)
(work_order, product_sn, archived_at, archive_id)
```

它们对应“某设备/工位最近文件”和“某工单某产品的追溯”高频条件，并包含稳定排序字段。
联合索引遵循最左前缀：例如 trace 索引能高效支持先按工单，再按产品和时间；只查 product SN
而没有 work order 不一定能充分使用该索引，需要根据真实查询比例再决定是否新增索引。

### 11.3 为什么索引最后还有 `archive_id`

`archived_at` 可能相同，单独按时间排序不稳定。`archive_id` 作为第二排序键形成全序，也让
`(archived_at, archive_id)` 游标能够准确继续下一页。

### 11.4 为什么不用 OFFSET 分页

较大的 OFFSET 需要扫描并跳过前面的记录，并且并发插入可能造成重复或遗漏。当前使用 keyset 游标：
下一页查询“小于上一页末尾 `(archived_at, archive_id)`”的记录，复杂度和稳定性更好。

### 11.5 唯一约束的作用

`UNIQUE(upload_id)` 防止完成重试产生两条归档记录，`UNIQUE(relative_path)` 防止两个记录指向
同一正式路径。应用层幂等判断和数据库约束同时存在：应用层处理正常流程，数据库唯一约束负责
处理最终的并发冲突。

### 11.6 如何避免 SQL 注入

SQL 由固定模板构造；整数经过严格解析，枚举和标识符经过白名单/格式校验，字符串使用 Workflow
MySQL 工具转义后再进入 SQL。查询参数不会未经处理直接拼接到 SQL 中。若数据库客户端支持完善的
参数化语句，可以进一步改为参数绑定；当前实现采用严格校验与字符串转义。

### 11.7 MySQL 不可用时系统如何处理

readiness 返回 503；依赖设备信息的认证、心跳和查询返回明确的 `MYSQL_UNAVAILABLE`，不会将
依赖故障报告为签名错误，也不会返回默认空列表。Collector 保留本地文件，待服务恢复后继续处理。

### 11.8 项目如何使用事务隔离和行锁

InnoDB 常用的默认隔离级别是 Repeatable Read，但项目没有依赖长事务快照解决文件一致性。核心
并发控制来自 Redis 会话原子操作、Workflow 命名文件队列和 MySQL 唯一约束。可以解释隔离级别、
MVCC、当前读和行锁原理，同时需要明确上传状态机并不是通过复杂的行锁事务实现的。

## 12. Collector 与边缘可靠性

### 12.1 Collector 状态机是什么

```text
DISCOVERED -> READY -> UPLOADING -> COMPLETING -> DONE
                    \-> FAILED
```

发现文件后生成 payload 快照；`READY` 状态创建会话，`UPLOADING` 状态查询并补传缺失分片，
`COMPLETING` 状态调用完成接口。网络错误使用指数退避，达到配置上限后保持最大重试间隔；永久性
4xx 错误、零字节文件和损坏的 sidecar 等情况进入 `FAILED`。

### 12.2 spool 如何保证重启恢复

正文先复制为不可变 payload；状态 JSON 写入临时文件并完成全部字节写入后，对文件执行 `fsync`，
再原子 `rename`，随后对
父目录 `fsync`。启动时加载正式状态，只删除名称严格匹配的中断临时文件。如果检测到状态文件
损坏，Collector 会拒绝启动并报告明确错误，避免在没有提示的情况下丢弃任务。

### 12.3 为什么要 `fsync` 文件后再 `fsync` 目录

文件 `fsync` 保证内容和 inode 持久化；原子 `rename` 更新的是目录项，目录 `fsync` 用来保证
这个名称变更在系统异常断电后仍然持久。只调用其中一个无法完整保证状态提交。

### 12.4 断网 8 小时如何估算 spool

容量应根据以下公式估算：

```text
spool需求 = 产线数 × 每秒文件数 × 平均文件大小 × 断网秒数 × 峰值/安全系数
```

还要考虑 NG 图片比例、设备导出大文件和恢复上传速度。恢复吞吐需要高于正常新增速率，否则积压
数据将持续增加。当前配置值来自仿真假设，不代表实际工厂容量。

### 12.5 spool 容量达到上限时如何处理

创建新快照前检查配置容量和文件系统最小剩余空间；达到容量阈值时暂停接纳新文件，并输出包含原因
和处理建议的告警。未完成的 payload 需要保留，相关任务不得标记为成功。生产环境还应要求设备侧
保留原文件，并触发运维处理流程。

## 13. 清理、健康与可观测性

### 13.1 为什么严格限制清理范围

错误删除文件的影响大于临时空间占用。清理只处理名称严格为 UUID `.part`、mtime 超过保留期，
且 Redis 无会话或处于可删除结束状态的临时文件。对于未知文件只记录告警，不执行删除；正式归档
始终不在该清理范围内。

### 13.2 liveness 和 readiness 有什么区别

- liveness 只证明进程能处理请求，失败通常意味着进程需要重启；
- readiness 检查 MySQL、Redis、临时区和归档区，失败表示不应继续接收业务流量，但进程可能仍
  正常运行并等待依赖恢复。

readiness 只返回总体状态，不暴露连接串和具体内部错误。

### 13.3 日志记录什么

记录 `request_id`、设备、上传会话、业务错误码和重要状态变化；不记录设备密钥、完整签名、
Bearer Token、密码和文件正文。`request_id` 用于关联一次异步任务链和客户端问题。

### 13.4 当前有哪些指标

首期主要通过结构化日志和固定压测观察在线设备、活跃会话、失败分片、完整性失败、归档成功、
临时区占用和接口延迟。当前尚未接入 Prometheus，因此项目还没有完整监控平台；后续可增加指标
端点和告警规则。

## 14. C++11 与 Linux 问题

### 14.1 项目如何使用 RAII

文件描述符、目录句柄、OpenSSL 摘要上下文、映射窗口和异步共享状态都需要在所有返回路径释放。
代码使用带自定义删除器的 `unique_ptr`、局部对象析构和明确封装管理资源，避免在每个错误分支
重复编写释放逻辑。RAII 的核心是将资源生命周期绑定到对象生命周期，而不仅是使用智能指针。

### 14.2 `unique_ptr` 和 `shared_ptr` 如何选择

独占资源如 OpenSSL context 或 DIR 使用 `unique_ptr`；异步任务链中多个回调需要共同持有请求
状态，使用 `shared_ptr<State>`。不应为了简化生命周期处理而全部使用 `shared_ptr`，否则所有权
不清晰且有原子引用计数开销。

### 14.3 Lambda 捕获要注意什么

异步回调不能引用已经离开栈的局部变量。共享请求状态按值捕获 `shared_ptr`，服务级对象在停止前
保证任务结束。按引用捕获只适用于等待组等生命周期明确长于回调的对象。还要避免两个对象通过
`shared_ptr` 相互捕获形成环。

### 14.4 右值引用和移动语义有什么价值

移动可以转移 string、vector、配置对象等内部缓冲区，减少复制；被移动对象仍然有效但状态未
指定。项目不会为了展示语言特性而过度使用 `std::move`。返回局部变量通常可以依赖 NRVO 或隐式
移动；对 `const` 对象使用 `std::move` 通常仍会调用复制操作。

### 14.5 异常如何处理

配置和启动失败在进程边界捕获并打印明确原因后退出；业务请求使用稳定错误码响应。只在能够
转换错误、补充上下文或释放资源的层捕获异常，不使用 `catch (...)` 忽略异常，也不把存储错误转换
成空结果。

### 14.6 原子变量和内存序在哪里使用

Collector 用 `std::atomic<bool>` 表示停止标志，信号等待线程以 release 写入，主循环以 acquire
读取，保证停止状态在线程间可见。复杂会话状态不能仅依赖一个原子变量，需要互斥保护多个字段和
跨资源不变量。

### 14.7 `pread/pwrite` 是否会改变文件偏移

不会。它们使用显式 offset，适合并发独立区间访问。简历中 `pread` 回读日志正文主要属于日志
检索项目；本项目的上传路径主要使用 `pwrite`。

### 14.8 虚拟内存和 `mmap` 的关系

`mmap` 将文件页映射到进程虚拟地址，首次访问可能触发缺页并由内核加载页缓存。映射不会一次性
将全部文件内容加载到物理内存。窗口化映射限制同时保留的虚拟区间和活跃页数量。

### 14.9 项目是否自己实现了线程池

没有。网络和异步任务调度使用 Workflow 的执行设施，Collector 有一个明确的信号等待线程。
可以解释线程池的任务队列、固定工作线程、条件变量、停止标志和 join，同时应明确项目使用的是
Workflow 提供的任务调度能力，并未自行实现通用线程池。

## 15. 网络与 HTTP 问题

### 15.1 TCP 为什么需要应用层请求 ID 和幂等

TCP 只保证一个连接内字节有序可靠，不保证客户端一定收到业务响应。连接可能在服务端完成写入
后断开，客户端无法判断结果，所以仍需要请求标识、防重放和业务幂等。

### 15.2 HTTP 401、403、409、422、429、503、507 怎么区分

- 401：缺少认证、签名错误、时间过期、Operator Token 错误；
- 403：设备已知但停用、IPC 绑定错误、设备访问其他设备的会话；
- 409：重放、会话状态冲突、分片内容冲突、缺片完成；
- 422：全部分片已上传，但整文件完整性校验失败；
- 429：活跃会话或预留配额超限；
- 503：MySQL/Redis 等必要依赖不可用；
- 507：磁盘剩余空间低于安全阈值。

客户端依赖稳定 `code`，不解析可变的 message。

### 15.3 为什么 5xx 重试由 Collector 负责

服务端内部重试会延长请求处理时间，并增加故障期间的请求量，同时还需要保证操作幂等。当前服务端
明确返回失败，由持有本地 payload 的 Collector 按照有最大间隔限制的指数退避策略重试幂等操作；
永久性 4xx 错误不重试。完成接口支持客户端幂等重试。

### 15.4 请求超时如何设置

Nginx 连接超时应较短，读写超时要覆盖正常最大分片传输时间；Collector 请求超时与分片大小、
现场带宽匹配。超时值不宜设置过大，否则异常连接会长期占用资源；也不应小于 p95/p99 正常耗时。

## 16. 性能与容量

### 16.1 做过什么压力测试

本机固定场景下，心跳和归档查询各 300 次，并发 6、到达率 30 RPS：心跳 p95 12.40 ms，查询
p95 4.20 ms，均 300/300 成功。无节流约 248～298 RPS 时出现少量 503，说明当前单实例和本机
MySQL 接入存在过载边界。

该结果仅作为本机回归基线，不代表实际工厂的容量指标。

### 16.2 如何定位 503

先根据 `request_id` 和业务码确认错误来自 MySQL、Redis、配额限制还是服务内部；随后检查连接数、
任务队列、CPU、磁盘、FD、MySQL 慢查询和 Redis 延迟。当前无节流心跳压力下降低并发仍有 503，
说明除了并发数之外还需要控制请求到达率。项目保留真实的过载结果，没有通过自动重试改变测试结果。

### 16.3 如何优化上传吞吐

建议按以下顺序定位和优化：

1. 测量网络、`pwrite`、哈希、MySQL 和 Redis 各阶段耗时；
2. 调整合理分片大小，减少过小分片的请求开销；
3. 保持不同会话并发，评估是否需要缩小按会话串行的文件任务范围；
4. 检查磁盘预分配、页缓存和哈希窗口；
5. 调整 Workflow/MySQL 连接与任务队列；
6. 只有在单实例达到经验证的容量瓶颈后，才设计共享存储和多实例方案。

### 16.4 Bitmap 的空间如何估算

例如 2 GiB 文件按 1 MiB 分片是 2048 片，Bitmap 只需要 2048 bit，即 256 byte；摘要 Hash
远大于 Bitmap，但用于严格幂等检查。容量评估还需考虑 Redis 对象和 Hash field 的额外开销，
不能仅按有效数据长度计算。

### 17.5 遇到过什么实际问题

可以用以下真实案例回答：

**案例一：E2E 条件不可达。** 最初要求 18 个文件同时处于 `UPLOADING`，但服务端限制单设备最多

**案例三：负载边界。** 无节流心跳测试出现 503。压测工具同时记录并发数和请求到达率，并在高于

用于后续分析系统容量边界。

## 18. 部署、备份与故障场景

### 18.1 systemd 做了哪些限制

DataStream 和 Collector 使用不同系统账户，设置 EnvironmentFile、WorkingDirectory、自动重启、
停止超时、`NoNewPrivileges`、`ProtectSystem=strict`、明确 ReadWritePaths、UMask 和 FD 上限，
减少进程被利用后的可写范围。

### 18.2 如何备份与恢复

为了获得一致的恢复点，备份在维护窗口内进行：先停止 Collector，避免产生新的上传任务，再停止
DataStream；使用 mysqldump 备份 MySQL，使用 tar 备份正式归档，并生成 SHA-256 清单。恢复时先
校验备份，再恢复归档正文和 MySQL，并根据数据库摘要进行抽样或全量核验。Redis 保存的是可重建的
运行状态，不作为归档结果的备份来源。

### 18.3 MySQL/Redis 短暂不可用怎么办

DataStream readiness 失败并拒绝相关业务，不使用内存降级。Collector 的不可变 payload 和任务
状态仍在本地，按照指数退避策略重试，达到上限后保持最大间隔等待恢复。恢复后从进度查询继续；
Redis 会话丢失时创建新会话。

### 18.4 磁盘空间不足时如何处理

创建会话前同时检查最小剩余字节、最小剩余比例、全局预留字节和活跃会话配额，提前返回 507。
已有合法会话不会因为创建新会话时的空间检查而被中断。运维侧应扩容或按照正式保留策略转移归档；
清理任务不能将未知文件作为自动清理对象。

### 18.5 如何升级和回滚

升级前记录二进制摘要、配置和迁移版本并完成备份；在预发布环境执行 acceptance；停止 Collector
和服务端，
替换版本化二进制，执行新迁移，先验证 DataStream 健康和心跳再启动 Collector。若已执行不可逆
迁移，需要使用该版本对应的书面回滚方案；直接删除迁移记录并不能恢复已经发生的结构变更。

## 19. 常见方案对比追问

### 19.1 Bitmap 还是 Set

Bitmap 对连续的小整数分片编号具有较高的空间利用率，并且可以直接统计完成情况；Set 更灵活，
但每个整数元素的存储开销较大。当前 `chunk_no` 连续且上限已知，因此 Bitmap 更合适，摘要则
单独使用 Hash 保存。

### 19.2 `mmap` 还是 `read`

两者都可以用于增量哈希。当前选择窗口化 `mmap`，但其性能并非在所有场景下都优于 `read`。如果
现场大量并发哈希导致页缓存压力，需要通过基准测试比较 `mmap` 与 `read` 缓冲方案。

### 19.3 offset 分页还是游标分页

数据量较小时，OFFSET 分页实现简单；对于大量时序归档，keyset 游标更稳定，也不需要扫描较大的
OFFSET。当前存在确定的复合排序键，因此选择游标分页。

### 19.4 Redis 状态还是 MySQL 状态

高频分片位图和临时会话保存在 Redis，避免每个分片都通过事务写入 MySQL；正式归档记录保存在
MySQL。该方案的代价是 Redis 状态丢失后，未完成会话需要重新上传；Collector spool 为恢复提供
原始数据。

### 19.5 HMAC 还是非对称签名

HMAC 计算快、设备实现简单，适合受控设备和服务端共享密钥；缺点是服务端持有可签名密钥，轮换
和泄露影响需要管理。非对称签名更利于私钥留在设备，但设备计算、证书和 PKI 管理更复杂。首期
选择 HMAC，外层仍使用 TLS。

## 24. C++ 通用高频追问

### 24.1 什么是封装、继承和多态

- 封装是把状态和约束放在类型内部，通过有限接口维护不变量；
- 继承表达 is-a 关系，派生类复用并扩展基类接口；
- 运行时多态通常通过虚函数和基类指针/引用调用派生实现。

本项目主要采用组合和按职责划分模块，没有引入大量继承层次。原因是当前只有一种实际实现，
因此没有额外抽象 Repository 接口或 Factory。

### 24.2 为什么多态基类析构函数通常要是 virtual

如果通过基类指针删除派生对象，而基类析构函数不是虚函数，会产生未定义行为，派生类资源可能
无法正确释放。如果类型被设计为多态基类，析构函数通常应声明为虚函数；不用于多态删除的值类型
则没有必要承担虚表开销。

### 24.3 什么是对象切片

把派生对象按值赋给基类对象时，只复制基类子对象，派生部分被切掉，之后虚调用也不再表现为原
派生对象。多态传递应使用基类引用或指针，并明确所有权。

### 24.4 `vector`、`list`、`map`、`unordered_map` 如何选择

- `vector` 连续、缓存友好、随机访问 O(1)，尾部追加摊销 O(1)，中间插入 O(n)；
- `list` 节点分散，已知迭代器位置插删 O(1)，但查找 O(n) 且缓存局部性差；
- `map` 通常是红黑树，有序，查找/插入 O(log n)；
- `unordered_map` 平均 O(1)，最坏 O(n)，无序且需要可靠哈希。

项目中按 ID 管理任务时使用 `std::map`，缺失分片列表使用 `std::vector`，唯一枚举集合使用
`std::set`。回答时除了时间复杂度，还应说明稳定顺序、内存开销和缓存局部性。

### 24.5 `condition_variable` 为什么要配合 while 谓词

条件变量可能虚假唤醒，通知发生后条件也可能已被其他线程消费，所以等待应写成
`cv.wait(lock, predicate)` 或 while 循环重新检查共享状态。修改条件和检查条件必须受同一互斥
保护，停止时通常设置 stop 标志并 `notify_all`。

### 24.6 栈、堆和对象生命周期怎么理解

栈上自动对象随作用域退出析构，堆对象需要由明确所有者释放；RAII 用栈上的管理对象控制堆、FD、
锁等资源。异步编程不能简单地通过“将所有对象分配在堆上”解决生命周期问题，还要保证回调执行时
被捕获对象仍然存活，并避免形成循环引用。

## 25. Linux、网络与 Reactor 通用追问

### 25.1 进程和线程的区别

进程有独立虚拟地址空间和资源隔离，线程共享进程地址空间、FD 和大部分资源，但有独立栈和调度
上下文。线程间通信开销较低，但数据竞争风险较高；进程隔离性更好，但 IPC 成本更高。日志检索
项目根据服务职责拆分进程；本项目的内部模块共享同一部署和存储边界，因此没有额外拆分进程。

### 25.2 文件描述符是什么

FD 是进程文件描述符表的整数索引，可指向文件、socket、pipe 等内核打开文件对象。`dup` 后多个
FD 可共享打开文件描述和偏移；`pread/pwrite` 不依赖共享偏移。泄漏 FD 最终会触发 `EMFILE`，
所以需要 RAII 和 systemd `LimitNOFILE`。

### 25.3 select 和 epoll 的区别

`select` 每次传入并扫描 fd_set，受集合大小和 O(n) 扫描影响；epoll 在内核维护关注集合，
`epoll_wait` 返回就绪事件，更适合大量连接。epoll 的整体处理复杂度不能简单概括为固定 O(1)，
因为还需要考虑就绪事件数量和业务处理成本。

### 25.4 epoll LT 和 ET 的区别

- LT 只要仍可读或可写就会继续通知，编程模型相对简单；
- ET 通常只在状态变化时通知，必须使用非阻塞 FD，并循环读/写到 `EAGAIN`，否则可能遗漏后续处理。

### 25.5 Reactor 模型是什么

Reactor 用 I/O 多路复用器等待事件，再把事件分发给连接处理器；业务任务可交给工作线程或异步
任务系统。它适合 I/O 密集场景。Proactor 更强调异步操作完成通知。Workflow 对这些机制做了
封装，本项目直接使用其任务 API。

### 25.6 TCP 和 UDP 如何选择

TCP 面向连接，提供可靠、有序的字节流，适合文件上传和 HTTP；UDP 无连接、保留报文边界，
数据报可能丢失或乱序，适合可容忍丢包或由应用层实现可靠性的场景。本项目要求完整文件与明确
响应，因此使用 HTTP/TCP。

### 25.7 TCP 粘包是什么

TCP 是字节流，没有消息边界，一次 send 不对应一次 recv。应用协议必须通过长度字段、分隔符或
固定格式拆包。HTTP 已定义请求行、头、Content-Length/chunked 等边界，项目由 Wfrest 解析。

### 25.8 HTTP Keep-Alive 的价值和风险

复用 TCP/TLS 连接减少握手开销，适合连续分片；但空闲连接占 FD 和内存，需要空闲超时、最大请求
数和代理容量控制。超时配置既要覆盖正常业务耗时，也要防止低速或长期空闲连接耗尽资源。

## 26. MySQL 与 Redis 通用基础追问

### 26.1 为什么 MySQL 索引常用 B+ 树

B+ 树具有较高的扇出和较低的树高，内部节点只保存键和子指针，叶子节点按顺序链接，适合磁盘页
访问和范围扫描。
普通二叉树高度更高，哈希索引不适合范围和排序。

### 26.2 聚簇索引和二级索引是什么

InnoDB 主键 B+ 树叶子保存整行，是聚簇索引；二级索引叶子通常保存主键，查询非覆盖字段时需要
回表。联合索引如果覆盖列表所需字段，可以减少回表；但不应仅为了覆盖所有字段而持续扩大索引。

### 26.3 四种事务隔离级别

- Read Uncommitted：可能脏读；
- Read Committed：避免脏读，但同一事务两次读取可能不同；
- Repeatable Read：一致性读通常保持同一快照，InnoDB 还用 next-key lock 处理部分幻读场景；
- Serializable：最强隔离，并发成本最高。

隔离级别解决的是数据库事务并发问题，无法使文件系统和 Redis 自动加入同一事务。

### 26.4 什么是 MVCC

MVCC 通过事务版本、undo log 和 read view 为一致性读选择可见版本，从而减少读写之间的阻塞。
加锁读、更新和删除等当前读仍需读取最新版本，并参与锁竞争。

### 26.5 索引缺失为什么会扩大锁影响范围

InnoDB 锁基于索引记录。缺少可用索引时更新要扫描大量记录并锁住更多范围；范围条件还可能使用
gap/next-key lock。应通过索引、短事务和一致访问顺序减少锁范围和死锁。

### 26.6 什么是 Cache-Aside

读：先查缓存，未命中查数据库并回填；写：通常先更新数据库，再删除缓存。它仍有并发一致性窗口，
需要根据业务一致性要求选择延迟双删、版本控制或消息失效等方案，具体方案需要结合业务场景确定。
本项目当前没有为归档查询增加缓存。

### 26.7 缓存穿透、击穿、雪崩分别是什么

- 穿透：大量查询访问不存在的数据，缓存持续未命中并直接访问数据库；可使用空结果缓存、布隆
  过滤器或输入约束；
- 击穿：单个热点键过期，大量请求同时访问数据库并重建缓存；可使用互斥重建、逻辑过期或提前刷新；
- 雪崩：大量键同时失效或缓存整体故障；可为 TTL 增加随机偏移，并结合限流、隔离和降级措施。

这些是简历中的通用技能，不应表述为 DataStream 已经实现的能力。

### 26.8 Redis 为什么快

主要数据在内存，命令执行模型避免大量共享数据锁竞争，数据结构针对场景优化，并用 I/O 多路复用
处理连接。Redis 6 之后部分网络 I/O 可以使用多线程，因此不能再将其简单概括为“所有工作始终
由单线程完成”；但单条命令的原子执行语义仍然成立。

## 27. 日志检索项目需要单独准备的内容

当前 DataStream 不包含以下实现，但它们写在简历的第二个项目中，面试官很可能连续追问：

### 27.1 Protobuf 相比 JSON 的特点

Protobuf 是基于 schema 的二进制序列化格式，体积较小、解析效率较高，字段编号支持兼容演进；
缺点是二进制内容不便直接阅读，并且需要生成代码。字段编号发布后不能重新用于其他字段，新增
optional/repeated 字段通常可以保持兼容。

### 27.2 SRPC 超时和错误如何映射

应区分调用前参数错误、网络不可达、RPC 超时、服务端业务错误和反序列化/协议错误，再映射为稳定
HTTP 状态与业务码。超时后客户端不知道服务端是否执行完成，写操作仍需要幂等。

### 27.3 倒排索引是什么

从 term 映射到包含该 term 的 posting list，posting 中可保存 docId、词频和位置。AND 查询可先
处理文档频率低的关键词，缩小候选集合，再计算相关性。

### 27.4 BM25 的核心思想

分数综合词频 TF、逆文档频率 IDF 和文档长度归一化。词频增加带来的收益逐渐饱和，常见参数
`k1` 控制饱和速度，`b` 控制长度归一化。业务精确字段可以作为额外的权重项，但权重需要通过
离线评估和真实查询数据验证。

### 27.5 Top-K 为什么用小顶堆

遍历候选结果时维护大小为 K 的小顶堆，堆顶是当前第 K 大分数；当新分数高于堆顶时执行替换。
时间复杂度约为 O(n log K)，低于对 n 个结果全部排序所需的 O(n log n)。

### 27.6 SLRU 是什么

SLRU 分 Probation 和 Protected。新对象先进入 Probation，重复命中后晋升 Protected；Protected
淘汰时可降回 Probation。这样一次性的历史扫描主要占用 Probation 空间，不容易淘汰稳定热点。

### 27.7 为什么索引记录 offset 和 length

索引只保存检索信息与原日志位置，命中后用 `pread` 从归档文件按偏移回读，避免正文同时复制到
索引和 MySQL。被索引文件需要保持不可变，否则记录的 offset 将失效；DataStream 的正式归档为
日志检索提供了这一上游条件。

由于 `02_SMT_LogTrace` 代码尚未完成，这部分目前只能作为原理和设计回答，尚无当前仓库中的编译、
E2E 测试和性能数据作为实现证据。

## 28. 工具链与协作追问

### 28.1 Debug 和 Release 有什么区别

Debug 通常保留调试符号，并关闭或降低优化，便于设置断点和运行 Sanitizer；Release 开启优化，
行为和时序可能不同。因此 Debug 和 Release 两种构建都需要编译并执行相应测试。

### 28.2 gdb 常用排查流程

使用带 `-g` 的二进制，设置断点或捕获信号，运行后查看 `bt`、`frame`、`info locals`、线程列表；
崩溃时结合 core dump 和完全匹配的二进制及调试符号。优化构建中部分变量可能因优化而无法直接
查看，函数调用也可能被内联。

### 28.3 CMake 在项目中负责什么

定义 C++11、依赖查找、核心库与两个可执行程序、`-Werror`、测试开关、集成测试、覆盖率和
Sanitizer 编译链接选项。构建目录与源码分离，可从空目录复现。

### 28.4 Git 冲突如何处理

先理解双方的修改目的，在最小范围内完成合并，再重新编译和测试。解决冲突时需要保留双方必要的
修改，不能只根据 `ours` 或 `theirs` 选项直接覆盖。数据库迁移发布后不再修改旧文件，而是新增
迁移版本，避免不同环境的 checksum 和数据库结构出现差异。

### 28.5 如何做代码评审

先确认输入边界和错误处理责任，再检查是否存在重复校验、捕获异常后不处理、使用默认值掩盖错误、
不可达分支、无实际用途的抽象、敏感信息日志和资源泄漏；最后检查测试是否验证真实接口约定，
而不是在生产代码中加入测试专用分支。

# 核心调用链与面试演示

## 1. 调用链

设备请求先由 Wfrest 路由接收，`DeviceAuthenticator` 校验头、MySQL 设备状态、正文摘要、HMAC 和
Redis 防重放。创建会话进入 `UploadRepository`，完成配额判断、`posix_fallocate` 和 Redis
Hash/Bitmap；分片接口按偏移 `pwrite` 后原子记录摘要和位图。完成接口冻结会话，
`ArchiveStorage` 用窗口化 `mmap` 校验整文件，`rename` 到确定性路径，`ArchiveRepository` 写入
MySQL 唯一记录，最后修正 Redis 状态。

Collector 的链路是 `scanInputs -> metadataTask -> SpoolStore::snapshot/save -> HttpUploader ->
processTask`。每轮只推进一个明确状态；重启从 JSON 状态恢复，服务端会话丢失回到 `READY`，
永久业务错误进入 `FAILED`。

## 2. 十分钟演示

```bash
export SMT_DATASTREAM_MYSQL_PASSWORD='123456'
export SMT_DATASTREAM_OPERATOR_TOKEN='e2e-operator-token-2026'
scripts/db.sh migrate --config conf/datastream.example.json
scripts/db.sh seed --config conf/datastream.example.json
cmake --build build --parallel 2
ctest --test-dir build -R 'UploadE2ETest|CollectorE2ETest' --output-on-failure
```

讲解重点：为什么 Redis Bitmap 适合缺片查询；为何 `pwrite` 之后才置位；为什么临时区与正式区
必须同文件系统；`rename` 与 MySQL 之间无法形成分布式事务时如何通过确定性路径、唯一约束和
重复完成恢复；为何 Collector 先做不可变快照；为何不会为每个理论异常增加兜底。

压测演示在独立测试实例执行：

```bash
python3 tests/http_load/http_load_test.py --scenario heartbeat --requests 300 --concurrency 6 \
  --rate-per-second 30
python3 tests/http_load/http_load_test.py --scenario query --requests 300 --concurrency 6 \
  --rate-per-second 30 \
  --operator-token "$SMT_DATASTREAM_OPERATOR_TOKEN"
```

面试时应说明这些数字是本机固定场景结果，不外推为工厂容量；真实容量需要设备节拍、平均/尾部
文件大小、断网时长、磁盘吞吐和恢复时间目标共同确定。

# 数据与状态模型

## 1. 标识与时间

- `line_id`、`station_id`、`device_id` 使用业务可读 ASCII 编号并由运维预置；
- `upload_id` 使用 OpenSSL `RAND_bytes` 生成的 UUIDv4；
- `archive_id` 使用 MySQL `BIGINT UNSIGNED AUTO_INCREMENT`；
- 数据库时间均存 UTC `DATETIME(3)`；
- HTTP 输入输出按 `API契约.md` 转换，不依赖服务器本地时区；
- SHA-256 在 HTTP/Redis 使用 64 字符十六进制小写，在 MySQL 使用 `BINARY(32)`。

## 2. MySQL 表

正式 DDL 在第一期代码阶段以版本化迁移生成，字段和索引必须符合本节。

### 2.1 `schema_migration`

| 字段 | 类型 | 约束 |
|---|---|---|
| `version` | `VARCHAR(32)` ASCII | 主键 |
| `checksum` | `CHAR(64)` ASCII | 非空，迁移文件 SHA-256 |
| `applied_at` | `DATETIME(3)` | 非空，UTC |

迁移工具发现同版本 checksum 不一致时停止，不修改已执行记录。

### 2.2 `production_line`

| 字段 | 类型 | 约束 |
|---|---|---|
| `line_id` | `VARCHAR(64)` ASCII | 主键 |
| `line_name` | `VARCHAR(128)` UTF-8 | 非空 |
| `enabled` | `TINYINT(1)` | 非空，默认 1 |
| `created_at` / `updated_at` | `DATETIME(3)` | 非空，UTC |

### 2.3 `station`

| 字段 | 类型 | 约束 |
|---|---|---|
| `station_id` | `VARCHAR(64)` ASCII | 主键 |
| `line_id` | `VARCHAR(64)` ASCII | 非空，外键到产线 |
| `station_name` | `VARCHAR(128)` UTF-8 | 非空 |
| `station_type` | `VARCHAR(16)` ASCII | 当前 `001` 为 `SPI`、`AOI`、`ICT`；第三期迁移为 `SPI`、`AOI`、`FCT` |
| `enabled` | `TINYINT(1)` | 非空，默认 1 |
| `created_at` / `updated_at` | `DATETIME(3)` | 非空，UTC |

索引：`INDEX idx_station_line(line_id, enabled)`。

### 2.4 `device`

| 字段 | 类型 | 约束 |
|---|---|---|
| `device_id` | `VARCHAR(64)` ASCII | 主键 |
| `station_id` | `VARCHAR(64)` ASCII | 非空，外键到工位 |
| `device_model` | `VARCHAR(128)` UTF-8 | 非空 |
| `software_version` | `VARCHAR(64)` ASCII | 可空，心跳更新 |
| `hmac_secret` | `VARBINARY(64)` | 非空，不参与普通查询输出 |
| `enabled` | `TINYINT(1)` | 非空，默认 1 |
| `last_seen_at` | `DATETIME(3)` | 可空，UTC |
| `created_at` / `updated_at` | `DATETIME(3)` | 非空，UTC |

索引：`INDEX idx_device_station(station_id, enabled)`。

设备密钥为随机字节，不是用户密码哈希；验证 HMAC 需要原始共享秘密。第一期内网部署依靠
MySQL 最小权限、TLS/本机网络和禁止输出保护该字段，不虚构尚未部署的 KMS。

### 2.5 `collector_device_binding`（第三期新增）

| 字段 | 类型 | 约束 |
|---|---|---|
| `collector_id` | `VARCHAR(64)` ASCII | 与 `device_id` 组成主键 |
| `device_id` | `VARCHAR(64)` ASCII | 外键到设备 |
| `enabled` | `TINYINT(1)` | 非空，默认 1 |
| `created_at` / `updated_at` | `DATETIME(3)` | 非空，UTC |

首期不单独给 IPC 建立登录账号。IPC 受控保存绑定设备的开发密钥，以数据源设备身份签名；服务端
在心跳和创建上传时校验 `collector_id` 与已认证 `device_id` 的启用绑定。

### 2.6 `archive_file`

| `device_id` | `VARCHAR(64)` ASCII | 非空 |
| `collector_id` | `VARCHAR(64)` ASCII | 非空 |

索引：

- `UNIQUE uk_archive_upload(upload_id)`；

列表查询选择与过滤前缀匹配的索引。第一期不增加每个字段的单列索引，也不保存文件正文。

## 3. SQL 构造规则

- 外部字符串经边界长度/字符校验后，再用 Workflow `MySQLUtil::escape_string_quote` 转义；
- 数字和枚举在解析成内部类型后格式化，不直接使用请求原文；
- 排序字段、方向和表名固定在源码，不能由请求字符串选择；
- 可选过滤条件只从 API 契约中的白名单生成；
- 写归档记录使用事务和唯一键，唯一冲突后按 `upload_id` 查询已有记录；
- 任何 MySQL 错误都检查 Workflow state、协议错误包和服务端错误码，不把空结果当错误兜底。

## 4. Redis 键与字段

开发环境前缀为 `smt:dev:`，集成测试使用每次运行唯一的 `smt:test:<run_id>:`。生产前缀由配置
固定，不能由 HTTP 请求指定。

### 4.1 防重放

```text
auth:req:<device_id>:<request_id>
```

String，值为请求时间，TTL 600 秒。使用单条 `SET key value NX EX 600`。已存在表示重放；
Redis 失败表示认证依赖不可用，不能放行。

### 4.2 心跳

```text
heartbeat:<device_id>
```

Hash 字段：`collector_id`、`software_version`、`runtime_status`、`work_order`、`reported_at`、
`last_seen_at`。TTL 为约定心跳周期三倍；首期心跳周期 30 秒，TTL 90 秒。`online` 不存储，
读取时由 `last_seen_at` 计算。

### 4.3 上传会话

```text
upload:<upload_id>
```

Hash 字段：

| 字段 | 内容 |
|---|---|
| `state` | `UPLOADING`、`VERIFYING`、`ARCHIVED`、`FAILED` |
| `device_id` / `station_id` / `line_id` / `collector_id` | 归属信息 |
| `work_order` / `product_sn` | 可空追溯字段，空值使用空字符串存储 |
| `file_type` / `result` / `original_filename` | 文件业务字段 |
| `file_size` / `file_sha256` | 整文件期望值 |
| `chunk_size` / `chunk_count` | 分片参数 |
| `temp_path` / `relative_path` | 服务端生成路径 |
| `produced_at` / `created_at` / `expires_at` | UTC 毫秒时间 |
| `archive_id` / `archived_at` | 成功归档后填写 |
| `failure_code` | FAILED 时填写稳定错误码 |

会话初始 TTL 24 小时。每个成功的新分片统一刷新会话、Bitmap 和摘要 Hash 的 TTL；重复分片
成功不刷新 TTL，进度查询不刷新 TTL。

### 4.4 分片位图和摘要

```text
upload:<upload_id>:chunks
upload:<upload_id>:digests
```

Bitmap 的 bit offset 是 `chunk_no`。摘要 Hash 的 field 是十进制 `chunk_no`，value 是
`X-Content-SHA256`。位图用于完整性和进度，摘要用于幂等内容比较，两者职责不重复。

## 5. Redis 原子操作

只为跨多个 Redis 键且必须原子的六处操作编写固定 Lua 脚本，不建设脚本框架：

1. 创建会话：确认会话键不存在，写会话 Hash 并设置 TTL；位图和摘要键在首个分片完成时创建；
2. 开始分片：确认状态为 UPLOADING，原子登记摘要；不同摘要直接冲突，相同摘要且位图已完成
   返回幂等成功；
3. 完成分片：`pwrite` 写满后置 Bitmap，并刷新会话、Bitmap、摘要和配额到期时间；
4. 开始完成：确认状态为 UPLOADING 且 `BITCOUNT == chunk_count`，切换为 VERIFYING。
5. 标记失败：写入稳定失败码，缩短会话、Bitmap、摘要和配额的诊断保留时间；
6. 标记归档：写入归档编号和路径，缩短三个会话键保留时间并原子释放配额。

脚本返回固定整数结果，由 Redis 适配层映射为领域结果。脚本不读写文件系统或 MySQL。

## 6. 上传状态机

```text
                 会话过期
UPLOADING ----------------------> Redis 键过期 / 清理任务判定 EXPIRED
    |
    | 全部分片完成，原子占有完成权
    v
VERIFYING ----------------------> ARCHIVED
    |
    | 大小或 SHA-256 不一致、确定不可恢复的文件错误
    v
FAILED
```

规则：

- `UPLOADING` 接受合法分片和进度查询；
- `VERIFYING` 拒绝新分片，允许同设备重试 complete 以恢复中断流程；
- `ARCHIVED` 的 complete 重试返回同一 `archive_id`；
- `FAILED` 不再接收分片或 complete，客户端创建新会话；
- EXPIRED 不作为长期 Redis 值保存，键过期后 API 返回 `UPLOAD_NOT_FOUND`，清理日志使用该词；
- MySQL 已存在唯一 `upload_id` 时，以 MySQL 归档事实为准并修复 Redis ARCHIVED 状态。

## 7. 文件系统模型

临时文件：

```text
<temp_root>/<upload_id>.part
```

正式文件：

```text
<archive_root>/<YYYY>/<MM>/<DD>/<line_id>/<station_id>/<device_id>/
<file_type>/<upload_id>.<validated_extension>
```

`validated_extension` 从原文件名提取并与文件类型允许列表比较；未知或不允许的扩展名在创建
会话时明确拒绝，不改名成任意默认扩展。所有父目录由服务端以固定权限创建。启动时使用设备号
比较确认 temp/archive 位于同一文件系统。

清理扫描只处理名称严格符合 `<upload_id>.part` 且 mtime 超过保留期的临时文件。其他文件记录
告警但不删除，避免把不明文件当垃圾兜底处理。

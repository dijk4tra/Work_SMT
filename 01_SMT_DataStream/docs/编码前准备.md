# 编码前准备与第一期终板契约

## 1. 已确定的工程基线

| 项目 | 首期约定 |
|---|---|
| 操作系统 | Linux |
| 语言标准 | C++11 |
| 构建 | CMake |
| HTTP 与异步任务 | Wfrest / Workflow |
| 数据库 | Workflow MySQL 异步任务；MySQL 保存设备、工位和归档元数据 |
| 缓存/状态 | Workflow Redis 异步任务；Redis 保存心跳、防重放、会话、位图和分片摘要 |
| 密码能力 | OpenSSL HMAC-SHA256、SHA-256、常量时间比较 |
| JSON | nlohmann/json |
| 日志 | spdlog |
| 文件 I/O | `posix_fallocate`、`pwrite`、窗口化 `mmap`、`rename` |
| 单元测试 | GoogleTest 1.11.0 + CTest |
| 生产入口 | Nginx TLS 反向代理到单个 Wfrest 服务实例 |
| 测试数据 | `data/mock_inbox/run_*` |

本机依赖已经完成 C++11 联编，Workflow 对 MySQL/Redis 的实际请求验证通过。终板 CMake
不链接本机按 C++17 构建的 redis++，Redis 访问统一使用 Workflow。

## 2. 首期配置项

配置基线见 `conf/datastream.example.json`。敏感值不直接写入配置文件：示例中的
MySQL 的 `password_env` 表示从对应环境变量读取密码。本机 Redis 已恢复为无密码
`default` 用户，因此示例中的 Redis `password_env` 为 `null`；配置加载器仍支持部署环境指定
密码变量。Operator Token 只从环境变量读取。设备 HMAC 密钥由设备登记数据提供，不得写在
普通日志或响应中。

以下值是开发环境默认值，可在压测后调整，而不是写死在业务代码中：

- 请求时间容差 300 秒，防重放键 600 秒；
- 设备心跳 TTL 90 秒；
- 上传会话 24 小时；
- 分片 1 MiB 至 8 MiB，单文件上限 2 GiB；
- SHA-256 的 `mmap` 窗口 64 MiB；
- 清理任务每 10 分钟运行一次。
- 查询默认 50 条、最多 100 条；无精确追溯字段时查询时间范围最多 31 天；
- 优雅停止最多等待 30 秒；单日志文件 100 MiB，保留 10 个轮转文件。

配置加载失败、字段类型错误、路径跨文件系统或目录不可写都属于启动失败，服务不带默认路径
继续运行。示例值只用于开发，不是解析失败时的兜底值。

## 3. HTTP 接口契约

业务接口统一前缀为 `/api/v1`，请求和响应 JSON 使用 UTF-8。设备接口执行 HMAC 校验；归档
查询执行 Operator Bearer Token 校验；健康检查不要求认证且不返回依赖详情。

| 方法与路径 | 用途 | 主要成功响应 |
|---|---|---|
| `GET /health/live` | 判断进程是否存活 | 固定存活状态 |
| `GET /health/ready` | 判断依赖与目录是否就绪 | 总体就绪状态 |
| `POST /devices/heartbeat` | 上报设备状态 | `last_seen_at`、计算后的状态 |
| `POST /uploads` | 创建上传会话 | `upload_id`、分片大小/数量、过期时间 |
| `PUT /uploads/{upload_id}/chunks/{chunk_no}` | 上传一个二进制分片 | 分片编号、是否为幂等命中 |
| `GET /uploads/{upload_id}` | 查询续传进度 | 状态、完成与缺失分片编号 |
| `POST /uploads/{upload_id}/complete` | 完整性校验并归档 | `archive_id`、归档时间 |
| `GET /archives` | 组合条件查询归档元数据 | 分页结果列表 |
| `GET /archives/{archive_id}` | 查询一条归档详情 | 完整元数据，不含文件正文 |

创建会话的请求体：

```json
{
  "station_id": "ST-AOI-01",
  "collector_id": "IPC-L01-01",
  "work_order": "WO-20260711-001",
  "product_sn": "CTRL-MB-A1-260711-000001",
  "file_type": "NG_IMAGE",
  "original_filename": "board_U3_bridge.png",
  "file_size": 921604,
  "file_sha256": "64位十六进制小写摘要",
  "chunk_size": 1048576,
  "produced_at": "2026-07-11T08:00:30.000+08:00",
  "result": "NG"
}
```

统一响应结构：

```json
{
  "code": "OK",
  "message": "success",
  "request_id": "设备提交的请求标识",
  "data": {}
}
```

失败响应的 HTTP 状态码表达协议层类别，`code` 表达稳定业务原因。`message` 用于诊断，不供
客户端逻辑匹配。

## 4. 业务错误码

| HTTP | `code` | 场景 |
|---:|---|---|
| 400 | `INVALID_ARGUMENT` | JSON、字段格式、分片编号或长度非法 |
| 401 | `AUTH_REQUIRED` | 缺少认证头 |
| 401 | `SIGNATURE_INVALID` | 正文摘要或 HMAC 不匹配 |
| 401 | `TIMESTAMP_EXPIRED` | 请求时间超出允许窗口 |
| 401 | `OPERATOR_TOKEN_INVALID` | Operator Token 缺失或不匹配 |
| 409 | `REQUEST_REPLAYED` | 请求标识已使用 |
| 403 | `DEVICE_DISABLED` | 设备已停用 |
| 404 | `DEVICE_NOT_FOUND` | 设备未登记 |
| 404 | `UPLOAD_NOT_FOUND` | 会话不存在或已过期 |
| 409 | `UPLOAD_STATE_CONFLICT` | 当前状态不允许写分片或完成 |
| 409 | `CHUNK_CONTENT_CONFLICT` | 已完成编号收到不同内容 |
| 409 | `CHUNKS_INCOMPLETE` | 仍有缺失分片 |
| 422 | `FILE_INTEGRITY_MISMATCH` | 最终大小或 SHA-256 不一致 |
| 500 | `STORAGE_IO_ERROR` | 文件系统操作失败 |
| 503 | `MYSQL_UNAVAILABLE` | 当前请求依赖的 MySQL 调用失败 |
| 503 | `REDIS_UNAVAILABLE` | 当前请求依赖的 Redis 调用失败 |

不要增加一个包住所有异常并统一返回 500 的全捕获层。各适配边界把库错误转换成上述必要
类别，未知程序错误由框架终止当前请求并记录，随后修复根因。

## 5. MySQL 模型

### 5.1 `production_line`

保存 `line_id`、名称、状态和创建/更新时间。`line_id` 唯一。

### 5.2 `station`

保存 `station_id`、`line_id`、工位名称、类型（SPI/AOI/ICT）和状态。
`station_id` 唯一，`line_id` 建普通索引。

### 5.3 `device`

保存 `device_id`、`station_id`、设备型号、软件版本、启用状态、HMAC 密钥、最近心跳时间和
创建/更新时间。`device_id` 唯一，`station_id` 建普通索引。HMAC 密钥字段不出现在列表查询
模型中；生产部署是否接入专用密钥系统属于部署安全设计，不在首期本地项目中伪造实现。

### 5.4 `archive_file`

确定字段：

| 字段 | 含义 |
|---|---|
| `archive_id` | `BIGINT UNSIGNED AUTO_INCREMENT` 主键 |
| `upload_id` | 上传会话编号，唯一 |
| `line_id` / `station_id` / `device_id` | 生产位置和数据源设备 |
| `collector_id` | 采集工控机 |
| `work_order` / `product_sn` | 追溯字段，可按文件类型为空 |
| `file_type` | 检测结果、报告、NG 图片、导出文件、运行记录 |
| `result` | PASS、NG 或不适用 |
| `original_filename` | 设备侧原文件名，仅展示 |
| `relative_path` | 归档根目录下的服务端安全路径，唯一 |
| `file_size` / `file_sha256` | 无符号文件大小与 `BINARY(32)` 摘要 |
| `produced_at` / `archived_at` | 设备产生时间与平台归档时间 |

联合索引见 `开发思路.md`。所有时间统一以带时区的输入解析后存 UTC，API 输出 ISO 8601。
数据库时间使用 UTC `DATETIME(3)`，API 输出带 `Z` 的 ISO 8601。摘要在数据库使用固定
32 字节二进制，HTTP 使用 64 字符小写十六进制，并由存储边界完成唯一转换。

上传中的临时状态不写 MySQL，避免每个分片产生事务；归档完成后才写 `archive_file`。

## 6. Redis 键

键统一带环境前缀，例如开发环境 `smt:dev:`。

| 键模式 | 类型 | TTL | 内容 |
|---|---|---:|---|
| `auth:req:{device_id}:{request_id}` | String | 600 秒 | 防重放占位值 |
| `heartbeat:{device_id}` | Hash | 心跳周期的 3 倍 | 状态、版本、工单、最后时间 |
| `upload:{upload_id}` | Hash | 24 小时 | 会话元数据、状态、失败原因 |
| `upload:{upload_id}:chunks` | Bitmap | 与会话一致 | 已完成分片位 |
| `upload:{upload_id}:digests` | Hash | 与会话一致 | 分片编号到 SHA-256 |

创建会话时一次设置相关 TTL。会话有活动时是否续期应由接口契约固定；首期约定每次成功写入
分片后刷新会话、位图和摘要键的 TTL。查询进度不续期，避免无人上传的轮询永久占用资源。

## 7. 文件类型与字段约束

| 文件类型 | `product_sn` | `result` | 常见扩展名 |
|---|---|---|---|
| `DETECTION_RESULT` | 必填 | PASS/NG | `.json`、`.csv` |
| `TEST_REPORT` | 必填 | PASS/NG | `.csv`、`.json`、`.pdf` |
| `NG_IMAGE` | 必填 | 固定 NG | `.png`、`.jpg`、`.jpeg` |
| `DEVICE_EXPORT` | 可选 | 不适用 | `.zip`、`.csv` |
| `RUNTIME_LOG` | 可选 | 不适用 | `.log`、`.txt` |

扩展名只参与展示和归档命名，不能替代内容类型和哈希校验。首期不解析设备文件正文来重新
推断业务字段；采集端提交的元数据经过边界校验后保存。

## 8. 测试准备

### 8.1 单元测试

- 规范签名串、正确/错误 HMAC 和常量时间比较封装；
- 时间窗口与防重放结果映射；
- 分片偏移、最后一片长度和缺失分片计算；
- 归档相对路径生成，覆盖恶意原文件名不进入路径；
- 分页和业务错误到 HTTP 状态映射。

### 8.2 集成测试

- MySQL 设备/工位查询和归档唯一约束；
- Redis 防重放原子写、位图、摘要和 TTL；
- 乱序上传、重复同内容分片、不同内容冲突和断点补传；
- 正确文件归档、缺片、大小不符、摘要不符和会话过期；
- `rename` 后元数据写入失败的幂等恢复；
- 清理过期临时文件但不触碰正式归档。

### 8.3 端到端与资源测试

使用 `data/mock_inbox` 清单逐个上传 JSON、CSV、PNG、ZIP 和 LOG，归档后重新计算大小与
SHA-256，并按设备、工单、产品 SN 和时间查询。大文件测试另生成稀疏或随机文件，不把巨大
二进制样本提交仓库。

使用并发请求验证多个会话和同一会话的顺序保护，使用 Valgrind/ASan 检查资源生命周期并执行
固定基础压力场景。没有指标前不提前加入缓存、批处理或复杂限流实现。

## 9. 首期验收标准

- 未登记、停用、签名错误、过期和重放请求被明确拒绝；
- 心跳可更新，且在线状态由最后心跳时间稳定计算；
- 至少完成 0、1、最后分片乱序上传和断线补传；
- 完成接口只在分片齐全且大小、SHA-256 正确时产生一条归档记录；
- 同一完成请求重复调用不产生重复文件和记录；
- 历史查询能用样本数据按设备、工单、产品 SN、文件类型和时间过滤；
- 过期/失败临时文件能被清理，正式归档不会被清理任务删除；
- C++ 源码符合根目录注释规范，能够通过构建、测试和资源检查。

## 10. 已锁定的正式编码前决定

- 没有既有采集客户端需要兼容，终板以本文 `v1` 契约实现；
- 单文件上限 2 GiB，分片范围 1 MiB 至 8 MiB，会话 TTL 为 24 小时；
- 使用本机 MySQL `root` 账号并通过环境变量注入密码；Redis 使用已恢复的 `default` 用户；
- 第一期开启设备写接口和 Operator 元数据查询，不提供文件正文下载；
- 部署为单个 C++ 服务实例、本地同文件系统归档、Nginx TLS 入口；
- 采用 `分期实施计划.md` 的五期顺序，五期完成后统一交付生产终板；
- 所有代码和测试由 AI 生成，阶段之间不保留需要用户补写的占位实现。

除非用户后续明确修改外部契约，正式代码按以上决定推进，不额外猜测多节点、对象存储、RBAC、
消息队列或日志全文检索需求。

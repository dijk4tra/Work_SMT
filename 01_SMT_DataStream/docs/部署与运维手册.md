# 部署与运维手册

## 1. 生产边界

DataStream 单实例监听 `127.0.0.1:8080`，Nginx 在 `8443` 终止 TLS。MySQL 保存业务元数据，
Redis 保存可重建的实时状态与上传会话，归档正文保存在本地文件系统。生产证书、数据库密码、
Operator Token 和九台设备密钥必须由部署系统写入 `/etc/smt-datastream`，不得提交仓库。

建议账户和目录：

| 路径 | 所有者 | 用途 |
|---|---|---|
| `/opt/smt-datastream/bin` | root | 两个只读可执行文件 |
| `/etc/smt-datastream` | root | JSON、环境文件和 TLS 私钥 |
| `/var/lib/smt-datastream` | smt-datastream | 上传临时区与正式归档 |
| `/var/lib/smt-collector` | smt-collector | 本地 spool |
| `/srv/smt-device-inbox` | smt-collector | 九个设备投递目录 |
| `/var/log/smt-datastream` | smt-datastream | 服务日志 |

## 2. 构建和安装

```bash
scripts/build.sh release
sudo install -m 0755 build-release-clean/datastream_server /opt/smt-datastream/bin/
sudo install -m 0755 build-release-clean/collector_agent /opt/smt-datastream/bin/
```

服务端从 `conf/datastream.production.example.json` 复制；采集端复制 `collector.example.json`
后，将 spool 和九个输入目录改为上表绝对路径。配置目录为 `0755 root:root`，普通 JSON
为 `0644`；环境文件权限必须为 `0640 root:<service-group>`，TLS 私钥为 `0600 root:root`。其中
Redis 默认用户仍为 `default`；只有部署环境实际启用 Redis ACL 密码时才配置 `password_env`。

## 3. 迁移和启动

先备份，再显式执行迁移；生产环境不执行开发 seed。

```bash
export SMT_DATASTREAM_MYSQL_PASSWORD='从密钥系统读取'
scripts/db.sh migrate --config /etc/smt-datastream/datastream.json
sudo systemd-analyze verify deploy/systemd/*.service
sudo systemctl daemon-reload
sudo systemctl enable --now smt-datastream.service
scripts/health_check.sh http://127.0.0.1:8080
sudo nginx -t
sudo systemctl reload nginx
sudo systemctl enable --now smt-collector.service
```

启动顺序固定为 MySQL/Redis、DataStream、Nginx、Collector。Collector 在服务未就绪时将已封口
文件保存在 spool，并按封顶退避继续尝试；DataStream 自身不提供无 MySQL/Redis 的降级模式。

## 4. 日常观测

- `/health/live` 判断进程是否响应，`/health/ready` 判断依赖和存储是否可用；
- `journalctl -u smt-datastream -u smt-collector --since today` 查看进程事件；
- `du -sb /var/lib/smt-datastream /var/lib/smt-collector` 观察归档、临时区和积压；
- MySQL 按设备/工位/工单索引查询，Redis `INFO memory` 观察会话内存；
- 告警关注 readiness 持续失败、spool 接近上限、磁盘水位、永久失败任务和 5xx。

## 5. 备份与恢复

一致恢复点采用维护窗口：先停 Collector 阻止新输入，再停 DataStream，执行数据库与归档目录
备份，完成后逆序启动。脚本生成 SQL、归档 tar 和独立 SHA-256 清单：

```bash
sudo systemctl stop smt-collector smt-datastream
scripts/backup.sh --config /etc/smt-datastream/datastream.json /backup/smt-20260712
scripts/verify_backup.sh /backup/smt-20260712
sudo systemctl start smt-datastream smt-collector
```

恢复必须先在隔离目录校验清单，再恢复归档目录，随后导入 MySQL；启动前抽样或全量比较
`archive_file.file_sha256` 与正文。Redis 不从备份恢复：未完成的 Collector 任务会查询不到旧会话
并创建新会话，已归档记录依靠 MySQL 唯一约束保持幂等。

## 6. 升级与回滚

1. 记录当前二进制 SHA-256、配置和迁移版本并完成备份；
2. 在预发布环境执行 `scripts/acceptance.sh`；
3. 停 Collector，再停 DataStream，替换带版本后缀的二进制并原子更新部署链接；
4. 运行新迁移、启动 DataStream、验证健康与一个签名心跳，再启动 Collector；
5. 若新版本在未执行不可逆迁移前失败，恢复旧链接和配置；若已执行迁移，必须按该版本的书面
   数据回滚方案处理，禁止直接修改 `schema_migration` 伪造回退。

## 7. 配置审查

`client_max_body_size` 必须不小于服务 `request_body_limit_bytes`，且服务限制不小于最大分片；
Nginx 上传超时应大于正常单分片传输时间。临时区和归档区必须处于同一文件系统。修改会话 TTL、
磁盘水位或并发配额前要依据产线速率和故障恢复目标重新容量评估。

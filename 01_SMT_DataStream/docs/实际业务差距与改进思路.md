# SMT 数据采集归档平台实际业务差距与改进思路

## 1. 文档目的

本文基于仓库根目录 `SMT产线项目.md`、当前项目 README、API 契约、数据模型、分期计划和
现有代码，对“SMT 检测工位数据采集与归档管理平台”的开发思路进行业务合理性复核，集中记录：

- 当前设计中已经合理的部分；
- 仍然存在的设计和工程问题；
- 与真实 SMT 产线业务之间的主要差距；
- 对应的改进方向、优先级和验收依据。

本文只分析 `01_SMT_DataStream`。日志解析、倒排索引、BM25、SLRU Cache 和日志检索服务属于
`02_SMT_LogTrace`，不应混入数据采集归档项目。

## 2. 评估依据与限制

本次评估依据包括：

1. 根目录 `SMT产线项目.md` 中对项目经历和技术事实的描述；
2. `README.md`、`开发思路.md`、`API契约.md`、`数据与状态模型.md` 和 `分期实施计划.md`；
3. 当前仓库中已经实现的配置、健康检查、设备认证、防重放和心跳代码；
4. SMT 现场常见的设备文件采集、质量追溯和生产运维场景。

`SMT产线项目.md` 是项目经历摘要，不是完整的工厂需求规格。当前资料没有给出实际工厂的设备
厂商清单、文件产生机制、日产量、峰值吞吐、网络条件、归档保留年限、停机容忍时间、备份恢复
目标和权限组织。因此，本文可以判断设计是否符合一般 SMT 场景，但不能据此证明已经适配某一家
具体工厂。

当前项目没有可直接接入的实际产线设备和真实设备数据。项目采用可控的仿真验证路线：由 Python
脚本持续模拟 SPI、AOI、FCT 和产线工控机产生文件，再由独立采集程序发现这些文件、建立本地
上传任务并调用 DataStream 接口。该路线符合当前开发条件，能够验证采集、断点续传、归档和查询
的完整链路。本文所称“真实业务差距”，是指仿真模型与典型现场行为之间仍需保持的对应关系，
不表示当前阶段必须取得真实设备数据后才能继续开发。

## 3. 总体判断

当前方案作为三线仿真、单服务实例、本地文件系统部署的首期试点是合理的，核心技术路线没有明显
方向性错误。MySQL、Redis 和文件系统的职责划分清楚，分片续传、完整性校验、原子归档和崩溃
窗口恢复也符合工业文件接收服务的基本要求。

如果 Python 模拟器、采集程序和 DataStream 服务按计划共同实现，项目可以形成一条完整的仿真
采集链路，而不只是文件接收服务。当前主要缺口是采集程序尚未落地，模拟器也需要从“生成已
完成文件”扩展为“模拟文件产生过程和异常过程”。单实例、本地磁盘、单一 Operator Token 和
缺少容量治理等约束，仍意味着该方案适合项目验证和单线试点模型，不能据此声称已经适配所有
真实产线。

### 3.1 已确认的优化决定

2026-07-11 已根据学习与面试目标确认以下决定：

- 目标仿真规模为三条 SMT 线，每线一台 IPC 及 SPI/AOI/FCT；
- 当前设备 HMAC v1 保持不变，IPC 保存本线设备密钥，服务端新增采集器与设备绑定校验；
- Redis 上传会话丢失后由采集端创建新会话重传，不把每个分片同步写入 MySQL；
- 第五期生成边界明确的 C++ 参考采集程序和增强文件生命周期模拟器；
- 原五期拆为六期，第六期负责部署、压力、资源检查和面试证据收口；
- 高可用、复杂 RBAC、同步 MES、正式归档自动删除和文件下载不扩展为首期实现。

拓扑、容量假设、文件封口和本地队列规则分别固化在 `模拟业务基线.md` 与
`采集端与文件生命周期契约.md`。本节之后仍保留完整差距，作为设计取舍和生产演进材料；已确认
事项不再作为第三期的阻塞问题。

## 4. 当前设计中合理的部分

### 4.1 项目边界清晰

DataStream 只负责设备认证、数据接收、文件归档、元数据管理和历史追溯；日志全文检索由另一个
项目负责。先保证原始文件可靠归档，再进行解析和索引，符合数据平台的正常建设顺序。

### 4.2 存储职责划分合理

- MySQL 保存产线、工位、设备和归档元数据；
- Redis 保存心跳、防重放、上传会话、Bitmap 和分片摘要等短生命周期状态；
- 文件正文保存在文件系统，不写入 MySQL；
- 临时目录和归档目录要求位于同一文件系统，以支持原子 `rename`。

这种划分适合 JSON、CSV、PNG、ZIP、PDF 和运行日志等大小差异明显的工业文件。

### 4.3 上传完整性和幂等思路合理

设计中包含文件预分配、偏移写入、乱序分片、断点续传、分片摘要、Bitmap、最终大小检查和
SHA-256 校验。完成接口通过确定的归档路径和 MySQL 唯一约束避免生成重复文件和重复记录，
符合弱网络和客户端重试场景。

### 4.4 对崩溃窗口有明确认识

设计已经考虑以下真实故障：

- 临时文件创建后 Redis 会话写入失败；
- `pwrite` 成功但 Bitmap 尚未置位；
- 分片完成但客户端未收到响应；
- 进入 VERIFYING 后进程退出；
- `rename` 成功但 MySQL 尚未入库；
- MySQL 已入库但 Redis 状态尚未更新。

这些恢复场景有明确业务依据，不属于为了展示架构而增加的通用补偿框架。

### 4.5 安全边界总体正确

设备 HMAC、正文 SHA-256、时间窗口、请求标识防重放、常量时间比较和敏感值环境注入均符合
内网设备接入服务的基本安全要求。基础设施故障、设备状态错误和签名错误被分别处理，也避免了
用错误的认证结果掩盖 MySQL 或 Redis 故障。

### 4.6 使用持续模拟数据驱动完整链路是合理的

在没有实际设备和真实产线数据的条件下，通过 Python 脚本持续模拟多类设备产出，再由采集程序
按照真实目录采集方式处理，是合理且可重复的开发方法。相比直接由测试代码调用上传接口，该方案
可以同时验证文件发现、稳定判定、本地排队、网络上传、断点续传、服务端归档和历史查询。

模拟器应提供固定随机种子、业务时间推进和清单期望值，使测试结果可复现；采集程序不能依赖
模拟器的内部状态，只能通过约定目录和文件行为观察数据，这样才能保持设备与采集器之间的真实
边界。

## 5. 已存在的问题与改进思路

### 5.1 项目经历描述与当前仓库实现状态不一致

#### 问题

`SMT产线项目.md` 使用完成时描述了上传会话、分片上传、断点续传、窗口化 `mmap` 校验、原子
归档、历史查询和定期清理。但是当前代码实际只注册了以下接口：

- `GET /health/live`；
- `GET /health/ready`；
- `POST /api/v1/devices/heartbeat`。

上传、归档、查询和清理目前仍主要存在于设计文档和后续分期计划中。第二期真实 MySQL/Redis、
HTTP 心跳端到端验收已完成，但尚不能证明后续上传和归档能力。

#### 影响

如果把当前仓库作为项目经历的证明材料，会出现“简历声明已经完成，但代码和验证证据尚未覆盖”
的问题。这会影响项目可信度，也会让后续开发人员误判当前基线。

#### 改进思路

- 保持分期状态、README、开发报告和实际路由一致；
- 未实现的能力明确标记为目标契约，不使用“已经实现”表述；
- 每一期完成后保留构建、测试、端到端和故障注入证据；
- 六期全部达到发布门槛后，再将当前仓库描述为完整复现了项目经历。

### 5.2 配置项与当前实现存在阶段性脱节

#### 问题

配置中已经包含上传、清理、查询和优雅停止参数，但其中一部分尚无对应业务实现。例如
`graceful_shutdown_timeout_seconds` 已加载，当前停止流程中尚未体现对在途文件任务的有界等待。

#### 影响

配置存在但不生效容易给部署人员造成错误预期，也会使配置文档和实际行为逐渐偏离。

#### 改进思路

- 配置项只在对应阶段实现时进入稳定对外契约；
- 对已经提前锁定但尚未生效的配置，在阶段文档中明确标注；
- 第三、四期引入文件任务后，落实停止接收请求、等待在途任务、超时退出的完整流程；
- 为关键配置增加行为级测试，而不仅验证配置能够被解析。

## 6. 与实际 SMT 业务的主要差距及改进方向

### 6.1 仿真边缘采集闭环尚需实现和验证

#### 业务差距

项目已经明确计划由 Python 脚本持续模拟产线设备产生数据，再由采集程序采集并上传。这一总体
方向合理，但当前模拟数据主要表现为已经封口的标准文件，尚不足以验证采集程序面对文件生成
过程时的行为。典型现场文件可能持续追加、先写临时名再改名、重复导出、覆盖旧文件或短暂保持
打开状态。

因此，当前缺口不是“没有真实设备数据”，而是模拟器和采集程序之间的文件生命周期契约尚未
完整定义，暂时不能验证以下关键行为：

- 如何判断文件已经写完；
- 服务重启后如何恢复未上传队列；
- 网络中断时能够缓存多少数据；
- 同一文件被重复发现时如何去重；
- 工控机磁盘不足时如何限流或告警；
- 不同厂商目录和文件格式如何映射为统一元数据。

#### 改进思路

应把“Python 模拟器 -> 模拟设备目录 -> 采集程序 -> DataStream 服务”作为正式的端到端验证
链路，并补充边缘采集契约，至少定义：

1. 文件稳定判定规则，例如临时文件改名、文件大小连续多次不变或厂商完成标识；
2. 本地持久化上传队列，不能只依赖进程内存；
3. 文件发现、上传中、上传成功、永久失败等本地状态；
4. 重启恢复、网络断开、服务端不可用和本地磁盘满的处理；
5. 文件唯一标识和重复发现规则；
6. 设备目录到 `device_id`、`station_id`、工单和产品 SN 的映射来源。

模拟器应按可配置场景产生以下行为：

- 一次写完后原子改名的结果文件；
- 分多次追加并在结束后封口的运行日志；
- 写入过程中保持打开、延迟关闭的文件；
- 相同名称覆盖、重复导出和相同内容重复出现；
- 损坏、截断、零字节和摘要不匹配文件；
- 不同设备并发产出以及短时间突发产出。

每个场景应生成独立清单，记录期望被采集的文件、最终大小、SHA-256、业务元数据，以及预期被
忽略或判定失败的文件。采集程序只读取模拟设备目录，不能直接读取清单来决定上传行为；清单只
用于测试结束后的结果核对。

完成上述闭环后，可以证明系统在已定义的仿真业务模型下具备完整采集能力。由于没有实际设备，
结论应限定为“覆盖典型 SMT 文件行为的仿真验证”，而不是“已经兼容特定厂商设备”。

### 6.2 请求主体与设备身份模型需要现场确认

#### 业务差距

当前模型要求请求使用实际数据源设备的 HMAC 密钥签名，但 HTTP 请求通常由产线工控机上的采集
程序发起。许多 SPI、AOI 和 FCT 设备本身并不具备调用自定义 HTTP 接口或管理 HMAC 密钥的
能力。一台工控机采集多台设备时，需要保存多台设备的密钥，扩大了密钥泄露范围。

#### 改进思路

在现场确认责任主体后，从以下模型中选择，而不是默认设备一定能够直接签名：

- 设备身份模型：采集器受控且可以安全保存每台设备的密钥，继续使用当前方案；
- 采集器身份模型：为工控机分配独立凭据，并维护其被授权采集的设备列表；
- 双重绑定模型：采集器认证后提交数据源设备编号，服务端校验采集器与设备、工位的绑定关系。

同时应补充密钥初始化、轮换、停用、泄露处置和审计流程。首期不一定需要完整 PKI，但不能只有
静态密钥而没有生命周期管理。

当前仿真已选择设备身份模型：每条线 IPC 保存本线三台设备密钥，服务端校验
`collector_id-device_id` 绑定。其余模型保留为未来现场校准的替代方案，不在 v1 并行实现。

### 6.3 设备时钟偏差可能影响认证可用性

#### 业务差距

设备或工控机的系统时间在现场可能不准确。固定 300 秒时间窗口能够限制重放，但设备断电、
主板电池失效或 NTP 不可用时，合法请求可能持续被拒绝。

#### 改进思路

- 明确时间戳来自设备还是工控机；
- 工控机部署时配置可靠 NTP，并监控时钟偏差；
- 在认证失败日志中记录不敏感的时间偏差值，便于排障；
- 不通过扩大到过长时间窗口来掩盖时钟问题；
- 如现场无法保证时钟，再评估一次性挑战值等替代协议，不能默认增加复杂度。

### 6.4 元数据模型不足以覆盖完整质量追溯

#### 业务差距

当前核心字段包含产线、工位、设备、工控机、工单、产品 SN、文件类型、结果和时间，能够满足
基本文件检索，但真实 SMT 追溯通常还可能依赖：

- 工厂、车间和班次；
- 产品型号、PCB 板号、Panel SN、单板 SN 和板面；
- 工艺配方或程序版本；
- 生产批次、物料批次和客户批次；
- 检测程序版本、设备型号和固件版本快照；
- 缺陷代码、缺陷位置和复判结果；
- MES 工单或追溯系统中的外部业务标识。

现有“一台设备固定归属一个工位、一个工位固定归属一条产线”的模型适合首期，但设备搬迁、
借机生产和工位调整后，需要保证历史归档仍反映文件产生时的关系，而不是只依赖当前主数据。

#### 改进思路

- 使用真实 MES、设备文件和现场查询案例确定必需字段，不一次性加入所有可能字段；
- 归档记录保存文件产生时的产线、工位、设备和版本快照；
- 明确工单、SN、批次等标识的字符集和长度，避免只允许理想化 ASCII 格式；
- 对高频组合查询建立联合索引，低频扩展字段可在需求明确后新增迁移；
- 原始文件内容解析仍由后续日志或数据解析项目负责，不在上传边界重复推断。

### 6.5 Redis 会话丢失后的恢复能力不足

#### 业务差距

上传会话、分片 Bitmap 和摘要计划主要保存在 Redis。若 Redis 重启未恢复数据、发生淘汰或键被
误删，临时文件可能仍在，但服务端已经无法继续识别已完成分片和会话元数据，只能让客户端重新
上传。

#### 改进思路

- 生产 Redis 明确持久化方式、备份、内存上限和 `maxmemory-policy`；
- 上传状态不得与普通可淘汰缓存混用不受控的淘汰策略；
- 为临时文件保留可验证的最小恢复信息，或明确接受 Redis 丢失后重新创建会话的业务代价；
- 故障恢复测试覆盖 Redis 重启、数据缺失和会话键部分存在；
- 不在没有明确恢复需求时把每个分片同步写入 MySQL，以免引入高频事务。

当前仿真已确认 Redis 使用持久化与 `noeviction`；会话仍丢失时由采集端创建新会话重新上传，
旧临时文件交由严格清理规则处理。该业务代价必须在第五、六期故障注入中验证。

### 6.6 缺少磁盘容量、并发和反压治理

#### 业务差距

单文件上限为 2 GiB，并计划使用 `posix_fallocate` 预分配空间。如果多个设备同时创建大文件
会话，临时目录可能迅速耗尽。当前配置没有体现：

- 全局在途文件大小上限；
- 单设备或单采集器并发会话数；
- 临时目录最低剩余空间；
- 每日数据量和归档增长速度；
- 上传速率、请求并发和文件描述符预算；
- 磁盘不足时的明确拒绝与告警策略。

#### 改进思路

- 根据实际产线节拍、文件大小分布和保留周期建立容量模型；
- 创建会话前检查可用空间和预留安全水位；
- 设置全局、设备和采集器维度的在途会话限制；
- 磁盘压力下拒绝新会话，不中断已接近完成的合法会话；
- 对临时目录占用、归档增长、失败会话和清理积压建立监控；
- 使用真实峰值和大文件场景压测，而不只验证少量模拟文件。

### 6.7 单实例和本地文件系统存在单点故障

#### 业务差距

当前明确采用单服务实例和本地文件系统。这对三线仿真的中小厂试点是合理的，但服务进程、服务器和磁盘
均可能成为单点。如果归档文件承担质量追溯、客户投诉或审计证据，磁盘损坏可能造成不可恢复的
业务损失。

#### 改进思路

先根据业务可用性目标决定是否需要高可用，不为了展示技术栈直接改成分布式系统。即使保持单机
方案，也应明确：

- RAID、UPS、SMART 和磁盘空间监控；
- MySQL 与归档目录的一致性备份流程；
- 异机或离线备份周期；
- 恢复时间目标和允许丢失的数据窗口；
- 服务不可用期间采集端的本地缓存容量；
- 定期执行恢复演练和 SHA-256 抽样校验。

只有业务明确要求跨节点连续服务时，再评估共享存储、对象存储或主备切换。

### 6.8 缺少归档保留和生命周期规则

#### 业务差距

当前清理设计主要针对临时文件，不删除正式归档，这是安全的。但项目没有明确正式文件保存
多久、何时转移到低成本存储、哪些数据必须长期保留，以及删除操作由谁审批。

#### 改进思路

- 按文件类型、产品、客户和质量制度确定保留周期；
- 区分临时数据清理与正式归档生命周期管理；
- 正式归档删除必须具备授权、审计和可恢复机制；
- 定期校验数据库元数据与文件实际存在性、大小和摘要；
- 在保留规则确定前继续坚持“不自动删除正式归档”。

### 6.9 单一 Operator Token 不满足多人生产运维

#### 业务差距

单一部署级 Token 适合首期内部联调，但无法区分谁进行了查询，也无法限制不同人员可访问的
产线、设备和文件类型。Token 泄露后只能整体更换。

#### 改进思路

- 首期若只有受控运维脚本，可继续使用单 Token，但要限定网络范围并记录调用来源；
- 多人使用前补充用户身份、最小权限、Token 生命周期和查询审计；
- 权限模型至少区分设备写入、归档查询和系统运维；
- 不在用户和权限需求尚未明确时提前建设复杂 RBAC 平台。

### 6.10 只查询元数据，业务取回闭环不足

#### 业务差距

当前终板计划只提供归档列表和详情，不提供文件正文下载。这可以减少首期安全范围，但质量工程师
最终需要查看检测结果、NG 图片和测试报告。只返回相对路径会迫使人员登录服务器或通过未受控的
共享目录取文件。

#### 改进思路

- 明确首期文件取回由受控运维流程、只读共享目录还是后续下载接口承担；
- 如果增加下载接口，需要独立评审权限、审计、路径校验、Range、限速和大文件传输；
- 不能把数据库中的 `relative_path` 直接拼成公开下载 URL；
- 在文件取回方式确定前，文档中明确平台当前只完成“定位”，未完成面向最终用户的“获取”。

### 6.11 与 MES、工单和设备主数据的集成边界不清晰

#### 业务差距

当前产线、工位和设备通过开发种子预置，工单和产品 SN 由采集端直接提交。真实环境中这些数据
通常来自 MES、设备程序或人工配置。若服务端只相信采集端，可能产生错误工单、错误 SN 或设备
与工位关系不一致的数据。

#### 改进思路

- 明确产线、工位、设备和工单的权威数据源；
- 决定哪些字段只做格式校验，哪些字段需要与 MES 或本地主数据核对；
- 明确 MES 不可用时是拒绝上传、允许延迟校验，还是只保存原始数据；
- 不在没有幂等性和可用性要求时默认同步调用 MES；
- 对外部系统标识保留原值，避免采集端和服务端各自重新解释。

### 6.12 模拟数据需要覆盖文件生命周期而不只是文件类型

#### 业务差距

当前模拟数据覆盖五类文件，是较好的接口联调起点。当前没有实际设备数据，因此模拟器本身就是
项目的主要业务输入源。其不足不应表述为“没有使用真实样本”，而应表述为当前场景规模和文件
行为覆盖仍然有限，例如尚不能充分验证：

- 厂商专有编码和非标准 CSV；
- 超长文件名、中文工单、特殊 SN 和错误时间格式；
- 大图片、大 ZIP、长时间追加日志和零字节异常文件；
- 同一秒内大量板件集中上传；
- 网络抖动、重复请求、半包和长时间断线；
- 设备升级后目录或字段格式发生变化。

#### 改进思路

- 为 SPI、AOI、FCT 和工控机分别建立可配置的设备行为模型；
- 建立正常、边界、损坏、格式变化和软件升级五类模拟场景；
- 支持持续运行、可调节拍、并发设备数、突发倍率和目标数据量；
- 模拟分段写入、临时名改名、追加、覆盖、重复、延迟关闭和异常中断；
- 固定随机种子时保证文件内容、时间线和期望清单可复现；
- 端到端测试核对文件大小、SHA-256、元数据和可追溯查询结果；
- 将模拟参数和假设写入测试报告，明确其代表的产线规模和业务边界；
- 将来如果能够取得真实脱敏样本，再用于校准模拟器，而不是作为当前开发的前置条件。

### 6.13 可观测性和现场排障信息仍不完整

#### 业务差距

当前设计强调结构化日志和敏感信息保护，但实际生产还需要快速回答：哪台设备停止上报、哪个
上传会话长期无进展、临时目录为何增长、Redis 或 MySQL 调用是否变慢、某类文件为什么持续
校验失败。

#### 改进思路

- 日志统一携带请求标识、设备、采集器、上传会话、稳定错误码和耗时；
- 建立在线设备数、心跳延迟、在途会话数、分片失败、摘要失败、归档成功率和目录占用指标；
- 对磁盘水位、清理积压、连续心跳丢失和基础设施错误设置告警；
- 保持日志不输出密钥、签名、Token、数据库密码和文件正文；
- 在没有监控平台时先提供可被采集的结构化日志和健康数据，不必提前引入复杂系统。

## 7. 建议的改进优先级

### 7.1 P0：进入上传开发前必须确认

1. 明确采集端责任边界和文件稳定判定；
2. 明确设备身份、采集器身份及密钥保存方式；
3. 定义 Python 模拟器与采集程序之间的目录、文件生命周期和期望清单契约；
4. 为日产量、峰值、最大文件、保留周期和可接受停机时间设定明确的模拟基线；
5. 明确 Redis 持久化、淘汰策略和会话丢失后的业务处理；
6. 建立仓库实现状态与项目经历声明之间的一致性规则。

状态：已通过 `模拟业务基线.md`、`采集端与文件生命周期契约.md`、六期实施计划和阶段开发报告
完成设计确认。这些问题会影响外部契约、数据安全和容量设计，不能留到第六期；第三期开始时
需将已确认规则转换为迁移、配置和测试契约。

### 7.2 P1：第三、四期实现时完成

1. 创建会话前的磁盘水位和并发配额；
2. 上传会话、分片和完成流程的故障注入测试；
3. 归档记录保存必要的生产关系快照；
4. MySQL、Redis、文件系统之间的确定性恢复；
5. 临时文件清理的严格名称、状态和时间条件；
6. 持续模拟文件的端到端发现、上传、大小、摘要和查询回验；
7. 明确文件取回方式，避免相对路径成为非受控下载入口。

### 7.3 P2：生产部署前完成

1. RAID、UPS、备份、恢复和归档校验方案；
2. Nginx TLS、请求大小、超时和安全响应头；
3. Redis/MySQL 生产账号、网络边界和最小权限；
4. Operator Token 的轮换、审计或正式用户权限方案；
5. 磁盘、上传、心跳和依赖调用的监控告警；
6. 按既定模拟峰值执行压力测试、长稳测试和服务重启演练；
7. 明确正式归档保留、迁移和删除规则。

## 8. 建议补充的业务输入清单

当前没有实际产线可供调研时，应先把以下信息定义为有依据、可说明的模拟假设，并固化为模拟器
配置和测试场景。未来具备现场条件后，再由业务、设备和运维人员使用实际参数校准：

| 类别 | 需要确认的内容 |
|---|---|
| 设备 | 厂商、型号、软件版本、输出目录、文件完成标志、是否锁文件 |
| 数据 | 文件类型、格式、编码、平均/最大大小、是否追加或覆盖 |
| 节拍 | 每小时板数、单板文件数、峰值集中度、班次和停线窗口 |
| 网络 | 工控机到服务端链路、断线频率、允许离线时长 |
| 标识 | 工单、SN、Panel、批次、配方及其权威来源和字符规则 |
| 安全 | 谁持有凭据、如何初始化、轮换、停用和审计 |
| 存储 | 日增量、保留年限、备份介质、恢复时间和允许丢失窗口 |
| 查询 | 谁查询、常用条件、是否需要下载、期望响应时间 |
| 运维 | 可接受停机时间、告警渠道、值班和故障处置流程 |
| 合规 | 客户或质量体系对留存、删除、审计和数据隔离的要求 |

在没有现场输入时，配置中的 2 GiB、24 小时、31 天和单实例等参数只能视为项目模拟基线，不能
视为已经得到真实产线验证的生产指标。模拟基线本身必须明确，不能在测试过程中任意变化。

## 9. 业务验收建议

六期完成并不自动等于符合实际业务。生产验收至少应增加以下场景：

1. 模拟三条线 SPI、AOI、FCT 文件从分段产生、封口、发现到归档的完整链路；
2. 工控机断网、重启和本地缓存达到上限后的恢复；
3. 同一文件重复发现、同一分片重复上传和完成接口重复调用；
4. Redis 重启、MySQL 短暂不可用、服务进程退出和服务器重启；
5. 临时目录空间不足、正式归档目录接近水位和清理任务积压；
6. 文件被篡改、截断、覆盖或摘要不一致；
7. 设备换线、停用、工单切换和软件版本升级；
8. 使用既定的模拟峰值、突发倍率和目标数据量运行长稳测试；
9. 从备份恢复 MySQL 与归档目录，并通过 SHA-256 校验；
10. 质量或运维人员能够按真实追溯条件定位并取得所需文件。

只有业务链路、故障恢复、容量和运维流程均通过，才能证明该平台完整实现了所定义的 SMT 仿真
业务。由于当前没有实际设备，验收结论必须注明模拟的设备数量、文件类型、节拍、异常比例和
持续时间，不能直接扩展为对任意真实产线的适配结论。

## 10. 结论

当前方案的服务端技术路线总体合理，分片上传、完整性校验、原子归档、元数据索引和失败处理
方向符合 SMT 检测文件归档业务。主要风险集中在采集端闭环、身份主体、模拟业务模型、容量反压、
Redis 状态可靠性、单机存储、权限审计和生产运维，而不是基础框架选型。

后续开发应把已确认的业务输入转换为第三、四期契约。对于当前三线仿真试点，可以保留简单
架构；对于质量追溯和长期审计用途，则必须在生产发布前补齐备份恢复、
文件取回、权限审计和容量治理。当前阶段应先完成“持续模拟设备产出、采集程序可靠采集、服务端
归档查询”的可复现端到端验收；未来获得实际设备条件后，再补充厂商兼容性校准和现场验收。

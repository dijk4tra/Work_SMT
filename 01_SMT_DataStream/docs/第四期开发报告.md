# 第四期开发报告

代码与验收完成日期：2026-07-12。

## 1. 完成范围

第四期完成上传状态机的最后一段：分片完整后进入 `VERIFYING`，核对临时文件大小和整文件
SHA-256，原子移动到正式目录，写入 MySQL 唯一归档记录并把 Redis 会话改为 `ARCHIVED`。
同时实现 Operator 历史列表、详情、稳定游标，以及严格的旧临时文件清理。本期没有文件下载
路由、自动重试、全表缓存或通用补偿平台。

新增接口：

- `POST /api/v1/uploads/{upload_id}/complete`；
- `GET /api/v1/archives`；
- `GET /api/v1/archives/{archive_id}`。

## 2. 完整性校验与原子归档

完成请求首先通过 Redis Lua 检查 `BITCOUNT == chunk_count`，再把会话从 `UPLOADING` 原子改为
`VERIFYING`。首次占有完成权时同时保存服务端归档毫秒时间和确定性相对路径；重试继续使用同一
时间和路径，不会产生第二份正式文件。

文件任务在 `datastream-complete-<upload_id>` 命名队列中执行：

1. `stat/fstat` 核对普通文件和精确字节数；
2. 按配置窗口进行页对齐 `mmap`，用 OpenSSL EVP 增量计算 SHA-256；
3. 创建固定的年/月/日、产线、工位、设备和文件类型目录；
4. 使用同文件系统 `rename` 把 `.part` 移到正式路径；
5. 插入 `archive_file`，成功后写回 Redis 并主动释放三类会话配额。

大小或摘要不符时写入 `FAILED/FILE_INTEGRITY_MISMATCH`，HTTP 返回 422。文件系统错误、MySQL
错误和 Redis 错误分别返回 500、503、503，不用成功或默认值掩盖失败。

## 3. 两个真实崩溃窗口

文件系统与 MySQL 没有分布式事务，本期只恢复确认存在的两个窗口：

| 窗口 | 重试行为 |
|---|---|
| `rename` 成功、MySQL 插入前失败 | 会话保留同一路径；重试校验已存在正式文件并补写唯一记录 |
| MySQL 插入成功、Redis 更新前失败 | 先按唯一 `upload_id` 读取归档事实，再修复 Redis 为 `ARCHIVED` |

端到端测试通过删除刚插入的测试记录并把会话恢复为 `VERIFYING`，验证第一种补写；随后保留
MySQL 记录并再次把 Redis 改为 `VERIFYING`，验证第二种修复。两次恢复均返回稳定的唯一记录，
正式路径没有复制文件。并发完成在 Lua 已返回 `ARCHIVED` 时也进入唯一记录读取，而不是错误地
返回状态冲突。

## 4. 历史查询与 UTC 游标

列表严格接受设备、工位、工单、产品 SN、文件类型、结果、UTC 时间范围、页大小和游标。
没有精确工单或产品 SN 时必须提供不超过 31 天的时间范围。排序固定为
`archived_at DESC, archive_id DESC`。

游标是 8 字节 UTC 毫秒加 8 字节无符号归档主键的大端布局，再编码成无填充 Base64URL。
解码严格检查 22 字符、字符集、字段范围和非零主键。测试使用页大小 1 对两条记录翻页，确认
没有重复或跳项。

测试过程中发现 MySQL 连接会话使用本机 `+08:00` 时，`UNIX_TIMESTAMP(DATETIME)` 会把按 UTC
写入的无时区字段偏移 8 小时，使下一页条件过早。生产代码已改为直接读取 `DATETIME(3)` 文本并
按 UTC 解析，游标不再依赖连接会话时区。

本机 `EXPLAIN` 结果显示：设备时间查询使用 `idx_archive_device_time`，工位时间查询使用
`idx_archive_station_time`，精确工单和产品 SN 查询使用 `idx_archive_trace`；三者均按联合索引
反向扫描完成既定降序，没有为每个可选条件补单列索引。

## 5. 严格清理

清理服务启动后立即扫描一次，随后按 `cleanup.interval_seconds` 周期运行。候选必须同时满足：

- 位于配置的临时根目录，且是普通文件；
- 文件名严格为小写标准 UUID 加 `.part`；
- mtime 早于诊断保留截止时间；
- Redis 会话不存在，或状态明确为 `FAILED/ARCHIVED`。

Redis 查询失败或会话仍为活动态时保留文件；未知文件只告警不删除；正式归档目录从不扫描。
结束态文件删除后同步删除会话、Bitmap 和分片摘要。单次失败记录错误并等待下个调度周期，不在
服务端循环重试。

# SMT 检测工位数据采集与归档管理平台开发思路

## 1. 文档定位

本文依据根目录 `SMT产线项目.md` 中的项目描述，说明首期服务端的业务边界、模块职责、
关键流程和落地顺序。文中设备、工位和参数来自当前模拟场景或首期工程约定，不代表实际
生产环境已经确认的容量指标。

## 2. 要解决的问题

SPI、AOI、FCT 等设备通常由不同厂商提供，输出目录和文件格式各不相同；工控机本地文件
容易分散、覆盖或随设备下线而丢失。平台需要提供统一上传通道，并保留足够的业务元数据，
让工程人员能够按设备、工位、工单、产品 SN 和时间追溯原始文件。

首期目标是：

- 只允许已登记且启用的设备提交数据；
- 对请求做完整性认证并限制重放；
- 支持大文件乱序分片和断线后补传；
- 校验最终文件大小和 SHA-256 后再进入正式归档；
- 保存可查询元数据，而不是把大文件正文写入 MySQL；
- 维护设备在线状态，并清理确定已过期的临时数据。

首期不实现：

- 设备协议驱动和 PLC 控制；第五期只实现面向约定目录的参考采集程序；
- 原始文件内容解析、统计报表、告警推送和 Web 前端；
- 日志全文检索、倒排索引、BM25 和 SLRU 缓存；这些属于 `02_SMT_LogTrace`；
- 跨机房复制、对象存储、分布式事务和多租户计费。

## 3. 业务对象

| 对象 | 职责 | 首期关键标识 |
|---|---|---|
| 产线 | 组织工位 | `line_id` |
| 工位 | 表示 SPI、AOI、FCT 等检测或测试环节 | `station_id` |
| 设备 | 产生检测文件并持有签名密钥 | `device_id` |
| 采集工控机 | 发现设备文件并调用平台接口 | `collector_id` |
| 上传会话 | 约束一个文件的一次分片上传 | `upload_id` |
| 归档记录 | 文件归档后的业务元数据 | `archive_id` |

一个设备固定归属一个工位，工位固定归属一条产线。总体仿真三条线，每线一台 IPC 采集本线
SPI、AOI、FCT。请求使用实际数据源设备身份签名，`collector_id` 必须与设备存在启用绑定，
不能只作为未经校验的审计字符串。完整规模和编号见 `模拟业务基线.md`。

## 4. 总体架构

```text
检测设备目录 ──> 产线工控机采集程序 ──HTTP/HMAC──> Wfrest 接入服务
                                                      │
                         ┌────────────────────────────┼────────────────────┐
                         │                            │                    │
                      MySQL                        Redis             本地文件系统
                  设备/工位/归档元数据       心跳/防重放/上传会话      临时分片/正式归档
```

首期采用单个服务进程，内部按职责拆分认证、设备、上传、归档查询和清理模块。没有独立部署或
扩缩容需求的模块不拆成微服务。Wfrest 负责 HTTP 路由，Workflow 组织数据库、Redis 与文件
任务；耗时的文件写入和哈希计算不能占用网络回调线程。

建议的后续源码布局为：

```text
include/datastream/
├── api/          # 请求解析、响应模型和路由声明
├── auth/         # HMAC 校验与防重放
├── device/       # 设备查询和心跳
├── upload/       # 会话、分片和完成流程
├── archive/      # 归档元数据与历史查询
├── storage/      # MySQL、Redis、文件存储的具体访问
└── common/       # 仅放确有多个模块共用的错误码、摘要等基础类型
```

不要在编码前创建空接口层、工厂或仓储基类。只有出现两个真实实现或测试替换需求时，再提取
对应抽象。

## 5. 请求认证

### 5.1 请求头

设备请求携带：

- `X-Device-Id`：设备编号；
- `X-Timestamp`：Unix 秒；
- `X-Request-Id`：设备侧生成的唯一请求标识；
- `X-Content-SHA256`：请求体 SHA-256 十六进制小写值；
- `X-Signature`：HMAC-SHA256 十六进制小写值。

首期签名版本固定为 `v1`，规范串为：

```text
v1\n
<HTTP_METHOD>\n
<PATH>\n
<DEVICE_ID>\n
<TIMESTAMP>\n
<REQUEST_ID>\n
<CONTENT_SHA256>
```

上传编号和分片编号位于路径中，因此自然进入签名；不再增加一套分片专用签名格式。空请求体
也使用空字节串的 SHA-256，不使用空字段代替。

### 5.2 校验顺序

1. 校验必要请求头的格式和时间窗口；
2. 查询设备是否存在、启用并获取 HMAC 密钥；
3. 计算请求体摘要并使用常量时间比较；
4. 计算并比较 HMAC-SHA256；
5. 通过 Redis `SET key value NX EX` 登记请求标识，已存在则判定为重放。

只有签名通过后才登记防重放键，避免无效请求消耗 Redis 空间。服务端不得在日志中输出设备
密钥、完整签名或请求体。设备查询失败、Redis 失败和签名失败分别返回明确错误，不把基础
设施故障伪装成“签名错误”。

## 6. 设备心跳

工控机定期上报设备软件版本、当前工单、运行状态和可选运行指标。Redis 保存短生命周期的
实时状态，MySQL 保存设备静态资料及最近一次心跳时间，用于历史管理。

在线状态由“最后心跳时间是否落在约定窗口内”计算，不额外维护容易不一致的在线布尔值。
首期心跳请求不承担设备自动注册；未知或停用设备直接拒绝。

## 7. 分片上传与归档

### 7.1 创建会话

客户端提交设备、工位、工控机、工单、产品 SN、文件类型、原文件名、总大小、整文件
SHA-256 和期望分片大小。服务端完成以下工作：

1. 验证设备与工位归属、文件类型、大小、摘要和业务字段；
2. 计算 `chunk_count = ceil(file_size / chunk_size)`；
3. 生成 `upload_id`，在临时目录创建文件并使用 `posix_fallocate` 预分配空间；
4. 在 Redis 写入会话字段、状态和空分片位图，并设置会话过期时间。

临时文件和正式归档必须位于同一文件系统，才能依赖 `rename` 的原子性。创建失败时不返回
可用的 `upload_id`。创建前还要校验采集器与设备绑定、临时目录安全水位、全局和单设备在途
会话配额；达到水位或配额时拒绝新会话，不中断已进入上传流程的合法会话。

### 7.2 上传分片

分片编号从 0 开始，写入偏移为 `chunk_no * chunk_size`。除最后一片外，分片长度必须等于
会话约定值；最后一片长度由总文件大小确定。处理流程为：

1. 查询并验证上传会话仍处于可写状态；
2. 校验分片编号、长度和认证头中的 `X-Content-SHA256`；
3. 若该位已经完成，只在分片摘要一致时返回幂等成功，否则返回冲突；
4. 使用 `pwrite` 循环写满目标区域；
5. 保存分片摘要并置位 Redis Bitmap。

如果进程在写入后、置位前退出，客户端重传会覆盖同一偏移，结果仍然正确。首期不增加服务端
自动重试；磁盘写入失败直接返回，客户端根据明确的幂等契约决定是否重试。

### 7.3 查询进度

进度接口返回会话状态、分片总数和已完成分片编号。客户端断线恢复后只补传缺失编号。
Redis Bitmap 是完成状态的唯一来源，不同时维护另一份完成计数真值；需要数量时即时统计。

### 7.4 完成上传

完成接口先将会话从 `UPLOADING` 原子切换到 `VERIFYING`，阻止继续写分片，然后：

1. 确认位图中的全部分片已完成；
2. 使用 `stat` 核对临时文件大小；
3. 按配置窗口分段 `mmap`，增量计算整文件 SHA-256；
4. 与会话中的文件大小和摘要比较；
5. 使用 `rename` 移动到正式归档路径；
6. 写入 MySQL 归档记录，`upload_id` 唯一约束保证完成请求幂等；
7. 将 Redis 会话标记为 `ARCHIVED` 并缩短保留时间。

文件系统与 MySQL 之间没有分布式事务。若 `rename` 成功而元数据写入失败，同一完成请求重试时
根据确定的最终路径和 `upload_id` 补写元数据；不复制第二份文件，也不生成新归档编号。
此处是实际崩溃窗口的必要恢复，不扩展成通用补偿框架。

建议状态为：

```text
UPLOADING ──完整分片──> VERIFYING ──校验及归档成功──> ARCHIVED
    │                       │
    └──会话超时──> EXPIRED  └──大小/摘要错误或归档失败──> FAILED
```

校验失败必须保留可诊断的失败原因。是否允许同一个失败会话重新上传由错误类型决定：大小或摘要
不一致需要新建会话；瞬时基础设施失败可对完成接口做幂等重试，但服务端不主动循环重试。

## 8. 归档路径与元数据

正式路径使用服务端生成的安全名称，不直接拼接客户端原文件名：

```text
<archive_root>/<YYYY>/<MM>/<DD>/<line_id>/<station_id>/<device_id>/
<file_type>/<upload_id><validated_extension>
```

数据库保存原文件名和相对路径。下载或回读时以配置的归档根目录加数据库相对路径定位文件，
任何客户端输入都不能直接成为文件系统路径。

归档记录至少包含：设备、工位、产线、采集工控机、工单、产品 SN、文件类型、原文件名、
相对路径、大小、SHA-256、设备产生时间和平台归档时间。正文和图片不写入 MySQL。

高频索引按已知查询建立：

- `(device_id, archived_at, archive_id)`；
- `(work_order, product_sn, archived_at, archive_id)`；
- `(station_id, archived_at, archive_id)`；
- `UNIQUE(upload_id)`；
- `UNIQUE(relative_path)`。

索引字段顺序以实际接口过滤条件为准；首期不为每个可选字段建立单列索引。

## 9. 历史查询

查询接口支持设备、工位、工单、产品 SN、文件类型、检测结果和归档时间范围的合理组合，
返回元数据摘要，不默认返回文件正文。分页大小设上限，稳定排序使用
`archived_at DESC, archive_id DESC`，避免相同时间记录顺序漂移。

首期不缓存历史查询。只有压测证明 MySQL 查询成为瓶颈并明确缓存一致性要求后，才考虑缓存。

## 10. 清理任务

定时任务只清理以下可判定对象：

- Redis 会话已过期且临时文件超过保留窗口；
- 状态为 `FAILED` 或 `EXPIRED` 且超过诊断保留窗口的临时文件；
- 已归档会话过期后的 Redis 位图、分片摘要和会话字段。

清理任务不删除正式归档文件，不根据一次数据库查询失败推断文件无主，也不扫描并修改业务
元数据。删除失败记录一次带路径和错误码的日志，留待下次调度，不在单次任务中无限重试。

## 11. 并发、资源和可观测性

- HTTP 回调只做轻量解析与任务编排；`pwrite`、`mmap` 哈希和大目录操作放入有界文件任务执行器。
- 不同上传会话允许并发；同一会话的分片按会话锁串行写入，完成状态切换后拒绝新写入。
- 文件描述符、映射区和数据库结果使用 RAII 管理，所有退出路径都释放资源。
- 日志记录请求标识、设备编号、上传编号、业务错误码和耗时；不打印密钥与文件正文。
- 关键指标为在线设备数、会话创建数、分片写入失败数、完整性校验失败数、归档成功数、
  临时目录占用和接口延迟。首期先保留结构化日志字段，不提前引入独立监控系统。

## 12. 代码生成与交付方式

本项目确定采用“分阶段生成、最终形成第一期生产终板”的方式，不采用一次性生成全部文件后
统一排错。分阶段只用于控制依赖顺序和验证范围，不代表交付阉割版，也不要求用户补写代码。
每一阶段的源码、SQL、配置、脚本、测试和文档均由 AI 完整生成，并在当前环境通过该阶段的
构建和测试门槛后再进入下一阶段。

正式代码共分六期：

1. 工程骨架、基础设施与健康检查；
2. 设备认证、请求防重放与心跳；
3. 上传会话、文件预分配、分片写入与断点续传；
4. 完整性校验、原子归档、元数据查询与过期清理；
5. C++ 参考采集程序、文件生命周期模拟和三线仿真闭环；
6. 生产部署、安全/资源/压力检查、学习证据与终板收口。

只有六期全部完成，才能标记为第一期生产终板 `v1.0.0`。各期的文件、接口、数据变更、测试
和退出条件见 `分期实施计划.md`。阶段中发现前置设计错误时在当前阶段修正，不把已知问题留给
最后一期。

## 13. 第一期开箱功能

终板服务必须同时具备：

- `GET /health/live` 进程存活检查；
- `GET /health/ready` MySQL、Redis 和归档目录就绪检查；
- 设备 HMAC-SHA256 认证、正文摘要、时间窗口和请求标识防重放；
- 设备心跳与在线状态查询依据；
- 上传会话创建、1 MiB 至 8 MiB 分片、乱序上传、重复分片幂等和缺片查询；
- `posix_fallocate` 预分配、`pwrite` 偏移写、窗口化 `mmap` SHA-256；
- 同文件系统 `rename` 原子归档和 MySQL 唯一约束幂等入库；
- 按设备、工位、工单、产品 SN、文件类型、结果和时间范围查询元数据；
- 上传会话、失败/过期临时文件和 Redis 状态清理；
- 三线 SPI/AOI/FCT 文件产生、稳定发现、本地排队、上传和重启恢复闭环；
- 结构化运行日志、优雅停止、数据库迁移、开发种子数据与生产部署样例；
- 单元、集成、端到端、静态检查、覆盖率、Valgrind 和基础压力验证。

第一期明确不提供文件正文下载接口。平台完成原始文件安全归档与元数据追溯，文件读取由同机
运维流程或后续经过权限模型评审的接口提供；不能把服务端相对路径当作可公开下载地址。

## 14. C++11 与客户端选型

MySQL 和 Redis 均通过 Workflow 的 `WFMySQLTask`、`WFRedisTask` 异步访问，避免网络访问阻塞
Wfrest 线程。SQL 字符串只由固定模板、严格类型字段和 `MySQLUtil::escape_string_quote`
处理后的字符串组成，禁止把外部字段直接拼接到 SQL。

本机 `redis++ 1.3.15` 按 C++17 模式构建，与项目 C++11 约束冲突，因此不作为项目依赖。
不为使用该库升级语言标准，也不再引入第二套 Redis 客户端。Workflow Redis 任务已完成实际
连接验证。

文件分片使用 Workflow 文件 I/O 任务或项目内受控文件任务执行器，不能在 HTTP 回调线程中
同步计算大文件哈希。OpenSSL 提供 HMAC/SHA-256，nlohmann/json 负责 JSON，spdlog 负责日志，
GoogleTest 负责测试。

## 15. 并发边界

第一期部署模型是单服务实例、单归档文件系统、一个 MySQL 和一个 Redis。Nginx 只代理到这
一个实例，不声称支持多节点同时写同一上传会话。该边界与本地文件归档模型一致，避免加入
未经需求支持的分布式锁和共享文件系统。

- 不同上传会话可以并发执行；
- 同一会话的不同分片允许乱序到达；
- 同一 `upload_id` 的状态改变和分片写入由会话级互斥串行化，防止完成请求与写请求交叉；
- 完成状态切换成功后不再接受任何新分片；
- 会话互斥只解决进程内真实并发，不包装成通用锁框架；
- 服务停止时先停止接受新请求，等待已进入文件写入/完成流程的任务在限定时间内结束，再释放
  数据库、Redis、日志和文件资源。

## 16. 崩溃一致性与恢复

文件系统、Redis 和 MySQL 之间不使用分布式事务。终板只处理下表中的实际崩溃窗口：

| 中断位置 | 可观察状态 | 恢复规则 |
|---|---|---|
| 临时文件创建后、Redis 会话写入前 | 无会话的旧临时文件 | 当前请求尽量删除；清理任务只删除超过保留期的无会话临时文件 |
| `pwrite` 完成后、位图置位前 | 文件区域已有数据但位图未完成 | 客户端重传同一偏移，覆盖写后再置位 |
| 位图完成后、客户端未收到响应 | 分片已经完成 | 相同摘要返回幂等成功，不同摘要返回冲突 |
| 进入 `VERIFYING` 后进程退出 | 会话停在校验中，无归档记录 | 完成请求在会话锁内重新执行确定性的大小/摘要校验 |
| `rename` 成功后、MySQL 插入前 | 最终文件存在、无元数据 | 按确定路径复用文件并补写唯一 `upload_id` 元数据 |
| MySQL 插入后、Redis 更新前 | 已有归档记录、会话状态未完成 | 查询唯一记录并把 Redis 状态修复为 `ARCHIVED` |
| 客户端未收到完成响应 | 归档可能已经完成 | 重试返回同一 `archive_id`，不生成新文件和记录 |

恢复只在同一业务操作重试或清理任务中完成，不建设通用补偿平台。遇到文件摘要不匹配时不
继续归档；遇到无法判定的正式文件不自动删除。

Redis 上传状态按持久化和 `noeviction` 部署。若会话仍因重启、误删或故障丢失，服务端不根据
临时文件猜测分片状态，采集端创建新会话重新上传；旧临时文件超过保留窗口后由严格清理规则
处理。该取舍避免把每个高频分片同步写入 MySQL。

## 17. 安全模型

- Nginx 在生产入口终止 TLS，C++ 服务只监听回环地址或受控内网地址；开发环境可直接 HTTP。
- 设备写接口使用设备 HMAC；历史查询和详情接口使用独立的 Operator Bearer Token。
- 健康检查只返回总体状态和稳定错误码，不泄露连接串、路径、SQL 或凭据。
- MySQL、Redis、Operator Token 和后续 TLS 私钥均从环境或只读部署文件注入，不写入仓库。
- 客户端原文件名只存为元数据；归档路径只使用服务端生成且经过枚举/标识符校验的片段。
- 请求体、分片和文件总大小均有限制；服务端不解压设备 ZIP，因此不存在服务端解压路径穿越。
- 日志对 Authorization、签名、设备密钥和密码字段做禁止输出约束，而不是事后字符串替换。
- MySQL 生产部署使用最小权限账号；当前 `root` 账号只用于本地开发和建库。
- 当前本机 Redis 恢复为原有的无密码 `default` 用户，并只绑定回环地址；生产部署若启用 Redis
  密码或 ACL，凭据通过可选环境变量注入，不在代码中固化用户名或密码。

## 18. 部署与运维

终板将生成：

- CMake Release/Debug 构建配置和依赖检查；
- 顺序执行且可重复判断版本的 MySQL 迁移脚本；
- systemd 服务样例，设置工作目录、环境文件、自动重启和文件描述符上限；
- Nginx TLS 反向代理样例，设置请求体上限、上传超时和必要转发头；
- logrotate 样例，避免服务日志无限增长；
- 初始化、启动、停止、健康检查、端到端上传和清理验证脚本；
- 配置、环境变量、目录权限、备份和回滚说明。

服务启动顺序为：加载并验证配置与环境变量、初始化日志、检查临时/归档目录在同一文件系统、
验证 MySQL/Redis、注册路由和清理调度、开始监听。任一必要条件失败则清晰退出，不以离线模式
或内存状态继续运行。

归档备份的责任边界是提供一致的目录结构、元数据和校验摘要；自动异地备份不在第一期代码中。
恢复时先恢复归档目录和 MySQL，再用归档记录的 SHA-256 抽样或全量校验。

## 19. 终板发布门槛

发布 `v1.0.0` 前必须全部满足：

1. Debug 与 Release 均能从干净构建目录完成配置、编译和 CTest；
2. `clang-format` 检查无差异，`clang-tidy` 对项目源码无未处理错误；
3. 核心业务单元测试行覆盖率达到 80%，错误分支以场景覆盖为准，不为数字写无意义测试；
4. MySQL/Redis 集成测试在独立前缀和测试表中通过，不污染已有数据；
5. `data/mock_inbox` 五类文件完成端到端上传、归档、查询和哈希回验；
6. 签名错误、重放、缺片、分片冲突、摘要错误、会话过期和基础设施不可用均返回约定错误；
7. ASan/UBSan 或 Valgrind 未发现项目代码的内存错误、泄漏和未定义行为；
8. 基础并发压测期间无文件损坏、重复归档、进程崩溃和不可解释的 5xx；
9. systemd、Nginx、配置和迁移步骤在本机完成一次部署演练；
10. 源码注释、README、API、配置、迁移和实际行为一致，且不存在占位 TODO。

任何一项未完成都必须在交付说明中标明，不能把阶段性可运行版本称为生产终板。

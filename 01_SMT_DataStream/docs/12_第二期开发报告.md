## 2. 认证和写入顺序

一次心跳严格按以下顺序执行：

1. 检查五个认证头、标识格式、摘要格式和时间窗口；
2. 查询 MySQL 的设备、所属工位和产线，区分未登记、停用和依赖失败；
3. 计算请求体 SHA-256 和 HMAC-SHA256，以常量时间比较摘要；
4. Redis `SET key timestamp NX EX ttl` 消费请求标识；
5. 严格解析心跳 JSON；
6. 以同一次服务端取时更新 MySQL `software_version` 和 `last_seen_at`；
7. 使用固定 Lua 脚本一次写入 Redis Hash 并设置心跳 TTL；
8. 两个存储步骤均成功后返回在线状态。

该顺序有意让设备状态优先于签名结果，避免把 MySQL 故障伪装成签名错误。通过认证的请求标识
在业务 JSON 校验前被消费，因此调用方修正请求后必须使用新请求标识。MySQL 成功、Redis
失败时返回 `REDIS_UNAVAILABLE`；不重试，也不把 Redis 状态伪装成已更新。

## 3. 主要产物

- `auth/crypto`：SHA-256、HMAC-SHA256、规范串和常量时间比较；
- `auth/device_authenticator`：请求头、时间、设备主数据、签名和防重放异步链；
- `auth/operator_authenticator`：严格 Bearer 格式和 Token 比较；
- `common/time_utils`：UTC 毫秒时间生成与带时区 ISO 8601 严格解析；
- `device/device_repository`：固定 SQL 模板的设备查询与心跳更新；
- `device/heartbeat`：拒绝未知字段的心跳模型解析；
- `api/heartbeat_controller`：心跳路由、MySQL/Redis 顺序写入和统一响应；
- `tests/e2e/heartbeat_e2e_test.py`：启动真实服务并验证成功、篡改、过期、重放、未知设备、
  停用设备、非法摘要、非法 JSON、TTL、数据库更新和日志脱敏。

## 4. 业务边界

- `collector_id` 使用 2 至 64 位大写字母、数字、下划线或连字符；
- 软件版本和工单为有限长度的可打印 ASCII；
- `runtime_status` 仅接受 `RUNNING`、`IDLE`、`ALARM`；只有 `IDLE` 允许空工单；
- `reported_at` 必须包含三位毫秒和 `Z` 或数值时区，校验日历日期并换算为 UTC；它只用于
  诊断，不参与在线时间判定；
- 在线时间以服务端接收时间为准，MySQL 与 Redis 使用同一次取时结果；
- Redis 在线键为 `<prefix>heartbeat:<device_id>`，请求键为
  `<prefix>auth:req:<device_id>:<request_id>`；
- 日志只记录稳定错误码和请求标识，不记录请求正文、签名、HMAC 密钥或 Bearer Token。

## 6. 端到端验收记录

真实验收使用本机 MySQL、无密码 `default` Redis 用户和临时 HTTP 端口执行：

```bash
export SMT_DATASTREAM_MYSQL_PASSWORD='123456'
export SMT_DATASTREAM_OPERATOR_TOKEN='e2e-operator-token-2026'
ctest --test-dir build -L integration --output-on-failure
```

测试结束已核对停用测试设备为 0 条、E2E Redis 测试键无残留。验收未修改 Redis ACL，仍使用
原有无密码 `default` 用户。

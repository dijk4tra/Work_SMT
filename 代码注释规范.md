# C++ 注释与代码文档规范 (C++ Documentation Standards)

为了保证代码库的可读性、工程严谨性以及 IDE 智能提示（IntelliSense）的友好度，所有生成的 C++ 代码必须严格遵守以下注释规范。

---

## 1. 文件头注释 (File Headers)
每个新建的 `.h`, `.hpp`, `.cpp` 文件顶部，必须包含文件头注释，用以说明文件的基本信息。统一使用普通块注释 `/* ... */`。

* **必须包含**：文件名（@file）、简要描述（@brief）。
* **选择包含**：作者、创建日期、版权/开源协议。

**示例：**
```cpp
/**
 * @file config_manager.h
 * @brief 负责系统配置文件的解析与全局访问管理类。
 * @author [Your Name/Agent]
 * @date 2026-07-10
 * @copyright Copyright (c) 2026. All rights reserved.
 */
```

---

## 2. 类、结构体与枚举注释 (Classes, Structs & Enums)
在类、结构体、枚举类型的声明上方，必须使用 **Doxygen 三斜线风格 (`///`)** 进行说明。

* **要求**：使用 `@brief` 明确该类型的职责，复杂的类可以使用空行分隔后书写详细设计。

**示例：**
```cpp
/// @brief 全局配置管理器，采用单例模式设计。
///
/// 该类在程序启动时读取指定的 INI 文件，并在内存中维护一个键值对映射表。
/// 所有的配置读取操作都是线程安全的。
class ConfigManager {
    // ...
};

/// @brief 配置文件解析结果的错误码。
enum class ConfigError {
    Success = 0,     ///< 解析成功
    FileNotFound,    ///< 配置文件不存在
    InvalidFormat    ///< 文件内容格式错误
};
```

## 3. 函数与接口声明注释 (Function & Interface Declarations)
**所有在头文件中的公共/私有函数声明、全局函数声明，必须使用 Doxygen 三斜线风格 (`///`) 搭配 `@` 标签。**

### 标签约束规则：
1. **`@brief`**：一句话简述功能（**必填**）。
2. **`@param`**：格式为 `@param 参数名 描述`。只要有入参，就**必须**为每个参数写一条（**有参数时必填**）。
3. **`@return`**：描述返回值。只要返回值不是 `void`，就**必须**编写（**有返回值时必填**）。
4. **`@throws`**：描述可能抛出的异常类型及触发条件（**有异常时必填**）。
5. **`@note`**：用于强调生命周期、线程安全限制、性能警示等重要注意事项（**选填**）。

**正确示例：**
```cpp
/// @brief 查询指定配置项的值。
/// @param key 要查询的配置项名称（大小写敏感）。
/// @return 配置值的常量引用。
/// @note 返回的引用生命周期与 ConfigManager 实例绑定，切勿跨生命周期存储。
/// @throws std::invalid_argument 当 key 为空字符串时抛出。
/// @throws std::runtime_error 当 key 在配置表中不存在时抛出。
const std::string& getValue(const std::string& key) const;
```

## 4. 函数内部实现注释 (Implementation Comments)
函数体内部（`.cpp` 文件或内联实现中）的代码注释，属于对内实现细节，**严禁使用 `///` 或任何 Doxygen 标签**，必须统一使用**普通双斜杠 (`//`)**。

### 编写原则：
1. **解释“为什么（Why）”，而不是“是什么（What）”**：不要用注释重复代码本身。
2. **位置**：单行注释写在被注释代码的**上方**，并保持缩进对齐。行尾注释仅在变量声明或非常简短的语句后使用（留出至少 2 个空格）。

**正反面对比示例：**

❌ **错误做法 (Anti-Pattern)：**
```cpp
int age = 18; // 定义一个变量 age 赋值为 18
if (age >= 18) { /// @brief 如果大于18 (在内部错误地使用了Doxygen)
    // 增加计数
    count++; 
}
```

✔️ **正确做法 (Best Practice)：**
```cpp
int age = 18;  // 默认初始投票年龄为 18 岁
if (age >= 18) {
    // 满足法定年龄后，需要同步更新全局审计计数器的状态
    updateAuditCounter();
}
```

## 5. 功能性标记注释 (Task Tags)
在代码中标记临时状态或后续任务时，统一在函数内部使用普通单行注释，格式为 **`// 标签名: 描述`**（标签名必须全大写）。

* **`// TODO: ...`**：待办事项，表示功能尚未实现或需要进一步重构优化。
* **`// FIXME: ...`**：已知此处有 Bug、边界漏洞或性能隐患，需要尽快修复。
* **`// NOTE: ...`**：强调此处特殊的业务逻辑、历史原因或特殊的 Trick，防止后人误改。
* **`// HACK: ...`**：承认为了赶进度或绕过某个外部库的 Bug 而写出的不优雅的代码。

**示例：**
```cpp
void processData() {
    // TODO: 考虑在未来引入多线程加速此处的循环解析
    for (auto& item : m_data) {
        // FIXME: 如果 item 内部包含非法 UTF-8 字符，可能会导致此处崩溃
        parseItem(item);
    }
}
```

## 6. 自动化与禁忌规则 (Strict Forbidden Rules)
* **禁止注释废弃代码**：严禁将整段不用的代码注释后留在源码中。如果不使用该代码，请直接删除。历史版本交由 Git 管理。
* **同步更新**：在修改函数行为、参数、返回值或抛出异常时，**必须**同步更新头文件中的 Doxygen 注释，严禁文档与实际代码脱节。
* **语言要求**：除非有特殊声明，所有代码注释的文本描述统一使用**简体中文**，标点符号在中文文本中统一使用中文标点。

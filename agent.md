# SMT 项目代码生成规则

## 1. 适用范围与资料优先级

本规则适用于 `01_SMT_DataStream` 与 `02_SMT_LogTrace` 的代码、配置、脚本、
数据库脚本、测试和文档。开始工作前必须阅读：

1. `SMT产线项目.md`：项目范围和简历中已声明的技术事实；
2. `代码注释规范.md`：C++ 注释的完整规范；
3. 当前项目的 README、`docs/`、配置示例和已有代码；
4. 距离目标文件最近的 `AGENTS.md`（若存在）。

当资料冲突时，用户本轮明确要求优先，其次是本文件、项目文档、已有实现。
不得为了展示技术栈而虚构业务需求，也不得把第二个日志检索项目的职责混入
第一个数据采集项目。

## 2. 实现原则

- 使用满足已确认需求的最简单实现，代码必须能够编译、运行和测试。
- 不添加未被要求的兼容层、降级路径、重试机制、特性开关、插件体系或扩展点。
- 遵循项目现有的错误处理方式，不为局部功能引入另一套错误模型。
- 信任已经由上游校验并写入接口契约的输入，不在多层重复校验同一条件。
- 只处理当前抽象边界内真实可能发生的失败。
- 仅当本层能够恢复、转换错误、释放资源或补充必要上下文时捕获异常。
- 禁止吞掉异常、忽略错误码，或用空字符串、零值、空对象等任意默认值掩盖失败。
- 只记录可执行的故障信息和重要状态变化，不为每个分支打印日志。
- 对违反内部不变量的程序错误，在适合时使用断言；不要将其改造成运行时兜底分支。
- 不能保证数据正确时应明确失败，不得带着可能无效的状态继续执行。
- 使用 C++11；除非需求和构建环境明确变更，不使用更高标准语法。

## 3. 范围控制

- 做完成当前任务所需的最小合理改动，不顺手重构无关代码。
- 不为单个调用点引入工厂、包装器、帮助类、通用配置对象或模板化抽象。
- 不保留假想的向后兼容行为，不实现“以后可能需要”的功能。
- 新依赖必须解决明确问题，并在 README 中说明用途、版本和获取方式。
- 不生成大段占位实现、伪代码或无法验证的演示接口。
- 测试应验证真实契约，不为了通过测试在生产代码中设置测试专用分支。

## 4. 边界与错误处理

优先在以下信任边界校验输入：

- 外部 HTTP/RPC 请求及上传内容；
- 命令行参数和配置文件；
- 数据库、Redis、网络和文件系统返回值；
- 设备产生的未受信文件、JSON、CSV、日志与元数据。

在已验证的内部调用链中：

- 依赖已记录的前置条件；
- 不重复做空值、范围、格式和存在性校验；
- 不为契约上不可达的状态增加防御分支；
- 不因 API “理论上可能失败”而并列实现另一套替代方案。

平台中允许且需要处理的典型失败包括：签名错误或过期、请求重放、设备不存在或
停用、上传会话过期、分片参数非法、文件 I/O 失败、大小或 SHA-256 不一致、
数据库/Redis/网络调用失败。是否重试由明确的接口幂等性和需求决定，不得默认重试。

## 5. C++ 注释规范

以 `代码注释规范.md` 为完整依据，并强制执行以下规则：

- 每个新建 `.h`、`.hpp`、`.cpp` 文件以 `/** ... */` 文件头开篇，必须包含
  `@file` 和 `@brief`；作者、日期和版权按项目约定选填。
- 类、结构体和枚举声明上方使用 `///` 与 `@brief`。枚举值使用简短的 `///<` 说明。
- 头文件内所有公共/私有成员函数和全局函数声明都使用 `///` Doxygen 注释。
  `@brief` 必填；每个入参都有 `@param`；非 `void` 返回值都有 `@return`；
  确实会抛异常时才写 `@throws`。
- 函数实现内部只使用普通 `//`，禁止使用 `///` 或 Doxygen 标签。
- 内部注释解释设计原因、业务约束或非显然选择，不复述代码行为。
- 任务标记只使用 `// TODO:`、`// FIXME:`、`// NOTE:`、`// HACK:`，描述必须具体；
  不得用 TODO 代替本次任务应完成的实现。
- 禁止注释掉废弃代码；修改行为、参数、返回值或异常契约时同步更新注释。
- 除专有名词、协议字段和外部接口原文外，代码注释使用简体中文和中文标点。
- 不给显而易见的 getter、局部变量、循环或条件分支添加逐行说明。

Python、Shell、SQL 等非 C++ 文件同样只注释非显然的原因和约束，不写逐行翻译式注释。

## 6. 项目一的业务约束

`01_SMT_DataStream` 只负责采集与归档：

- 设备身份认证、HMAC-SHA256 请求校验、时间窗口与请求标识防重放；
- 设备心跳和状态维护；
- 上传会话、乱序分片、断点续传、完整性校验与原子归档；
- 文件元数据持久化、历史记录查询和过期临时数据清理。

原始业务文件包括检测结果、测试报告、NG 图片、设备导出文件和运行记录。
正式归档文件正文保存在文件系统，MySQL 保存可查询元数据，Redis 保存有明确生命周期的
心跳、上传会话、分片位图和防重放标识。禁止把日志全文检索、倒排索引、BM25 或 SLRU
缓存提前实现到项目一中。

项目一按 `01_SMT_DataStream/docs/09_分期实施计划.md` 的六期顺序生成，六期全部通过发布门槛后
才可称为第一期生产终板。阶段中不得保留需要用户补写的空实现或固定成功响应。MySQL 与
Redis 统一使用 Workflow 异步任务；禁止引入本机要求 C++17 的 redis++。密码、设备密钥和
Operator Token 只从环境或受控数据源读取，禁止把真实值写入源码、配置示例、测试快照和日志。

## 7. 开工前检查

每次编码前先简要确认：

1. 实际输入边界是什么；
2. 当前层负责哪些失败；
3. 调用方已经保证哪些条件；
4. 是否准备加入兜底、重试、兼容分支或抽象。

第 4 类内容只有在用户明确要求或仓库已有稳定约定时才能实现。需求不清但可由现有资料
唯一推导时应继续工作；若不同选择会实质改变外部契约、数据安全或项目范围，则先询问。

## 8. 完成与评审

完成代码后必须按风险比例执行格式检查、编译和测试，并复核变更，删除：

- 重复校验和不可达的防御分支；
- `catch (...)` 等无意义的全捕获；
- 静默兜底值和推测性兼容代码；
- 只有一个使用点且没有降低复杂度的抽象；
- 解释不可能场景或逐字复述代码的注释；
- 未使用的依赖、配置项、函数、文件和测试数据。

交付说明必须区分已验证事实与尚未验证事项，不声称未执行的测试已经通过。
